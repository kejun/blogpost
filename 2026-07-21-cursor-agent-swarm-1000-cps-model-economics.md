## 一、一个反直觉的实验结果

2026 年 7 月 20 日，Cursor 团队发布了一篇题为 *"Agent swarms and the new model economics"* 的技术博客，在 Hacker News 上迅速获得 93 分、41 条评论。文章披露了一组令人震惊的数据：

| 指标 | 旧版 Swarm（2026 年初） | 新版 Swarm（2026 年 7 月） |
|------|------------------------|--------------------------|
| 峰值提交速率 | ~1,000 commits/hour（Git） | ~1,000 commits/second（自研 VCS） |
| SQLite 测试通过率（4 小时） | 11%–77%（模型依赖） | 73%–85%，最终全部达到 100% |
| 合并冲突数 | >70,000（2 小时内，仍在加速） | <1,000（完整 4 小时） |
| 最热文件冲突数 | 7,771 次（1,173 个 Agent 触碰） | 47 次 |
| Crate 数量（Rust 包） | 54 个（含 3 个重复 SQL 包） | 9 个（早期确定，从未增加） |
| 引擎代码行数（Fable 5 配置） | 64,305 行 | 9,908 行 |
| 成本范围 | — | $1,339（Opus 混合）到 $10,565（GPT-5.5 单体） |

**最反直觉的发现**：所有模型配置（从全 GPT-5.5 到 Opus 4.8 + Composer 2.5 混合）产出了**相似的质量**，但成本差异高达 **8 倍**。

这意味着什么？**在 Agent 群体智能时代，"用什么模型"的重要性正在让位于"怎么组织模型"。** 架构设计——而非模型能力——正在成为决定 Agent 系统产出质量的核心变量。

## 二、从"浏览器实验"到"SQLite 基准"：Cursor 群体智能的演进路径

### 2.1 第一阶段：证明可行性（2026 年初）

Cursor 的群体智能研究始于一个野心勃勃的实验：**让数百个 Agent 协作从零构建一个 Web 浏览器**。

这个实验的结果是"成功但不完美"——Agent 们运行了近一周，写了超过 100 万行代码、跨越 1,000 个文件，产出了几乎完全可运行的代码。但距离"精打磨的软件"还有很大距离。

更关键的是，这个阶段的协调机制极其原始：

- **扁平自协调**：所有 Agent 地位平等，通过共享状态文件协调。结果是锁竞争严重——20 个 Agent 的有效吞吐量退化到 2-3 个。
- **风险规避**：没有层级结构意味着没有 Agent 愿意承担困难任务。它们倾向于做小而安全的修改，而不是端到端地解决问题。
- **Git 瓶颈**：基于 Git 的粗粒度锁在每小时 1,000 次提交的速率下已经不堪重负。

### 2.2 第二阶段：工程化重构（2026 年 7 月）

新版 Swarm 的核心改进不是"用更强的模型"，而是**重新设计了 Agent 的组织架构和基础设施**。Cursor 团队用 SQLite 作为基准测试——让 Agent 群仅凭 835 页 SQLite 手册，用 Rust 从零实现一个完整的 SQLite，不提供源码、测试套件、SQLite 二进制文件或互联网访问。

评估使用 sqllogictest——SQLite 项目官方的跨引擎一致性测试套件，包含数百万条已知正确答案的查询。Agent 群从未被告知这个测试套件的存在。

## 三、"树与叶"架构：为什么上下文效率比并行更重要

### 3.1 单 Agent 的根本困境

新版 Swarm 的核心架构洞察可以用一句话概括：**Agent 群体的可扩展性来自上下文效率，而非并行本身。**

当一个单 Agent 承担完整任务时，它必须独自遍历整棵任务树——深入到每个叶节点，同时始终在上下文中保持祖先节点、当前位置和全局目标。这解释了为什么长时间运行的单 Agent 会"漂移"：

```
单 Agent 的上下文困境：

┌─────────────────────────────────────────────────┐
│  选择 A：聚焦当前工作                            │
│  → 上下文被底层细节填满                          │
│  → 失去全局视野                                  │
│  → "只见树木，不见森林"                          │
├─────────────────────────────────────────────────┤
│  选择 B：保持全局视野                            │
│  → 上下文被高层规划占据                          │
│  → 当前工作的执行质量下降                        │
│  → "只见森林，不见树木"                          │
└─────────────────────────────────────────────────┘

无论哪种选择，单 Agent 都在做零和博弈。
```

### 3.2 Planner-Worker 分离：科斯定理的 Agent 版本

新版 Swarm 的解法是**角色分离**：

- **Planner Agent**（由最聪明的模型驱动）：只负责分解目标、委派任务，**从不实现代码**。因此其上下文永远不会被底层细节污染。
- **Worker Agent**（由更快、更便宜的模型驱动）：只负责执行被分配的具体任务，**从不规划**。因此其全部上下文都可以用于一个狭窄的工作片段。

Cursor 团队明确引用了经济学家 Ronald Coase 的企业理论：协调成本的增长速度快于工作本身，因此组织会稳定在"有界单元"的层级结构中，而不是让所有人互相通信。

**这个类比精确到令人不安的程度。** 人类工程团队的 Code Review、Ownership、Standup、Merge Queue——这些机制在人类节奏下运作良好。但在每秒 1,000 次提交的速率下，人类团队不会遇到的失败模式开始涌现。

### 3.3 为什么这比"更多并行"更重要

一个关键数据点：即使在新版 Swarm 中，**Worker 承载了至少 69% 的 Token 消耗，大多数情况下超过 90%**。但美元支出的分布完全不同——Planner 虽然消耗少量 Token，却使用了最昂贵的模型。

这揭示了一个深层结构：**Agent 群体的价值不在于"同时做更多事"，而在于"让每个 Agent 在更窄的上下文窗口中做更精确的事"。** 并行是副产品，上下文效率才是核心。

## 四、每秒 1000 次提交：为什么 Cursor 必须自研 VCS

### 4.1 Git 的结构性瓶颈

旧版 Swarm 在 Git 上的峰值是每小时约 1,000 次提交。新版系统的峰值是**每秒约 1,000 次**——提升了 **3,600 倍**。

这不是简单的性能优化问题。Git 的设计假设是**人类节奏的并发**：几个开发者偶尔推送代码，合并冲突是低频事件。当数百个 Agent 同时向同一个代码库推送时，Git 的粗粒度锁机制完全崩溃。

但吞吐量不是自研 VCS 的唯一原因。Cursor 团队指出：**每个变更都经过 VCS，因此 VCS 是冲突首先变得可见的地方。** 多个协调机制直接实现在 VCS 内部——VCS 不仅是存储层，更是**协调层**。

### 4.2 五种"人类团队不会遇到"的失败模式

在每秒 1,000 次提交的速率下，Cursor 发现了五种在人类工程团队中不常见的失败模式：

**① 脑裂设计（Split-Brain Design）**

两个互不知情的 Planner 在代码库的不同部分以不同方式实现了同一个概念。

**解法**：通过 Prompt 工程——Planner 自己做设计决策而不是委派，并确保没有两个被委派的子树决定同一个问题。

**② Planner 间竞争（Contention Between Planners）**

更棘手的情况：两个 Planner 知道彼此的存在，通过反复修改同一文件来"打架"。问题的本质是**两个现实图景**，合并工具无法修复分歧。

**解法**：Agent 将决策记录在共享设计文档中。依赖某个决策的代码携带一个**编译时检查的引用**指向其文档。当 Planner 无意中互相矛盾时，一个 Reconciler 合并文档，引用将解决方案传播到下游。

**③ 合并冲突（Merge Conflicts）**

Agent 不断在同一文件上碰撞。要解决碰撞，它们需要停下来、吸收另一个 Agent 的上下文、然后围绕它合并。Worker Agent 不擅长这个——实际上，它们要么覆盖对方的更改，要么放弃自己的。

**解法**：引入**中立的第三方 Agent** 介入合并冲突，代表所有方解决。它的唯一目标是公正和高效——类似于工程团队中的 Merge Queue。

**④ 巨型文件（Megafiles）**

某些文件是 Agent 工作的"热门地点"。每个 Agent 可能只添加少量代码，但没有单个 Agent 负责保持文件小巧。这些"巨型文件"会阻塞一切——传输、diff、merge 都很昂贵，并成为持续碰撞的地点。

**解法**：给 Worker Agent 一种标记膨胀文件的方式。一旦被标记，阻止新提交，一个外部 Agent 将过大的文件分解为更小的模块。

**⑤ 骨化（Ossification）**

Agent 从与人类协作的现有代码库中学到了"不要碰核心代码"——即使它需要改变。

**解法**：**许可有意破坏**。一个判断核心变更值得的 Agent 可以在其范围之外做一个聚焦的补丁，并留下注释解释原因。编译器将变更传播到系统其余部分，所有依赖旧设计的代码都会构建失败。每个遇到这些错误的 Agent 找到注释、阅读推理、然后更新自己的工作以匹配。

### 4.3 数据说话：旧版 vs 新版

| 指标 | 旧版 Swarm | 新版 Swarm |
|------|-----------|-----------|
| 2 小时内提交数 | 68,000 | ~1,000（4 小时总计） |
| 合并冲突 | >70,000（仍在加速） | <1,000（4 小时） |
| 最热文件冲突 | 7,771 次 / 1,173 Agent | 47 次 |
| Crate 数量 | 54（含 3 个重复 SQL 包） | 9（早期确定） |
| 代码行数（Fable 5） | 64,305 行 | 9,908 行 |
| 代码行数（Opus 混合） | 19,013 行（97%） | 4,645 行（100%） |

旧版 Swarm 的 68,000 次提交看起来"更高产"，但其中大部分是**忙功（thrash, contention, churn）**。新版用 1/70 的提交量产出了 6.5 倍更少的代码和更好的测试结果。

**这是一个关于"少即是多"的深刻教训：在 Agent 系统中，协调质量比活动数量重要得多。**

## 五、Review Lens 与 Field Guide：群体智能的"免疫系统"和"文化传承"

### 5.1 堆叠式审查透镜（Stacked Review Lenses）

在长时间运行的多 Agent 系统中，错误会累积。Swarm 需要一种方式在小错误变成基础性错误之前自我纠正。

Cursor 实验了多种"审查透镜"：
- 给审查 Agent 完整的 Worker 转录
- 只给输出
- 只给代码库
- 使用不同模型、不同训练、不同"个性"的审查者

**核心发现：没有单一透镜能捕获一切，但不相关的透镜可以堆叠。** 这类似于自动驾驶系统——没有任何单一完美组件，但堆叠的冗余系统可以达到超越人类的可靠性。

审查的计算成本是高回报的，因为**审查比它审计的工作便宜得多**。Cursor 团队认为这个堆叠审查系统是运行持续质量的主要贡献者。

### 5.2 Field Guide：Agent 的"文化传承"机制

**Stigmergy（标记效应）** 是蚂蚁和白蚁等群体生物在没有直接通信的情况下协调的机制——它们塑造环境，环境塑造下一个有机体。

Cursor 将这个概念工程化为 **Field Guide**：一个完全由 Agent 拥有的文件夹，其 `index.md` 在每个 Agent 启动时自动注入。Agent 负责策划 Guide 的内容，唯一的约束是行数预算。

Field Guide 的底层逻辑是：**模型权重是冻结的，因此恰恰是"意外遭遇"值得被捕获，以便下一个 Agent 的轨迹更短。**

```
Field Guide 的信息流：

Agent A 遇到意外 → 记录到 Field Guide → Agent B 启动时读取
→ Agent B 的轨迹更短 → Agent B 遇到新意外 → 更新 Field Guide
→ Agent C 受益 → ...

这是一个正反馈循环：
群体经验 → 环境知识 → 个体效率 → 更多群体经验
```

这个机制与人类工程团队的"机构知识"惊人地相似——但有一个关键区别：**人类的机构知识通过文档、代码注释、口头传承传播，而 Agent 的机构知识必须通过结构化的、机器可读的格式传播。**

Cursor 团队指出，Field Guide 的收益在 Agent 不完全拥有的代码库上会更大。训练模型"为后继者写作"——更好的捕获带来更好的奖励——是一个有趣的后续研究方向。

## 六、模型经济学：8 倍成本差异背后的结构性洞察

### 6.1 四种配置的对比

Cursor 测试了四种跨越能力和成本的配置：

| 配置 | Planner | Worker | 4 小时通过率 | 成本 |
|------|---------|--------|-------------|------|
| 全 GPT-5.5 | GPT-5.5 | GPT-5.5 | ~85% | $10,565 |
| 全 Grok 4.5 | Grok 4.5 | Grok 4.5 | ~80% | — |
| Opus 混合 | Opus 4.8 | Composer 2.5 | ~82% | $1,339 |
| Fable 混合 | Fable 5 | Composer 2.5 | ~73%（1 小时达 2/3） | — |

**所有配置最终都通过了 100% 的测试套件。** 质量差异远小于成本差异。

### 6.2 Token 消耗 vs 美元支出

一个关键结构：**Worker 承载了至少 69% 的 Token 消耗（大多数情况下超过 90%），但美元支出的分布完全不同。**

这意味着：
- **Planner 的 Token 消耗少但单价高**（使用最昂贵的模型）
- **Worker 的 Token 消耗大但单价低**（使用便宜模型）
- **总成本由 Planner 的模型选择主导**

这产生了一个深刻的经济学洞察：**在 Agent 群体中，"思考"比"执行"贵得多，但"思考"的量远小于"执行"。** 最优策略是用最贵的模型做最少的思考，用最便宜的模型做最多的执行。

### 6.3 对行业的启示

这个发现对 AI 基础设施行业有深远影响：

**① 模型路由不再是"选最好的模型"，而是"为每个角色选最合适的模型"。** 这与我们此前分析的 IBM Research 模型路由报告（2026-07-18）形成呼应——路由的核心不是"哪个模型最好"，而是"哪个模型最适合这个特定任务"。

**② "前沿模型"的护城河正在从"能力"转向"规划能力"。** 当 Worker 可以用便宜模型替代而不损失质量时，前沿模型的不可替代性集中在 Planner 角色——即分解问题、做设计决策、协调全局的能力。

**③ 成本优化的杠杆不在 Token 价格，而在架构设计。** 同样的任务，架构优化（从旧版到新版）带来的效率提升远超模型降价。

## 七、更深层的思考：Agent 群体智能的"相变"

### 7.1 从"量变"到"质变"

Cursor 的实验揭示了一个重要现象：Agent 群体智能存在一个**相变点（phase transition）**。

在旧版 Swarm 中，增加 Agent 数量导致的是**负收益**——更多 Agent 意味着更多冲突、更多重复工作、更多忙功。68,000 次提交中大部分是浪费。

在新版 Swarm 中，增加 Agent 数量产生的是**正收益**——因为架构确保了每个 Agent 在狭窄的上下文中高效工作，冲突被系统性地解决。

**相变的关键不是模型能力，而是协调基础设施。** 自研 VCS、角色分离、合并冲突仲裁、巨型文件分解、骨化许可——这些"基础设施"决定了群体智能是涌现还是崩溃。

### 7.2 与人类组织理论的深层对应

Cursor 团队引用了 Coase 的企业理论，但这个对应关系比他们明确阐述的更深：

| 人类组织 | Agent Swarm | 功能 |
|---------|------------|------|
| CEO / CTO | Planner Agent | 战略分解、资源分配 |
| 工程师 | Worker Agent | 具体执行 |
| Code Review | Review Lens | 质量保障 |
| 设计文档 | 共享设计文档 + 编译时引用 | 决策传播 |
| Merge Queue | 中立第三方合并 Agent | 冲突解决 |
| 代码重构 | 巨型文件分解 Agent | 结构维护 |
| 机构知识库 | Field Guide | 经验传承 |
| "打破常规"文化 | 骨化许可 | 避免路径依赖 |

**但有一个关键差异：人类组织的协调成本随规模超线性增长（Brooks 定律），而 Agent Swarm 的协调成本可以通过基础设施工程化来亚线性化。**

这是 Agent 群体智能相对于人类团队的结构性优势——不是"更快"，而是**协调成本的可扩展性根本不同**。

### 7.3 HN 社区的争议："为了做按钮而发明宇宙"？

Hacker News 上的评论呈现出有趣的分裂：

**质疑派**："Talk about inventing the universe to make a button"（为了做按钮而发明宇宙）——自研 VCS、复杂的协调机制，是否过度工程化？

**支持派**："If at the end Shakespeare-level literature is produced, does it matter whether we arrived there by random keystrokes?"（如果最终产出了莎士比亚级别的文学，是否通过随机敲击到达又有什么关系？）

**务实派**："I would have loved to see more of the harness engineering shared as code"（我更希望看到 harness 工程以代码形式分享）——但承认 harness 本身就是 Cursor 的产品。

**批评派**："I call this junk 'meta-agentic engineering'... ultimately these are trends pushed on us from model providers because they 10x token consumption"（我称之为"元 Agent 工程"垃圾……这些趋势是模型提供商推动的，因为它们 10 倍增加了 Token 消耗）。

这些争议本身揭示了一个行业张力：**Agent 群体智能是真正的范式突破，还是 Token 消费主义的精致包装？**

我的判断是：**两者都是，但前者的比重正在增加。** Cursor 的数据——用 1/70 的提交量产出 6.5 倍更少的代码和更好的结果——不是 Token 消费主义能解释的。这是真实的工程效率提升。

## 八、对开发者的实践启示

### 8.1 短期（6 个月内）

- **角色分离立即可用**：即使不用自研 VCS，将 Planner 和 Worker 分离（用不同模型驱动）也能显著提升多 Agent 系统的效率。
- **审查透镜堆叠**：在你的 Agent 工作流中加入多视角审查——不同模型、不同上下文、不同"个性"的审查者。
- **Field Guide 模式**：为你的 Agent 系统建立一个共享的、Agent 可维护的知识库。不需要复杂——一个 `index.md` 就够了。

### 8.2 中期（6-18 个月）

- **VCS 层将成为 Agent 基础设施的关键组件**。Git 不是为 Agent 节奏设计的。期待看到更多"Agent-native VCS"的出现。
- **模型路由将变成"角色路由"**。不是"这个任务用哪个模型"，而是"这个 Agent 角色用哪个模型"。
- **协调基础设施将成为竞争壁垒**。Cursor 的自研 VCS 和协调机制是其护城河的一部分。

### 8.3 长期（18 个月以上）

- **"训练模型为后继者写作"将成为一个研究方向**。Field Guide 的自动化策划目前依赖 Prompt 工程，未来可能通过 RL 优化。
- **Agent 群体的"组织设计"将成为一个工程学科**。类似于人类的管理学，但基于完全不同的约束条件。
- **成本结构将重塑 AI 商业模式**。当 $1,339 可以产出与 $10,565 相同质量的结果时，"按 Token 计费"的商业模式将面临根本性挑战。

---

## 九、结语：协调即智能

Cursor 的群体智能实验揭示了一个深刻的真理：**在 Agent 时代，智能不仅存在于模型权重中，更存在于模型之间的协调结构中。**

一个由便宜模型组成的、良好协调的 Agent 群，可以产出一个由昂贵模型组成的、协调不良的 Agent 群无法企及的结果。这不是"更多算力"的故事，而是"更好组织"的故事。

从 Coase 的企业理论到蚂蚁的标记效应，从人类工程团队的 Code Review 到 Agent 的 Review Lens，从机构知识库到 Field Guide——Cursor 的实验表明，**群体智能的涌现需要的不是更聪明的个体，而是更聪明的协调方式。**

当 Agent 群以每秒 1,000 次提交的速度写代码时，真正的瓶颈不是模型能力，不是 Token 价格，不是硬件算力——而是**我们能否设计出与这个速度匹配的协调基础设施**。

Cursor 给出了一个答案。这个答案不完美——HN 社区的质疑有其道理，harness 没有开源，实验结果还需要更多独立验证。但方向是清晰的：

**Agent 群体智能的未来，不在于让每个 Agent 更聪明，而在于让 Agent 之间的"组织"更聪明。**

协调即智能。这是 2026 年 7 月，Cursor 用 1,000 commits/second 写下的注脚。
