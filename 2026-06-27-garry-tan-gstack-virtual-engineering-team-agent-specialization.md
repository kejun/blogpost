## 二、从"一个 Agent 干所有事"到"23 个专家各司其职"

### 2.1 单 Agent 模式的结构性缺陷

在此之前，大多数 AI 编码工具的工作模式很简单：你告诉 Claude Code 或 Cursor 做什么，同一个模型负责理解需求、设计方案、编写代码、审查质量、修复 bug。

这就像让一个人同时扮演产品经理、架构师、前端开发、后端开发、测试工程师和运维——**不是做不到，而是每个角色都需要不同的思维方式和约束条件**。

```
单 Agent 模式（之前）
═══════════════════════════════════════════════════════

用户 → "做个用户管理功能"
       │
       ▼
┌─────────────────────────────┐
│  Claude Code（同一模型）      │
│  ├─ 理解需求                 │
│  ├─ 设计架构                 │
│  ├─ 写代码                   │

└─────────────────────────────┘
       │
       ▼
   问题：自己审自己的代码，
   盲区永远存在
```

gstack 的核心洞察是：**在 AI 时代，"角色隔离"比"模型能力"更能决定产出质量**。

## 三、810 倍生产力的底层逻辑

### 3.1 "逻辑代码行" vs "原始代码行"

Garry Tan 的数据引发了一个关键争议：**用代码行数衡量 AI 时代的生产力还有效吗？**

他的回应很有意思。他区分了两个概念：

- **Raw LOC**：AI 生成的所有代码行数——确实会膨胀，而且没有意义
- **Logical code change**：经过规范化后的、有实际意义的代码变更

### 3.2 生产力倍增的三个杠杆

仔细分析 gstack 的工作流，可以发现生产力提升来自三个杠杆的叠加：

```
生产力倍增模型
═══════════════════════════════════════════════════════

杠杆 1：编码加速（~50×）
  AI 直接生成代码，跳过手敲

（与 Garry 的 810× 基本吻合）
```

这三个杠杆中，**第二个最容易被忽视，也最有启发性**。

### 3.3 "角色切换"为什么有效

人类工程师之所以能一人扮演多角色，靠的是**思维模式的切换**。写代码时你是实现者，审查代码时你是挑剔的评审员，做设计时你是用户代言人。

gstack 用 slash command 强制了这种切换。当你运行 `/review` 时，Claude Code 不会"继续写代码"——它进入了一个完全不同的审查模式，有独立的约束和输出格式。

这背后的认知科学原理是：**同一个模型（或人），在不同约束条件下会产生不同质量的输出**。这就是为什么"自己写代码 + 自己审查"的效果远不如"写完 + 用审查角色的约束重新审查"。

这个转变的本质是：

| Prompt Engineering | Role Engineering |
|-------------------|------------------|
| 一次性的指令 | 可复用的角色定义 |

gstack 的每个 slash command 本质上就是一个"角色定义"——包含这个角色的**职责、约束、工具和输出格式**。

## 七、实操指南：如何构建你自己的"gstack"

你不需要等待别人发布下一个 gstack。基于今天的分析，你可以立即开始构建自己的角色系统：

### 7.1 最小可行角色集

```markdown
# CLAUDE.md 或 AGENTS.md 中添加

## 角色定义

### 审查者 (/review)
当你被要求审查代码时：
- 用全新的视角阅读代码，不假设之前的决策是正确的
- 检查：安全性、性能、可维护性、错误处理
- 给出具体的改进建议，而不是泛泛而谈

### 测试者 (/qa)
当你被要求测试时：
- 先理解功能需求和边界条件
- 列出测试场景：正常路径、异常路径、边界值
- 如果有浏览器访问能力，进行真实端到端测试

### 安全官 (/security)
当你被要求进行安全审查时：
- 按 OWASP Top 10 逐项检查
- 特别注意：输入验证、认证授权、敏感数据保护
- 给每个发现标注严重级别
```

### 7.2 角色隔离的关键原则

1. **独立上下文**：每个角色启动时，清除上一个角色的思维状态，从零开始
2. **明确约束**：每个角色有明确的"什么不该做"
3. **结构化输出**：每个角色产出格式化的输出，方便下游角色消费
4. **可追溯**：每个角色的决策都有记录，方便复盘

### 7.3 逐步扩展

```
你的角色系统演进路径
═══════════════════════════════════════════════════════

阶段 1：最小集（今天就可以开始）
  规划 → 编码 → 审查 → 测试 → 发布

阶段 2：专业化（当你开始做多项目）
  + 设计 → 安全 → 文档 → 性能

阶段 3：团队化（当你需要和他人协作）
  + 角色模板 → 项目级配置 → 团队共享
```
