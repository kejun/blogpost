# 混合架构的崛起：当确定性管线遇上 LLM Agent——从 Alibaba open-code-review 到 Allen AI Shippy 的生产级可靠性设计模式

> 最可靠的 AI 系统，不是最强的模型，而是最聪明的"约束"。

2026 年 7 月，两个看似不相关的项目同时登上了 GitHub Trending：阿里巴巴开源的 [open-code-review](https://github.com/alibaba/open-code-review)（11,000+ stars）和 Allen AI 团队发布的海事 AI Agent [Shippy](https://huggingface.co/blog/allenai/shippy-tech-blog) 技术博客。一个做代码审查，一个做海洋监控，领域天差地别。但当你拆开它们的架构，会发现一个惊人的共同点：**它们都不信任 LLM。**

更准确地说，它们都采用了同一种设计哲学——**把 LLM 关进确定性工程打造的笼子里**。该确定的部分，用代码保证；该灵活的部分，交给模型。这不是妥协，而是 2026 年生产级 AI 系统最核心的架构范式。

我把它叫做**混合确定性架构（Hybrid Deterministic Architecture）**。这篇文章要回答三个问题：为什么纯 LLM Agent 在生产中不可靠？混合架构到底怎么工作？以及，你该怎么在自己的系统里落地？

---

## 一、纯 LLM Agent 的三重生产困境

在讨论解决方案之前，先看清问题。

过去两年，"让 LLM 直接调用 API"几乎成了 Agent 开发的默认范式。给模型一个工具描述，让它自己决定怎么调用。听起来优雅，但在生产中会撞上三堵墙。

### 1.1 位置漂移（Position Drift）

Alibaba open-code-review 团队在内部数万开发者的使用中，最先系统性地记录了这个问题：当 LLM 被要求对代码变更生成行级评论时，**它报告的代码位置经常是错的**。不是偶尔错，而是系统性地漂移——行号偏移、文件引用错位、上下文张冠李戴。

这不是模型"笨"。代码审查要求精确到行号的定位，而 LLM 的本质是概率性文本生成。你让它同时理解代码语义、定位精确行号、生成结构化输出，这三件事对注意力机制的竞争是零和的。模型越关注"这个 bug 是什么"，就越容易丢失"这个 bug 在第几行"。

### 1.2 覆盖不完整（Incomplete Coverage）

Allen AI 在构建 Shippy 的早期原型时发现了另一个问题：当 Agent 需要查询 Skylight API（一个有数十种输入类型、嵌套过滤对象、分页游标和复杂几何输入的 API）时，**LLM 会"偷工减料"**。

具体表现：面对大型变更集，Agent 选择性地只审查部分文件；面对复杂 API，它构造出看似正确但实际错误的查询——分页格式错误导致结果被静默丢弃，几何编码错误导致查询返回空结果，过滤器类型误解导致返回错误数据。

这些错误的共同特征是：**它们看起来像成功了**。没有报错，没有异常，只有静默的数据丢失。这比崩溃更危险，因为你甚至不知道出了问题。

### 1.3 质量不稳定（Unstable Quality）

这是最隐蔽的问题。自然语言驱动的 Skill 文件（比如 Claude Code 的 Skills、Cursor 的 Rules）本质上是**用自然语言编写的"程序"**。但自然语言没有编译器，没有类型检查，没有单元测试。

open-code-review 团队发现，**微小的 prompt 变化会导致审查质量的显著波动**。同一个 Skill 文件，改一个词，输出就可能从"精准到位"变成"泛泛而谈"。你无法 debug 一个自然语言程序，因为你无法确定性地复现问题。

这三个问题指向同一个根因：**纯语言驱动的架构缺乏对执行过程的硬约束**。

## 二、混合确定性架构：把 LLM 关进笼子里

open-code-review 的核心设计哲学用一句话概括：

> **确定性工程处理"不能出错"的部分，Agent 处理"需要判断"的部分。**

这不是一个新想法——软件工程里"关注点分离"原则已经存在了半个世纪。但在 LLM Agent 的语境下，它获得了一种全新的、更紧迫的含义：**你需要明确划定"模型可以自由的边界"和"工程必须保证的底线"。**

### 2.1 确定性层：硬约束

让我们拆解 open-code-review 的确定性层做了什么：

**精确文件选择。** 哪些文件需要审查、哪些应该过滤，不由模型决定，而由工程逻辑决定。Git diff 解析、文件类型过滤、变更范围计算——这些都是确定性操作，没有理由让概率性模型来做。

**智能文件分组。** 相关文件被打包成一个审查单元。比如 `message_en.properties` 和 `message_zh.properties` 会被自动归为一组。每个分组作为一个独立的子 Agent 运行，拥有隔离的上下文。这是一个经典的**分治策略**——把一个大问题拆成多个小问题，每个小问题的上下文窗口足够小，模型的注意力不会被稀释。

**细粒度规则匹配。** 审查规则不是塞进 system prompt 里让模型"自己看着办"，而是通过模板引擎根据每个文件的特征精确匹配。Java 文件匹配 NPE 和线程安全规则，SQL 文件匹配注入规则，前端文件匹配 XSS 规则。**规则匹配是确定性的，规则执行才交给模型。**

**外部定位与反思模块。** 评论的行号定位不由生成评论的模型完成，而由一个独立的定位模块负责。生成和定位解耦，各自可以独立优化和测试。

### 2.2 Agent 层：动态决策

确定性层处理完"骨架"后，Agent 层负责"血肉"：

**场景化 Prompt。** 不是通用的"请审查这段代码"，而是针对代码审查场景深度优化的 prompt 模板。这些模板是从大规模生产数据中的工具调用轨迹分析中提炼出来的——包括调用频率分布、每个工具的重复调用率、新工具对整体调用链的影响。

**场景化工具集。** 不是给模型一个通用的工具箱（读文件、搜索、执行命令），而是一个为代码审查量身定制的工具集。这个工具集是从生产数据中"蒸馏"出来的——只保留模型真正需要的工具，去掉那些看起来有用但实际上会分散注意力的工具。

**动态上下文检索。** Agent 可以读取完整文件内容、搜索代码库、检查其他变更文件以获取上下文。这是模型真正擅长的事情——在大量信息中找到与当前问题相关的上下文。

### 2.3 效果：用 1/9 的 Token 做到更高的精度

open-code-review 团队构建了一个包含 50 个流行开源仓库、200 个真实 Pull Request、10 种编程语言的基准测试，由 80+ 名高级工程师交叉验证（1,505 个标注的 ground-truth issue）。

结果：

| 指标 | open-code-review | 通用 Agent (Claude Code) |
|------|-----------------|------------------------|
| Precision | **显著更高** | 较低 |
| F1 | **显著更高** | 较低 |
| Token 消耗 | **~1/9** | 基准 |
| 审查速度 | **更快** | 较慢 |
| Recall | 较低 | 较高 |

注意 Recall 的取舍：open-code-review 故意牺牲了一部分召回率，换取更高的精确率。这是一个**深思熟虑的工程决策**——在代码审查场景中，误报（false positive）的代价远高于漏报（false negative），因为每一个误报都需要人类开发者花时间去分诊。

## 三、Shippy：同一模式在高风险领域的验证

如果 open-code-review 是混合架构在开发者工具中的实践，那么 Allen AI 的 Shippy 就是它在**高风险操作领域**的验证。

Shippy 是一个海事 AI Agent，服务于全球 70 多个国家的政府机构和 NGO。它的工作是回答海事分析师的问题：某片海域有没有非法捕鱼活动？某艘船的行踪是否可疑？某个海洋保护区附近有没有违规操作？

**在这个领域，错误答案的代价不是"用户体验不好"，而是"巡逻船被派往错误的方向，浪费本已紧张的资源，甚至危及人员安全"。**

### 3.1 确定性 CLI：把 API 复杂度封装成可预测的接口

Shippy 的架构团队做了一个关键决策：**不让 LLM 直接构造 API 调用，而是通过一个专门构建的 CLI 来与 Skylight API 交互。**

这个 CLI 把 Skylight API 的数十种输入类型、嵌套过滤对象、分页游标和复杂几何输入，封装成简单的类型化命令。Shippy 只需要执行一条命令——`skylight events search --type fishing --region panama-eez --time-range last-month`——CLI 负责处理认证、分页和结构化输出。

CLI 的输出始终写入本地 JSON 文件，而不是通过 shell 管道传递。早期原型中，大型结果集会命中管道缓冲区限制或破坏下游工具。写入磁盘绕过了这两个问题，并让 Agent 可以在后续步骤中以编程方式访问查询结果。

### 3.2 分层测试：每一层都可以独立验证

这个分层设计——类型化 API → 确定性 CLI → Agent Skills——意味着每个组件都可以独立测试：

- **API 层**有自己的测试套件
- **CLI 层**可以由人类或 Agent 执行
- **Agent Skills 层**引用 CLI 命令，不需要重新发明轮子

**每一层都收窄了下一层可以犯错的范围。** 这是软件工程中"防御性编程"原则在 AI Agent 时代的自然延伸。

### 3.3 灵魂（Soul）：用 System Prompt 划定行为边界

Shippy 的"灵魂"——它的 system prompt——明确定义了它**不会做什么**：

- 不会对船舶是否违法做出法律判定——这是人类的决定，不是 Agent 的
- 不会超出数据支持的范围进行推测
- 每个回答都会标注数据来源、数据截止时间和查询时间戳

这些边界是**显式写在 system prompt 里的，而不是隐含在微调中的**。这使得它们可审计、可修改、可版本控制。

## 四、模式提炼：混合确定性架构的设计原则

从 open-code-review 和 Shippy 的实践中，可以提炼出一套通用的设计原则：

### 原则一：确定性优先，模型兜底

**任何可以用确定性逻辑完成的任务，都不应该交给 LLM。** 文件选择、数据格式化、分页处理、权限检查、输入验证——这些都是确定性操作。让模型做这些事，不仅浪费 token，还引入了不必要的不确定性。

判断标准很简单：**如果你能为这个任务写一个单元测试，那它就不应该由 LLM 来做。**

### 原则二：生成与定位解耦

LLM 擅长生成内容，但不擅长精确定位。open-code-review 的做法是：让模型生成评论内容，让独立的定位模块确定评论应该放在哪一行。

这个原则可以推广到很多场景：让模型生成 SQL，让确定性层验证语法和权限；让模型生成代码，让编译器检查正确性；让模型生成建议，让规则引擎检查合规性。

### 原则三：分治隔离上下文

大型任务不应该在一个巨大的上下文中完成。open-code-review 把文件分组成独立的审查单元，每个单元作为一个子 Agent 运行，拥有隔离的上下文。Shippy 把不同的数据查询封装成独立的 CLI 命令，每个命令返回结构化的 JSON。

**上下文越小，模型的注意力越集中，输出越稳定。**

### 原则四：工具集要"蒸馏"，不要"堆砌"

给模型 50 个工具，不如给它 5 个精心挑选的工具。open-code-review 的工具集是从大规模生产数据中的工具调用轨迹分析中提炼出来的——只保留模型真正高频使用的工具，去掉那些"看起来有用但实际上会分散注意力"的工具。

这与 IBM Research 在模型路由文章中的发现一致：**更多的选择不等于更好的结果。** 在路由中，更多的模型选项增加了决策复杂度；在工具集中，更多的工具增加了模型的认知负担。

### 原则五：边界要显式，不要隐式

Shippy 的行为边界写在 system prompt 里，而不是依赖微调。open-code-review 的审查规则通过模板引擎匹配，而不是塞进 prompt 里让模型"自己看着办"。

**显式边界可审计、可测试、可版本控制。隐式边界是黑箱。**

## 五、这不是"反 AI"，而是"真正的 AI 工程"

有人可能会说：混合架构是在"限制" AI 的能力。如果模型足够强，为什么还需要确定性层的约束？

这个论点在理论上成立，但在工程实践中不成立。原因有三：

**第一，概率性系统的可靠性上限不是 100%。** 即使模型的准确率达到 99%，在每天处理数万次请求的生产系统中，1% 的错误率意味着每天数百个错误。在代码审查中，这意味着每天数百个错误的行号定位；在海事监控中，这意味着每天数百个可能误导巡逻决策的错误答案。

**第二，"足够强"是一个移动靶。** 今天的 SOTA 模型明天就会被超越。如果你的架构假设"模型足够强所以不需要约束"，那么每次模型升级你都需要重新验证整个系统。而混合架构的确定性层不依赖于模型能力——**无论模型多强或多弱，文件选择、分页处理、权限检查的正确性都不会变。**

**第三，可审计性是生产系统的硬性要求。** 在金融、医疗、海事、法律等受监管领域，你需要能够解释系统的每一个决策。"模型觉得应该这样做"不是一个可接受的审计答案。确定性层提供了可追溯的决策链，而模型的推理过程是不可追溯的。

## 六、落地指南：从你的项目开始

如果你正在构建一个生产级 AI 系统，以下是混合确定性架构的落地路径：

**第一步：画出你的"确定性边界"。** 列出系统中所有可以用确定性逻辑完成的任务。文件选择、输入验证、数据格式化、权限检查、分页处理——这些都是确定性任务。把它们从 LLM 的职责中剥离出来。

**第二步：为 LLM 构建"类型化接口"。** 不要让模型直接调用原始 API。像 Shippy 的 CLI 一样，把复杂的 API 封装成简单的、类型化的命令。每个命令有明确的输入格式、输出格式和错误处理。

**第三步：分治大任务。** 把大型任务拆分成多个小任务，每个小任务在隔离的上下文中执行。open-code-review 的文件分组策略是一个好参考。

**第四步：蒸馏工具集。** 分析你的 Agent 在生产环境中的工具调用轨迹。哪些工具被高频使用？哪些工具从未被调用？哪些工具的调用总是导致错误？保留高频、高价值的工具，去掉其余的。

**第五步：显式定义行为边界。** 把系统"不应该做什么"明确写出来，而不是依赖模型的"常识"。这些边界应该是可审计、可测试、可版本控制的。

## 结语

2026 年的 AI 工程正在经历一个重要的认知转变：**从"让模型做一切"到"让模型做它最擅长的事"。**

open-code-review 用 1/9 的 token 消耗实现了更高的精确率，不是因为它用了更强的模型，而是因为它把模型不擅长的事（精确定位、文件选择、规则匹配）交给了确定性工程。Shippy 在高风险海事领域获得了用户信任，不是因为它用了最聪明的模型，而是因为它把模型的"自由裁量权"限制在了"动态判断和上下文检索"这个它真正擅长的领域。

这不是 AI 的退步，而是 AI 工程的成熟。正如数据库行业从"一个数据库打天下"走向"多模数据库各司其职"，AI 系统也在从"一个模型做一切"走向"确定性工程 + LLM 各司其职"。

**最可靠的 AI 系统，从来不是最强的模型，而是最聪明的约束。**
