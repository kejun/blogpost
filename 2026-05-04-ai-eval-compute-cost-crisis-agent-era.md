# AI 评测的计算成本危机：当评估比训练更贵，Agent 时代谁来为真相买单？

**文档日期：** 2026 年 5 月 4 日  
**标签：** AI Evaluation, Compute Cost, Agent Benchmark, HAL, Flash-HELM, tinyBenchmarks, CLEAR, 成本工程

---

## 一、引言：一个反直觉的事实

2026 年 4 月 29 日，Hugging Face 官方博客发表了一篇题为 *"AI evals are becoming the new compute bottleneck"* 的文章，揭示了一个令整个行业不安的事实：

> **AI 评估（evaluation）的计算成本已经跨越了一个关键阈值——对于某些模型和任务，评估一个 checkpoint 的成本，甚至超过了预训练这个模型本身的成本。**

这不是危言耸听。让我们先看一组数据：

| 场景 | 成本 | 来源 |
|------|------|------|
| HAL（Holistic Agent Leaderboard）单次完整运行 | ~$40,000 | Kapoor et al., ICLR 2026 |
| GAIA 单次前沿模型评估 | $2,829（缓存前） | HAL 公开账本 |
| Exgentic Agent 配置扫遍 | $22,000 | 发现 33× 成本差异 |
| The Well 评估单一新架构 | ~960 H100 小时 | Scientific ML 基准 |
| HELM 全量评估（30 模型 × 42 场景） | ~$100,000 | Stanford CRFM |

与此同时，Perlitz 等人（2024）对 EleutherAI Pythia 模型的评估发现：

> **在模型开发全周期中，评估成本"甚至可能超过预训练成本"。对于小模型，评估已成为整个开发周期中最大的计算支出项。**

这意味着，我们可能正在进入一个"评估税"（eval tax）比"训练税"更重的时代。对于一个年收入数千万美元的 AI 实验室来说，这可能只是财务报表上的一行数字。但对于独立研究者、学术团队和开源社区，评估成本的飙升正在**系统性地缩小谁有资格参与 AI 前沿竞争的圈子**。

本文将从技术角度深入剖析这场成本危机的成因、影响和可能的解决路径。

## 二、成本曲线的演变：从静态 Benchmark 到 Agent 时代

### 2.1 静态 Benchmark 的"黄金时代"

2022 年，Stanford CRFM 发布了 HELM（Holistic Evaluation of Language Models），这是 LLM 评估领域的一个里程碑。论文自带了每个模型的详细成本核算：

| 模型 | API 成本 | GPU 小时 |
|------|----------|----------|
| OpenAI code-cushman-001 | $85 | — |
| AI21 J1-Jumbo (178B) | $10,926 | — |
| BLOOM (176B) | — | 4,200 |
| OPT (175B) | — | 540 |

30 个模型 × 42 个场景的总成本约 $100,000。在当时的行业语境中，这个数字"昂贵但可接受"。

更重要的是，静态 Benchmark 有一个可以被利用的"弱点"：**模型之间的差异往往集中在数据集的一小部分样本上**。这意味着排名信息可以通过激进的数据子采样来大幅压缩。

一系列论文证明了这一点：

- **Flash-HELM**（Perlitz et al., 2023）：通过粗到精的两阶段评估流程，实现 **100×~200× 的计算量缩减**，同时保留了几乎相同的排名顺序。
- **tinyBenchmarks**（2024）：利用项目反应理论（Item Response Theory），将 MMLU 从 14,000 题压缩到 100 个锚点题，排名误差仅约 2%。
- **Anchor Points**：仅用 1~30 个样本就能对 87 个语言模型/提示对进行 GLUE 排名。
- **Open LLM Leaderboard**：从 29,000 个示例压缩到 180 个。

这些方法的核心洞察是：**静态 Benchmark 的信息密度极低，大量计算在确认一个社区早就知道的事情**。

### 2.2 Agent 时代：为什么压缩不再有效？

当评估对象从"单次输入-输出对"变成"多轮 Agent 交互"时，情况发生了根本性变化。

Holistic Agent Leaderboard（HAL, Kapoor et al., ICLR 2026）提供了一个非常透明的公共账本。截至 2026 年 4 月，HAL 已执行 **26,597 次 Agent rollout**，覆盖 9 个模型和 9 个基准任务。

关键发现如下：

**1. 单次运行成本跨越四个数量级**

HAL 中，最便宜的单次 benchmark 运行和最贵的相差 **10,000 倍**。即使在同一个 benchmark 内部，不同配置的成本差异也达到 **1,000 倍**。

**2. Scaffold 是第一阶成本驱动因素**

Exgentic 在 $22,000 的 Agent 配置扫遍实验中发现：

> **相同任务、相同模型、不同 scaffold 的组合中，成本差异最高达 33 倍。**

这意味着：Agent 评估的不是"模型"，而是 **模型 × scaffold × token-budget 的复合产物**。一个糟糕的 scaffold 选择可以让成本飙升一个数量级，而精心的设计可以大幅降低开支。

**3. 更高的花费 ≠ 更好的结果**

HAL 论文中有一组令人在意的对比：

| 配置 | 成本 | 准确率 | 性价比 |
|------|------|--------|--------|
| Browser-Use + Claude Sonnet 4 (Online Mind2Web) | $1,577 | 40% | 极差 |
| SeeAct + GPT-5 Medium (Online Mind2Web) | $171 | 42% | 优秀 |
| HAL Generalist + o3 Medium (GAIA) | $2,828 | 28.5% | 极差 |
| 另一种 Agent (GAIA) | $1,686 | 57.6% | 良好 |

更令人震惊的是 CLEAR 研究的发现：

> **在 6 个 SOTA Agent 执行 300 个企业任务的评估中，"准确率最优配置"的成本比帕累托最优配置高出 4.4~10.8 倍，而实际性能差异微乎其微。**

### 2.3 定价鸿沟：API 价差是成本危机的放大器

Agent 评估的成本还受到模型定价策略的深刻影响。当前的 API 定价存在 **两个数量级** 的差距：

| 模型 | 输入价格（$/M tokens） | 输出价格（$/M tokens） |
|------|------------------------|------------------------|
| Claude Opus 4.1 | $15 | $75 |
| Gemini 2.0 Flash | $0.10 | $0.40 |
| 价差倍数 | 150× | 187.5× |

考虑到 Agent 任务通常需要数万到数十万 token 的上下文窗口，模型选择本身就决定了成本的数量级。这不仅是技术问题，也是经济学问题。

## 三、Agent 评估的"脏数据"本质

### 3.1 为什么 Agent Benchmark 不可压缩？

静态 Benchmark 的压缩之所以有效，是因为每个样本是独立的、确定性的、低方差的。但 Agent 评估有三个根本不同的特征：

```
静态 Benchmark vs Agent Benchmark 的对比

┌─────────────────────┬─────────────────────────────────┐
│ 维度                │ 静态 Benchmark                  │ Agent Benchmark                   │
├─────────────────────┼─────────────────────────────────┤
│ 样本独立性          │ 高 — 每题独立                   │ 低 — 多轮交互，状态耦合            │
│ 输出方差            │ 低 — 同一模型多次运行结果稳定   │ 高 — 温度采样、网络延迟、环境变化  │
│ 评估粒度            │ 单次推理                        │ 多次 rollout 才能收敛             │
│ Scaffold 敏感度      │ 无 — 直接 prompt                │ 极高 — 不同 scaffold 差异巨大     │
│ 可压缩性            │ 100×~200×                      │ 2×~3.5×（当前最优）               │
└─────────────────────┴─────────────────────────────────┘
```

Ndzomga 的 mid-difficulty filter（选择历史通过率在 30%~70% 之间的任务）是目前对 Agent Benchmark 最有效的压缩方法，实现了 **2×~3.5× 的成本缩减**。这很有用，但远远无法企及静态 Benchmark 的 100×~200× 的压缩率。

根本原因在于：

> **当每个评测项本身就是一个多轮 rollout 且自带方差时，昂贵的对象不再是"大量样本"，而是"单个样本的长轨迹"。**

你无法通过减少样本来显著降低成本，因为每个样本本身就是成本的主要来源。

### 3.2 训练即评估：The Well 案例

有些评估的成本框架完全不同于 API 调用——因为评估协议本身就是一个训练过程。

**The Well**（Arxiv 2412.00568）是科学机器学习领域的一个标志性基准。它捆绑了 16 个分子和材料科学的数据集，评估协议要求：

- 评估一个**新架构**：约 **960 H100 小时**
- 完整四基线扫描：约 **3,840 H100 小时**

这里的"评估"实际上是"从头训练模型到收敛，然后比较性能"。这本质上是一个完整的 ML 实验流程。对于这类基准，讨论 API 成本毫无意义——它们消耗的是 **GPU 集群的独占时间**。

更广泛地说，任何 **training-in-the-loop** 的评估方法（比如用 RL 微调 Agent 后再评估）都天然昂贵。UK-AISI 最近将 Agentic 步骤扩展到**百万级**来研究推理时计算扩展效应（inference-time compute scaling），其评估成本本身就构成了一个研究预算量级的问题。

## 四、成本危机的连锁影响

### 4.1 对开源生态的系统性挤压

当评估成本达到单次 $40,000 的量级时，影响不再是线性的：

1. **小团队无法独立复现评估结果** → 他们只能信任大厂或大型实验室的评估数据 → 评估变成了另一种形式的"算力垄断"
2. **开源模型的竞争力被低估** → 因为完整的 benchmark 运行太贵，开源模型往往只被评估在少数几个"便宜"的任务上 → 这些任务可能恰好不是它们的强项
3. **研究方向的扭曲** → 研究者会倾向于选择"评估成本较低"的研究方向，而不是"最重要的"方向 → 评估成本本身成为了隐形的研究议程设定者

### 4.2 对模型开发的反馈循环

Perlitz 等人的研究揭示了一个关键模式：开发者在模型开发过程中**反复支付评估费用**。Pythia 项目为每个模型的 16 种尺寸各发布了 154 个 checkpoint，总计 2,464 个 checkpoint。如果每个 checkpoint 都要跑完整的评估流程，评估成本将是一个天文数字。

这创造了一个危险的反馈循环：

```
评估太贵
  → 减少评估频率
    → 基于更少的数据做决策
      → 模型质量提升变慢
        → 需要更大的模型来补偿
          → 评估变得更贵
```

打破这个循环的关键在于：**让评估变得更智能，而不是更频繁**。

### 4.3 对 Agent 开发的特殊影响

对于 Agent 系统，成本问题还有一个独特的维度：**scaffold 的迭代成本**。

一个 Agent scaffold 的改进（比如改进了 tool selection 逻辑或添加了自我反思步骤）需要重新运行完整的评估才能验证。如果一次评估需要 $40,000，那么 scaffold 的迭代速度就被锁死在"我们能承担多少次评估"的约束下。

Exgentic 的 $22,000 扫遍实验已经说明，scaffold 选择可以带来 33× 的成本差异。但如果每次 scaffold 迭代都需要重新做完整的 benchmark，大多数团队根本负担不起这种探索。

## 五、解决路径：成本感知的评估工程

### 5.1 多保真度评估（Multi-Fidelity Evaluation）

Flash-HELM 的核心思想可以扩展到 Agent 评估：

```
多保真度 Agent 评估流程

第一阶段：快速筛选（低成本，低保真度）
├── 使用便宜的模型（如 Gemini 2.0 Flash）作为 proxy
├── 在每个 benchmark 上运行少量样本（~10%）
├── 只保留排名靠前的候选配置
└── 成本：原始成本的 1%~5%

第二阶段：精细评估（中等成本，中等保真度）
├── 使用目标模型
├── 运行 30%~50% 的样本
├── 重点测试第一阶段筛选出的 top-N 配置
└── 成本：原始成本的 10%~30%

第三阶段：最终验证（全成本，高保真度）
├── 仅对最终候选配置运行完整评估
├── 包含可靠性维度（多次 rollout 测方差）
└── 成本：原始成本的 1×（但只跑 1~2 个配置）

总成本：原始成本的 3%~10%
```

这种方法的核心假设是：不同保真度之间的排名相关性足够高。对静态 Benchmark，这个假设已被证明成立。对 Agent Benchmark，Ndzomga 的 mid-difficulty filter 提供了初步支持——虽然压缩率只有 2×~3.5×，但排名保真度在不同 scaffold 和时间偏移下得到了保持。

### 5.2 成本-性能帕累托前沿

CLEAR 研究给出的启示是：**不应追求单一的"最高准确率"，而应追求成本-性能帕累托前沿**。

在工程实践中，这意味着：

1. **为每个 Agent 任务设定预算上限**：不是"能花多少花多少"，而是"在预算内最大化性能"
2. **报告成本-性能曲线**：不仅报告准确率，还报告达到该准确率所需的成本
3. **使用帕累托效率作为选择标准**：在准确率相近的配置中，选择成本更低的

具体来说，可以引入 **"每百分点准确率的美元成本"（$/point）** 作为核心指标：

| 配置 | 成本 | 准确率 | $/point |
|------|------|--------|---------|
| Browser-Use + Sonnet 4 | $1,577 | 40% | $39.43 |
| SeeAct + GPT-5 Medium | $171 | 42% | $4.07 |

后者的性价比是前者的 **近 10 倍**。在生产环境中，这意味着用同样的预算可以运行 10 倍的评估，或者用 1/10 的成本获得可比的结果。

### 5.3 开源评估基础设施

成本危机的最终解法可能是 **评估基础设施的公共化**：

- **HAL 模式**：集中式的评估平台，通过规模效应降低单次评估的边际成本
- **标准化 scaffold**：社区维护的"参考 scaffold"，减少因 scaffold 选择带来的成本差异
- **共享评估结果**：一旦某个模型 × scaffold 组合的评估结果被产出，就应该被社区共享，避免重复花费

UK-AISI 的百万级 Agentic 步骤研究已经表明，**大规模评估本身就是一种科学研究**。当评估的规模大到成为研究产出时，它应该被视为公共基础设施，而不是私人成本。

### 5.4 评估维度的选择性聚焦

不是所有评估维度都值得同样的投入。一个实用的框架：

```
评估优先级矩阵

                    高方差           低方差
               ┌──────────────┬──────────────┐
    高业务价值 │  重点投入     │  快速验证     │
               │ （需要多次    │ （少量样本    │
               │  rollout）   │  即可）       │
               ├──────────────┼──────────────┤
    低业务价值 │  降优先级     │  跳过/最小化  │
               │ （或仅用      │              │
               │   proxy模型）│              │
               └──────────────┴──────────────┘
```

例如：
- **高业务价值 × 高方差**：Agent 在真实 web 导航任务中的表现（Online Mind2Web）——需要多次 rollout 来建立置信区间
- **高业务价值 × 低方差**：Agent 的代码生成正确率（SWE-bench）——相对确定性高，少量运行即可
- **低业务价值 × 高方差**：一些学术 benchmark 的长尾任务——可以降低优先级或使用 proxy 模型

## 六、深层反思：我们到底在评估什么？

### 6.1 评估的元问题

当评估成本本身成为瓶颈时，我们需要回答一个更根本的问题：**我们是否在用正确的方式评估 Agent？**

当前的 Agent 评估范式本质上是 **"让 Agent 做一堆任务，看它做对了多少"**。这个范式有三个假设：

1. 任务的集合是完备的（覆盖了 Agent 应该能做的一切）
2. 任务的难度分布是合理的
3. 单次评估的结果是可靠的

但 HAL 的数据已经表明，这些假设都存在问题：

- 同一个 Agent 在不同 benchmark 上的表现差异巨大（说明任务集合不完备或不均衡）
- scaffold 选择对成本的影响达到 33 倍（说明评估的不是模型本身，而是模型+scaffold 的组合）
- 同样的配置多次运行结果不同（说明单次评估的可靠性存疑）

### 6.2 从"评估模型"到"评估系统"

Agent 评估的核心挑战之一是：**你评估的从来不是"模型"，而是"整个系统"**。

一个 Agent 系统的性能取决于：

```
Agent 系统性能 = f(
    基础模型能力,          ← 这是传统评估关注的
    Scaffold 设计,          ← 被严重低估
    Tool 接口质量,          ← 几乎不被评估
    上下文管理策略,         ← 几乎不被评估
    Error handling 逻辑,    ← 几乎不被评估
    Token 预算分配,         ← 几乎不被评估
    环境噪声和网络延迟,     ← 完全不被控制
    评估时的温度参数设置     ← 很少被报告
)
```

当评估成本如此之高时，我们更应该**理解每个因素对最终性能的贡献度**，而不是简单地比较"系统 A 比系统 B 好多少"。

### 6.3 成本透明度的缺失

HAL 的一个突出贡献是它公开了每次评估的成本账本。但大多数评估（包括学术界的 benchmark）根本不报告成本。

这意味着：

- 论文中声称的 "SOTA 性能" 可能是在极高的成本下实现的
- 读者无法判断这个 SOTA 是否有实际工程价值
- 社区无法复现结果（因为不知道需要多少预算）

我们建议：**所有 Agent 评估论文都应该强制报告成本信息**，包括：

- 每次 rollout 的平均 token 消耗（输入/输出）
- 使用的模型和对应的 API 价格
- 总成本（以美元或等效 GPU 小时计）
- 成本-性能帕累托分析（如果评估了多个配置）

## 七、展望：成本约束下的 Agent 评估新范式

### 7.1 短期可做的

对于正在做 Agent 开发和评估的团队，以下措施可以立即落地：

1. **采用多保真度评估流程**：先用低成本 proxy 模型筛选，再用目标模型精细评估
2. **报告成本-性能双指标**：不仅报告准确率，还报告成本
3. **聚焦帕累托最优**：选择"性价比最高"而非"绝对性能最高"的配置
4. **利用 mid-difficulty filter**：聚焦最有区分度的任务子集
5. **共享评估结果**：建立团队内部的评估结果数据库，避免重复花费

### 7.2 中期需要社区协作的

1. **标准化参考 scaffold**：减少因 scaffold 选择带来的成本差异
2. **开源评估平台**：类似 HAL 的集中式评估基础设施
3. **成本透明化规范**：所有评估论文强制报告成本信息
4. **评估维度的精简**：确定哪些评估维度值得投入，哪些可以降优先级

### 7.3 长期愿景

最终，我们可能需要一个根本性的范式转变：**从"评估模型的性能"转向"评估模型的能力边界"**。

如果我们能更精确地理解一个 Agent 在哪些条件下会成功、在哪些条件下会失败、失败的模式是什么，我们就不需要为每个任务都运行昂贵的完整评估。我们可以用更少的评估，获得更深的理解。
