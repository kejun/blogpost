## 正片

**主持人**：  
大家好，欢迎回到 No Priors。今天我和 Andre Karpathy 有一场非常广泛的对话，内容包括：代码智能体、工程的未来、AI 研究、更多人如何参与研究、机器人领域正在发生什么、他对 agent 如何触达真实世界的预测，以及这个新时代下的教育。欢迎你，Andre。

**Andre**：  
谢谢邀请。

**主持人**：  
最近几个月 AI 真的非常刺激。

**Andre**：  
可以这么说。

**主持人**：  
我记得有次走进办公室，你整个人都“锁死”在工作里。我问你在做什么，你说：  
“我每天要写 16 小时代码——不，‘写代码’都不是准确动词了。我得把我的意志表达给我的 agents，让它显化。”  
因为能力出现了跳变。到底发生了什么？说说你的体验。

**Andre**：  
我感觉自己一直处在一种“AI 精神错乱感”里，到现在也经常是这种状态。  
因为作为一个个体，你能做到的事出现了巨大解锁。以前你受限于打字速度这类瓶颈；但有了这些 agents 之后，我会说大概在 12 月，某个东西突然翻转了：我从“80/20（自己写代码/委托 agents）”变成了“20/80”。而且现在甚至可能比 20/80 更极端。  
从 12 月开始，我几乎就没再亲手敲过代码。这个变化非常大。  
我甚至跟我爸妈聊过，我觉得普通人并没有真正意识到这件事已经发生，更没有意识到它有多戏剧性。  
你随便找一个软件工程师，看他现在在工位上的默认工作流——从 12 月起几乎已经完全不同了。  
所以我一直处在一种“精神错乱感”里：到底还有什么可能？怎样把它推到极限？我怎样才能不只开一个 Claude Code 或 Codex 的会话，而是开更多？如何正确地做这件事？这些 claws 到底是什么？  
新东西太多了。我想站在最前沿；而我又很焦虑自己不够前沿。我在 Twitter 上看到很多人做各种事，听起来全是好点子，我就会觉得我也必须在最前沿，否则会非常紧张。  
所以，本质上我就是处在这种“可能性还没被探索完”的精神状态里。

**主持人**：  
如果你都紧张，那我们其他人更紧张。  
我们在 Conviction 合作的一支团队，工作方式是：工程师几乎不再手写代码，大家都戴着麦克风，全天对着 agents 低声下指令。那是我见过最奇怪的工作环境。  
我一开始觉得他们疯了，现在我完全接受了，我会想：“哦，这才是路径。只是他们更早。”  
你现在怎么看自己的能力边界？比如探索能力、做项目能力，现在到底受什么限制？

**Andre**：  
受什么限制？我觉得几乎什么都限制。  
很多事情即便没做成，你也会觉得这是“技能问题”，而不是“能力不存在”。你会觉得：只是我还没找到把现有能力正确串起来的方法。  
比如可能是我在 agents.md 里指令写得不够好；或者我没有把足够好的记忆工具接进去。  
所以当系统失败时，你很容易感觉“是我不够会用”。  
你会想怎么并行化它们，等等。你会想成为 Peter Steinberg。  
Peter 很有名，有一张很有意思的照片：他站在显示器前，屏幕里全是 Codex agent 窗口。每个窗口如果你 prompt 正确、开 high effort，大约会跑 20 分钟。  
他会同时 checkout 多个仓库，来回给它们派活。  
这时你对仓库的操作不是“加一行代码”“写一个函数”，而是“给 agent1 一个完整新功能”“给 agent2 一个不冲突的新功能”，然后你再尽可能审查它们的结果（审查到什么程度取决于你对那段代码有多在乎）。  
同时，另一个 agent 在做研究，另一个在写代码，另一个在做实现计划。  
仓库上的一切都在“宏动作”层面推进。你在训练的是这种操作的肌肉记忆。  
这非常有回报，第一是它真的有效；第二是这是全新的技能学习。  
所以这就是为什么会出现那种 psychosis 状态。

**主持人**：  
我也有同样的直觉：每次我在等一个 agent 完成任务，最显然的下一步就是“我还能再派更多活”。  
如果我还有 tokens，就该继续并行化、多开任务。  
这会很有压力，因为如果你不太受 token 花费约束，那系统里的瓶颈就是你自己——你没有把整体能力拉满。

**Andre**：  
是的，至少你没有把订阅价值榨干。  
理想上是多 agent、多平台联动：比如 Codex 的额度用完了就切 Claude，或者别的。我最近也在这样试。  
只要我看到订阅余额还剩，我就会紧张：这说明我没有最大化 token 吞吐。  
这其实很像我读博时的感受：GPU 没跑起来你就会焦虑。你有算力，却没把可用 FLOPs 吃满。  
现在从 FLOPs 变成了 tokens。  
所以问题变成：你的 token 吞吐是多少？你能调度多大吞吐？  
我还觉得很有意思的一点是，过去很多工程任务中，人们并不觉得自己被算力卡住。  
现在整个行业都在体感“资源约束”。  
当能力跃迁后，你会发现：真正的约束不再是“拿不到算力”，而是“我自己就是瓶颈”。

**主持人**：  
这是一种技能问题，同时也让人有掌控感，因为你还能继续变强。  
也许这就是它很上瘾的原因：你一变强就会出现新解锁。  
那你觉得它会走到哪里？如果你和大家都每天 16 小时练 coding agents，一年后达到“精通”，那会是什么样？

**Andre**：  
我觉得大家都在“往栈的上层走”。  
首先不再是“单个 agent 会话”，而是“多个 agents 如何协作、如何组队”。大家都在摸这个形态。  
其次，claw 这个方向也很有意思。  
我说 claw 的意思，是那种把“持久性”带到新层级的系统：它会持续循环；你不用一直在交互中盯着；它有自己的沙箱；你不看它也会替你做事。  
另外它可能还有更复杂的记忆系统（而这些在很多默认 agents 里还没实现）。  
OpenClaw 的记忆能力就比默认 agent 强不少；默认往往只是 context 用完之后做一次 memory compaction。

**主持人**：  
你觉得 OpenClaw 里最让用户有共鸣的是这块（记忆/持久性），还是更广泛的工具接入？

**Andre**：  
我觉得里面有很多特别好的想法。Peter 做得非常好。  
我最近见过他，聊过这些。他人很谦虚，但我认为他在多个方向上同时创新并把它们整合起来了。  
比如 SOUL.md 文档：他确实刻意塑造了一个有吸引力的人格。  
我认为很多当下 agent 工具并没有把“人格”这件事处理好。  
我觉得 Claude 的人格就比较好，感觉像个队友，会跟你一起兴奋。  
而 Codex（作为 coding agent）更“干”。有点有趣的是，在 ChatGPT 里 Codex 的风格会更 upbeat、更活跃；但在 coding agent 形态下它显得很冷，只像在说“我做完了”。  
但你会想：你到底理解我们在造什么吗？  
另外，Claude 在“讨好”程度上的阈值我觉得也调得比较好：  
它不是无脑夸我。比如我给它一个还没成形的想法，它不会很夸张地反应；但当我自己也判断这个点子真的好时，它会更明显地正反馈。  
所以我会有一种“想赢得它表扬”的心理，这很奇怪。  
因此人格真的很重要。很多工具在这方面可能重视不够。Peter 在这件事上很在意，这点是对的。  
再加上记忆系统，以及“用单一 WhatsApp 门户接入全部自动化”的设计，整体就很强。

**主持人**：  
你个人有没有把 claws 用到软件工程之外？有没有什么你觉得有趣的例子？

**Andre**：  
有。1 月我有一段“claw psychosis”时期。  
我做了一个管理我家的 claw，叫它 Dobby（家养小精灵 claw）。  
我让 agents 去找出我家局域网里所有智能家居子系统。  
让我震惊的是，这几乎是开箱即用。我只说“我家应该有 Sonos，你去找一下”。  
它就做了局域网 IP 扫描，真的找到了 Sonos。  
结果发现几乎没有额外密码防护，它就连进去了，然后说“我看到你装了这些 Sonos，我来逆向它怎么工作”，接着去 web 搜索，找到 API endpoints。  
它问我要不要试，我当时都愣了。  
我说“你能不能在书房放点音乐？”  
它就放了，音乐真的出来了。  
我当时想：这也太夸张了，三条 prompt 左右。  
它对灯光系统也做了同样的事。  
总之，它把这些系统都“打通”了：做了 API，做了 dashboard，让我看到全屋灯光控制中心，然后它可以开关灯。  
我可以说“Dobby，睡觉时间”，它就把灯都关了。  
它还控制我的 HVAC、窗帘、泳池和 spa，以及安防系统。  
我有一个朝屋外的摄像头，外面有人出现时，先触发变化检测，再把画面送给 Quinn 模型（转写如此）分析，然后它会发 WhatsApp 给我：附一张外部图片，并告诉我“有一辆 FedEx 车刚到，你可能有邮件或包裹”。  
Dobby 这样给我发消息，真的很不可思议。  
所以现在 Dobby 在管理我的房子，我通过 WhatsApp 和它交互。  
这种“宏动作”维护房子体验非常有趣。  
我还没把它推到更疯狂的边界，我知道很多人做得更激进。  
但对我来说，仅仅把原来 6 个完全不同的 app 统一起来，我就已经不想再用那些 app 了。  
Dobby 用自然语言控制一切，真的很棒。  
我觉得我还没把这个范式推到极限，但它已经非常有帮助、很有启发。

**主持人**：  
你觉得这是否说明了用户体验层面的真实需求？  
因为人们常常忽略“学习新软件/新 UI”本身要花大量人类精力。

**Andre**：  
某种程度上是。  
这有点像从“人们心中 AI 应该是什么”反推系统形态。  
用户脑中想的 AI，不是“原始 LLM（token 生成器）”那种形态；他们想的是一个有身份的人格体，你告诉它事情，它会记住，它像是 WhatsApp 背后的一个实体，这样更好理解。  
但底层其实有很多技术细节。  
LLM 作为原语太“生”了，对很多人来说还不够“像 AI”。

**主持人**：  
我也同意。把它命名为 Dobby、给它人格，确实更容易被理解。  
而且你把原来 6 套软件体验统一了，这会让人反过来问：  
我们真的需要今天这么多软件吗？  
你保留了硬件，但几乎抛弃了它们原来的软件层和 UX 层。  
这是不是才是用户真正想要的？

**Andre**：  
我确实有这种感觉：App Store 里很多智能家居 app，在某种意义上根本不该存在。  
是不是应该只暴露 API，再让 agent 直接调用？  
因为很多跨系统自动化，单个 app 做不到，但 LLM 可以把工具链串起来，做复杂操作。  
所以这确实指向一个方向：我们可能过度生产了很多定制 app，这些 app 会被 agents“揉碎重组”；更合理的是 API endpoint 暴露，agent 作为智能胶水连接全部工具。  
再举个例子：跑步机也有 app。我想追踪有氧频率，但我不想登录某个 web UI 再走一套流程。  
这些都应该 API 化。  
这其实就是往 agentic web、agent-first tools 的方向走。  
行业需要大规模重构：客户不再只有人，也包括代表人行动的 agents。  
这个重构会很大。  
有人会质疑：普通人要不要自己 vibe code 才能用这些？  
我觉得这是“当下阶段”而非“终局形态”。  
今天确实需要一点 vibe coding，而且我也要盯着系统。  
但我感觉一两三年后，这些能力会变成基础能力，甚至开源模型都能轻松做。  
从非技术用户意图到可执行系统，会非常容易转换。  
今天还得人工做一些设计决定（比如抽帧策略），但门槛会持续下降。  
最终软件会更“临时可塑”，某种 claw 在后台处理细节，你本人不需要介入。  
它有自己的机器，会自己搞定，然后把 UI 呈现给你。你只需要表达意图。

**主持人**：  
那你为什么没有把 claws 的个人实践边界推得更远？是因为你在做更重要的项目（比如 autoresearch）？还是你在爬“精通曲线”？还是别的？

**Andre**：  
我觉得自己就是被太多事情分散了。  
我在 claws 上密集花过一周，而且还有很多想做的。  
但最近很多工具本身都很忙。  
另外，我并没有给它太多敏感权限，比如邮箱、日历等。因为它还很新、还比较粗糙，我在安全和隐私上仍然比较谨慎，不想现在就给它我整个数字生活的完全访问权。  
所以有一部分原因是这个。可能这是主因。  
另一部分就是注意力分散：我刚沉浸一周 claws，别的事情又涌上来了。

**主持人**：  
你之前提过很久：希望 agents 能直接承担“训练/优化模型”。Auto Research 的动机到底是什么？

**Andre**：  
我之前发过一条推文，核心是：要最大化利用现在这波工具，你必须把自己从瓶颈位移走。  
你不能永远在环里负责下一个 prompt。你要把自己拿出去。  
你要把系统组织成完全自治。  
核心是：怎样最大化 token 吞吐、尽量不在环里。  
我当时说“现在游戏的名字是提高杠杆率”：我偶尔投入很少 token，但系统代表我做大量工作。  
AutoResearch 就是这个逻辑的一个直接后果。  
很多人喜欢那条推文，但可能还没推演完它的含义。  
对我来说，AutoResearch 的含义是：我不应该作为研究员一直盯在环里看结果，那样我反而拖慢系统。  
问题变成：我如何重构抽象，让系统一次配置后就能持续跑。  
目标就是：让更多 agents 在更长时间里不用我介入，持续替我做事。  
AutoResearch 的形式就是：给目标、给指标、给边界（可做与不可做），然后开跑。

**主持人**：  
而且你当时其实也对效果感到惊讶。

**Andre**：  
是的，我原本不太预期它会这么有效。  
我有个 data chat 项目。很多人会困惑我为什么一直痴迷训练 GPT-2 这种模型。  
对我来说，训练 GPT 模型只是一个小型 harness，一个 playground。  
我更关心的是“递归自我改进”：LLM 在多大程度上可以改进 LLM。  
我觉得所有前沿实验室都在做这个，而且很合理。  
所以这对我来说是一个小型实验场。  
我已经按传统方式手动调过很多次了。  
我做研究二十年，有一种“经过验证的自信”。  
我已经做了大量实验、超参数调优，都是我熟悉的工作流。  
我觉得仓库已经算调得比较好了。  
然后我让 Auto Research 跑了一夜，它找到了我没看到的调优点：  
比如我确实漏了 value embeddings 的 weight decay，Adam betas 也没调到足够好。  
而且这些项是联动的：你调一个，另一个也可能要变。  
所以我不该成为瓶颈。  
我不该自己跑超参搜索，也不该自己盯结果。  
在这种任务里有客观指标，你就该把系统组织成能一直跑下去。  
这还只是一个单回路版本。  
而前沿实验室有成千上万 GPU，所以你很容易想象他们在小模型上大规模自动化探索，再做外推。  
说到底，前沿智能很多工作都依赖外推与 scaling loss：先在小模型上探索，再外推到大模型。

**主持人**：  
也就是说，如果实验自动化做得更好，研究效率会提升，而且规模化方向会更准？

**Andre**：  
是的。  
我觉得最有意思、也可能是前沿实验室正在做的，就是：在小模型上尽可能自治，把研究员从执行环路里拿掉。  
研究员常常“过度自信”，他们可以贡献想法，但不该亲自执行每个实验。  
可以有一个想法队列：自动科学家基于论文和 GitHub 仓库提出想法；研究员也可提交想法。  
然后由执行 worker 拉任务去试。  
有效结果进入 feature branch，再由少量人看着，择机 merge 到 main。  
核心就是尽可能拿掉人，提升 token/s 吞吐。  
这要求重构全部抽象和流程。  
我觉得这非常令人兴奋。

**主持人**：  
再递归一步：模型什么时候会写出比你更好的 program.md？

**Andre**：  
你这个点很好。  
program.md 本质就是我对“auto researcher 应该怎么工作”的粗糙描述：先做这个，再做那个，优先看哪些方向（架构、优化器等）。  
我只是用 markdown 写下来。  
你完全可以想象，不同 program.md 会带来不同进展。  
某种意义上，每个研究组织都可以由一组 program.md 定义。  

**主持人**：  
对，研究组织就是一组 markdown 文件：定义角色和协作关系。  
你甚至可以比较组织设计：有的减少晨会（因为没用），有的更激进，有的更保守。  
你还能并行多个研究组织。既然全是代码，就能调参优化。  
我之前给你发过一个比赛想法：让大家提交不同 program.md，在同样硬件下比较改进幅度。  
再把这些数据喂给模型，让模型写出更好的 program.md。

**Andre**：  
完全同意。  
我们一定能得到更好的版本。  
你可以分析改进来自哪里，再反推 program.md：如何提高“这类有效改进”的出现频率；也可以分析失败类型。  
这就是元优化，完全可行。  
只是现实往往要一层层来：先一层流程，再二层，再三层。  
像剥洋葱。  
现在 LLM 层被默认了，agent 层被默认了，claw 层也逐渐被默认了；接着是多 claw 协作，再到给它们指令，再到优化指令本身。  
信息量太大。  
但这也是 psychosis 的来源：它看起来像无限，而且一切都仍然是 skill issue。  
所以我才会觉得这件事疯狂。

**主持人**：  
如果只是诊断当下时刻：什么是“关键技能”？  
你现在描述的 loop——定义指标、让 agents 在你不在环时持续优化——看起来可以迁移到很多领域。  
那我们还需要性能工程吗？

**Andre**：  
我会先给两个 caveat。  
第一，这套范式极其适合“有客观、易评估指标”的任务。  
比如写更高效的 CUDA kernel，行为完全一致但速度更快，这就是完美适配。  
所以很多 auto research 任务都很适配。  
但也有很多任务不适配：如果无法评估，就做不了 auto research。  
第二，我们虽然知道下一步方向，但系统现在仍然“接缝处在开裂”：有裂缝、并非完全顺滑。  
如果你过度超前，整体反而可能没用。  
模型虽然进步很大，但仍然毛刺明显。  
我常常同时感觉自己在和“极聪明、做了一辈子系统编程的博士生”以及“一个 10 岁小孩”对话。  
这种 jaggedness 很怪。  
人类也有能力不均，但没这么锯齿化。  
agent 的锯齿更明显：有时你要一个功能，它给你完全错的东西，然后你们进入错误循环。  
我现在仍然经常被 agents 惹恼，因为你确实感到它很强，但它偶尔会做非常荒谬的事。

**主持人**：  
我也会很生气，尤其是它在明显错误上浪费大量算力时——它本该一眼识别。

**Andre**：  
我猜底层原因之一是：这些模型本身也是在强化学习范式下被优化，而它们也受同样约束。  
实验室最容易改进的是“可验证、有奖励”的任务：程序对不对、单测过不过。  
但它们在一些“更软”的事情上会弱，比如我真正想要的语义细节、什么时候该先问澄清问题。  
所以你会感觉：  
要么你在 rails 上，进入超强轨道；  
要么你不在可验证域里，系统就开始漂。  
再比如，如果你今天问最先进模型“讲个笑话”，你知道会得到什么吗？  
那个笑话（模型最爱说的）是：“科学家为什么不相信原子？因为原子会 make everything up（构成一切/编造一切）。”  
这个笑话三四年前是它，今天还是它。  
这很说明问题：  
模型在 agentic 任务上能跑很久、替你搬山；  
但笑话还是五年前那个。  
因为这个维度不在主要 RL 优化轨道里。  
这就是 jaggedness 的一部分：  
按直觉模型更强了，笑话也该更好或更多样，但因为没被优化，就卡住。

**主持人**：  
这是不是意味着我们并没有看到那种“广义泛化”？  
比如代码能力上升，并不会自动带来笑话能力上升。

**Andre**：  
对，我认为存在一定解耦。  
有些东西可验证，有些不可验证。  
实验室会优先优化可验证部分，有些维度被优化、有些没有。  
有些研究团队的前提是：可验证领域变强后，会在一切领域变强。  
但“笑话现象”显示并不完全如此。  
我不认为那件事在令人满意的程度上发生了。  
可能有一点，但不够。

**主持人**：  
当然人类也有类似锯齿：数学很强的人也可能讲很差的笑话。

**Andre**：  
确实。  
但它仍说明“模型增强会免费覆盖所有智能维度”的叙事并不完全成立。  
有盲区、有没被优化的区域。  
而这些都被卷在一个不透明神经网络里：  
你在轨道上时就是光速推进；不在轨道上就漂。  
所以虽然方向看起来清晰，但现在还不能完全放手。  
系统还不够稳。  
当然也可能是我们还没学会用。  
所以到底是模型问题还是 skill issue，有时很难分清。

**主持人**：  
我问个有点“异端”的问题：  
如果这种锯齿持续存在，而我们看到的是一个单体接口、单模型，这样合理吗？  
要不要拆成多个可分别优化的智能域？

**Andre**：  
你是说，把系统更明确地拆成多个专家？

**主持人**：  
对，而不是全都塞在一个你看不到内部的黑盒里。  
不然用户会觉得：为什么它这件事这么强、那件事这么弱？

**Andre**：  
我的感受是，目前实验室还在追求“单一模型单一文化”：希望一个模型在所有域都强，把一切塞进参数。  
但我确实认为长期会出现更多“分化（speciation）”。  
生物界大脑高度多样，各有生态位。  
有些动物视觉皮层很强，有些是别的模块更强。  
AI 也应该出现类似分化。  
不一定要一个全知神谕。  
你可以有更小但认知核心仍完整的模型，再在特定任务上专化。  
这样在你关心的场景里，延迟和吞吐都更好。  
比如数学证明场景（如 Lean）已经有一些明显定向优化发布。  
所以这种解耦会出现。  

**主持人**：  
我还有个问题：算力基础设施的容量压力，会不会推动这种分化？  
因为效率变得更关键。你不可能给每个场景都上最大模型。这个问题成立吗？

**Andre**：  
成立。  
但我纠结的是，我们还没看到太多真实分化。  
目前还是在做单模型主线。  
即便有压力，大家也常常是“把代码能力做强，然后并回主模型”。  
也许你说得对，短期供给紧张可能会推动更多分化。  
从实验室视角，他们面对的是不确定用户问题集合，所以必须做多任务覆盖。  
但如果在企业合作里聚焦几个高价值场景，分化更有机会。  
现在他们更多在追“总量覆盖”。

**主持人**：  
是不是因为“操控这些智能脑”的科学还不成熟？

**Andre**：  
是的，这也是原因之一。  
我说“操控”包括：比如微调而不丢能力。  
我们现在缺少比 context window 更好的可编程原语。  
context window 好用、便宜、易控，所以很多定制靠它。  
但更深层调整还不成熟：  
怎么持续学习、怎么在某一域变强、怎么改 weights 而不仅仅是喂上下文。  
动 weights 比动 context 难得多，因为你在改整个模型，可能影响整体智能。  
所以 speciation 不只是意愿问题，也有技术成熟度问题。  
而且成本还得足够低，分化才划算。

**主持人**：  
再问你一个延展到 open ground 的问题：  
你说需要更大的协作面让更多人参与研究，具体怎么想？

**Andre**：  
单线程 auto research（一个循环）只是起点。  
并行化才是关键。  
我最近也在试不同方案，但还没有一个让我非常满意、足够简洁的版本。  
如果你有很多并行节点，很容易做“多个 auto researcher 通过共享系统协作”。  
我更感兴趣的是：如何让“互联网上不可信 worker 池”参与。  
比如在 auto research 里，我们想得到“能把模型训练到更低验证损失的代码”。  
如果网上有人给你一个候选 commit，你其实很容易验证它是否真的更好。  
他们可能撒谎，但你可以验证。  
所以结构有点像区块链：  
只不过不是 blocks，而是 commits；  
commits 彼此叠加，包含代码改动；  
“工作量证明”是大量实验搜索有效 commit，这一步很难；  
而“验证”很便宜：你只需验证候选是不是有效。  
当前奖励只是排行榜，还没有金钱激励。  
我不想过度类比区块链，但核心确实类似：  
生成难、验证易。  
你需要一个系统让不可信 worker 池与可信验证池协作，异步运行，并且安全。  
因为执行外部提交的任意代码本身很危险。  
但原则上完全可行。  
SETI@home、Folding@home 这类项目都符合相同结构。  
比如蛋白折叠：找到低能态构型很难，但别人给你一个候选后，验证它就容易。  
凡是“求解贵、验证便宜”的问题，都适合这种范式。  
所以无论是 Folding@home、SETI@home，还是 Auto Research @ home，都很适合。  
长话短说：互联网 agent swarm 可能协同改进 LLM，甚至在某些方向跑赢前沿实验室，也不是不可能。  
前沿实验室有大量可信算力；地球上有更大量不可信算力。  
如果机制设计到位，外部 swarm 也许真的能产出更优方案。  
人们可以把算力周期贡献给自己在乎的问题。  
公司也可以有自己的 track：比如你关心某个癌症方向，不只是捐钱，也可以直接买算力并加入该方向的 auto research。  
如果一切都被重构成 auto researchers，那么“算力”本身会成为主要贡献物。

**主持人**：  
这很鼓舞人。  
也很有意思的是，至少在一部分人群里，“个人算力”重新变得重要。  
也许先是为自己的 claws 配算力，然后就能把多余算力贡献给 auto research。  
会不会将来表面上人人谈钱，本质上人人谈 FLOPs？  
现在很多时候有钱也不一定马上拿到算力，似乎 FLOPs 更主导。

**Andre**：  
这是个有趣想法。  
“你控制多少 FLOPs”像是“你控制多少财富”的平行叙事。  
我不认为它会完全替代财富概念，但确实值得思考。

**主持人**：  
你最近还发过一个就业数据小分析，对吗？  
你只是可视化公开数据，但好像触发了不少讨论。你当时在好奇什么？

**Andre**：  
我只是想更具体地看 AI 对就业市场的影响。  
我先看结构：各职业分布、人数、走势。  
然后逐项去思考：随着 AI 演化，这些职业会更偏“工具增强”还是“替代压力”？会增长、调整，还是出现新职业？  
这主要是为了驱动我自己的思考链。  
数据来源是美国劳工统计局（Bureau of Labor Statistics），他们给了每个职业未来增长预估（大致是近十年尺度，基于 2024 年版本）。  
我也不 100% 清楚他们全部方法细节。  
我当时还做了一个颜色标注思路：  
如果当前主要发展的是“数字型 AI”——像幽灵/精神体，能在数字世界里处理信息——  
而不是有完整物理身体，那么“物理世界的变革”会慢一些，因为你在操控原子。  
比特复制和处理速度远快于物质世界。  
所以从系统动力学角度，我预期数字空间会有极高强度活动，像“沸腾”一样重写；相较之下物理空间会慢不少。  
这意味着数字信息处理职业会先经历剧烈重构。  
这不自动等于岗位数量必然增减（还涉及需求弹性等因素），但“工作内容重写”几乎确定会发生。  
我当时高亮的就是这类可在家完成、核心操控数字信息的职业。

**主持人**：  
你看完数据后，有没有对求职者或学习者的建议？  
比如现在该学什么、练什么能力？

**Andre**：  
这个问题很难统一回答，因为就业市场非常异质。  
但有一点很现实：工具很新、能力很强，先尽量跟上。  
很多人会排斥或害怕，这是可理解的。  
但就当前阶段，它首先是赋能工具。  
职位本质是任务捆绑，其中一部分任务会因 AI 加速。  
所以先把它作为工具来掌握。  
长期会怎样，确实很难预测。  
我不是职业经济学家，这类判断最好由经济学体系化研究。  

**主持人**：  
不过你毕竟是工程师。  
有个很有意思的现象：工程岗位需求还在增长。  
我不确定这是不是阶段性。你怎么看？

**Andre**：  
我知道你在说什么。  
有一种可能是：软件一直稀缺，过去需求上不去是因为成本太高。  
当门槛下降，会出现杰文斯悖论：更便宜反而让需求更大。  
经典例子是 ATM 与银行柜员。  
当年很多人担心 ATM 会替代柜员，结果 ATM 降低网点运营成本，网点数量增多，柜员总数反而上升。  
所以我对软件工程是谨慎乐观。  
软件需求看起来会很大，而供给成本正在下降。  
中短期我确实觉得软件活动会更多。  
因为软件本质是数字信息处理，而且现在代码变得可塑、可临时生成、可快速改写。  
你不再只能被迫使用既有工具。  
所以我觉得数字空间会出现非常大规模的“重连线”，这会带来需求。  
当然长期不确定性仍大。  
即使看 auto research：OpenAI、Anthropic 这些实验室也就雇一两千研究员。  
而这些研究员本质上正在积极自动化自己。  
我在一些团队里也感受到那种 psychosis——因为他们看见它真的在工作。  
大家会有一种“这也会轮到我”的不安。  
我甚至在 OpenAI 里跟人说过：  
“你们知道如果我们成功，我们都可能没工作了吧？我们像是在给 Sam 或董事会造自动化机器。”  
这确实会让人紧张。

**主持人**：  
我可以问你一个 Nome 常问的问题吗？  
你明明可以去前沿实验室，用更大算力和更多同事做 auto research，为什么不？

**Andre**：  
这个问题很复杂，也很“重”。  
我其实去过前沿实验室，也离开过。  
我认为外部生态位同样可以有很大影响力，不只是产业岗位，生态层面的工作也很关键。  
你现在的工作就是生态位角色；我现在某种程度也是。  
我对这类影响力是认可的。  
反过来，过度绑定前沿实验室也会带来问题：  
你会有很强的财务激励绑定；  
同时你又承认 AI 会深刻改变社会；  
这会形成“既推动技术又在经济上深度绑定”的张力。  
这其实就是 OpenAI 早期要解决的悖论之一，今天也没有完全解决。  
在前沿实验室里，你不是完全自由代理。  
有些话你不能说，有些话组织希望你说。  
没人会硬压你，但你会感受到叙事压力。  
如果偏离，会有很尴尬的组织氛围。  
所以从某种角度，我在外部会感觉更接近“对齐人类整体”，因为不被这些组织压力约束。  
当然，这不意味着内部没有影响力。  
在内部你也可能做出巨大贡献，也可能正好坐在关键决策室里。  
但当赌注真正升高时，作为员工你到底有多大实际主导权，我不确定。  
你在房间里可以提意见，但不代表你掌舵。  
这就是我看到的一些潜在失配。  
另一方面，我也同意一种担忧：  
前沿实验室在能力边缘，且比较不透明。  
如果你长期在外部，你对“即将发生什么”的判断会漂移。  
我也担心自己会漂。  
你会逐渐不理解系统内部真实状态与发展轨迹。  
所以我也认同“保持和前沿实际进展接触”很重要。  
某种理想形式可能是“往返”：去做一段高质量工作，再出来。  
这样既连接前沿，又保持一定独立性。  
所以我会说：Nome 在 OpenAI 里可以做很有价值的工作；但他最有影响力的工作，也可能在 OpenAI 之外。  
内外两边都能做出重要贡献。  
我自己的轨迹也是：进去过、出来了，未来也可能再进去。

**主持人**：  
一个相关问题是：世界（或 AI 生态）到底能看到多少前沿？  
开源离前沿有多近？这种关系可持续吗？

**Andre**：  
这段演化其实很令人惊讶。  
从“几乎没有可比开源”到“落后 18 个月”，再到现在大家讨论“落后 6-8 个月”，是明显收敛。  
我一直是开源支持者。  
类比操作系统：有闭源 Windows 和 macOS，也有 Linux。  
Linux 是极成功项目，覆盖了大量计算设备。  
因为行业长期需要一个共同开放底座，让大家有可依赖的公共平台。  
我觉得现在也一样，企业层面对“可用开源底座”有需求。  
最大的不同是：这次是重资本游戏（capex 很大），所以竞争更难。  
但即便如此，当前开源模型在很多消费场景已经很好用了。  
再往后几年，很多基础场景会被开源充分覆盖，甚至本地运行。  
当然，前沿智能需求会持续存在，而且可能是很大一块市场。  
那部分可能对应更重大的问题（比如诺奖级研究、超大工程迁移等）。  
闭源前沿系统可能更多做这类高难任务，而开源会覆盖大量基础用例。  
“今天的前沿”会逐步变成“明天的开源能力”。  
我预期这个动态会继续：  
前沿实验室维持闭源神谕层，开源以数月时差追赶。  
我觉得这是个相对健康的结构。  
我对“智能完全闭源且只有这一种选择”有系统性风险担忧。  
集中化在历史上通常表现不佳。

**主持人**：  
你是指政治系统或经济系统里的中心化风险？

**Andre**：  
对。  
历史里有不少不好的先例。  
所以我希望存在一个“虽然略落后，但全行业可访问”的公共智能工作层。  
这对权力平衡很重要。

**主持人**：  
我也觉得要继续推进前沿能力，因为有很多人类大问题不推进就解决不了，这依然是高成本且必要的路线。  
但与此同时，把当下前沿能力逐步民主化也很重要。

**Andre**：  
我同意。某种程度上我们“误打误撞”处在一个还不错的位置。  
而且这个动态持续越久，生态可能越健康。  
即便在闭源侧，我最近也感到有进一步中心化趋势。  
这并不理想。  
我希望有更多前沿实验室。  
我天然倾向“房间里有更多人”。  
就像模型集成通常优于单模型，我希望最难问题也由“多人集成认知”来决策，而不是两三个人关门决定。  
长话短说：我希望有更多实验室，也希望开源能持续扮演它的角色。  
现在开源略滞后，反而在某种意义上是好事。

**主持人**：  
你做过自动驾驶，这某种意义上是通用机器人自主性的前身。  
最近机器人公司也有很快进展：任务泛化变强、长程任务变多、资金涌入。  
你觉得“机器人时刻”会发生吗？最近你的判断有变化吗？

**Andre**：  
我对机器人的看法受自动驾驶经历影响很深。  
自动驾驶是我眼中的第一波机器人应用。  
十年前有很多创业公司，但长期存活并成功的不多。  
这类事需要很大资本投入和很长时间。  
机器人之所以难，是因为现实世界太乱、太难、太重资本，而且需要长期信念。  
原子世界很难。  
所以我判断：它会滞后于数字空间。  
数字空间会先出现巨量去瓶颈化（以前低效流程可能被提升一个数量级以上），因为 bit 更容易处理。  
所以短期最剧烈变化在数字空间，物理空间会慢一步。  
我同时很关注数字与物理的接口层。  
如果未来更多 agents 代表人类行动、彼此协作并参与“agent economy”，你迟早会碰到纯数字任务的边界。  
最后必须回到宇宙本身做实验，向现实提问，拿反馈学习。  
我们现在有大量数字任务，是因为人类还没消化完已经数字化的信息库存。  
随着系统能力提升，这个库存会被更快处理。  
到一定阶段，你必须去采现实新数据。  
所以我认为路径大致是：  
先数字空间大规模重构；  
再走向接口层（传感器读世界、执行器改世界）；  
再更深进入物理世界。  
我甚至觉得物理世界总市场更大，但门槛也高很多。  
所以节奏上它会滞后，但一旦到来，规模会很大。  
我现在主要兴趣在数字空间，其后是接口层，再后是更完整的物理层。

**主持人**：  
这个框架很有启发。  
在原子世界里，也有些事相对容易，比如“读物理世界”（传感器、摄像头）就已经有大量现成硬件。  
如果方法聪明，不一定要巨额投入，也能拿到价值。

**Andre**：  
是的。  
比如我朋友 Liam（Periodic 的 CEO），我上周刚去看他们。他们在做材料科学方向的 auto research。  
这时“传感器”不只是摄像头，可能是昂贵实验室仪器。  
生物工程也是类似，输入层会更复杂。  
我还看到有公司在做“按任务付费采训练数据”，程序化喂给系统。  
这些都属于接口层能力。  
我很期待一个时刻：  
我给出一个现实任务、标一个价格，让 agent 自己想办法完成并带回数据。  
我也挺惊讶我们还没有足够成熟的信息市场。  
比如在重大地缘事件期间，为什么不能出现“某地一张照片/视频值多少钱”的即时市场？  
这也是喂给智能系统的数据机制。  
最终可能不是人看这些信息，而是 agents 在市场中使用这些信息进行预测和行动。  
现在 agentic web 还很早，机制未完善。  
有本书《Demon》里有类似想象：智能系统有点像通过人类感知和执行来“操纵”现实流程。  
某种意义上，人既是传感器也是执行器。  
我认为行业层面可能会朝这个方向重排：自动化系统有需求，人类会在很多环节去满足这些需求。

**主持人**：  
在你这个点上，我们之前确实碰到过一个具体问题：  
我们缺一些训练数据，而要补这个缺口，需要像 auto research 那样更机械化的训练闭环。  
如果模型不能自己跑训练回路，那“按价格采数据并自动提升质量”的闭环就很难成立。  
我这样理解对吗？

**Andre**：  
完全对。  
不过就 LLM 训练来说，它其实很适配这个闭环。  
它的指标比较干净，代码效率优化也容易围绕指标推进。  
我确实担心完全围绕少数指标会出现 Goodhart 效应（为了指标而过拟合）。  
但你也可以让系统生成更多指标，提高覆盖。  
总体上这仍是很适配的任务类型。

**主持人**：  
结束前聊一个你的小项目：microGPT。

**Andre**：  
好。  
我有一个持续很久的执念：把 LLM 训练流程尽可能“熬干”，只剩算法本质。  
所以我做过 nanoGPT、makemore、micrograd 等。  
microGPT 是这个方向的最新形态。  
神经网络训练代码通常很大，但大量复杂度来自性能优化。  
如果不追求最快，只看算法骨架，其实可以非常短。  
你需要：文本数据集、模型结构、前向、反向、自动求导、优化器（如 Adam）和训练循环。  
核心可以压到几百行级别。  
以前如果我做出这种项目，我会想做长视频逐行讲解。  
我也尝试过。  
但后来我意识到，附加价值在变化：  
因为它已经足够简单，用户可以直接让 agent 以不同方式解释。  
我现在有点像不是在“给人解释”，而是在“给 agents 解释”。  
只要 agent 理解了，它就能作为路由层，以不同表达、无限耐心，把内容讲给不同背景的人。

**主持人**：  
对，比如我不懂某个函数时，可以让 agent 换三种方式解释；我不太可能从你这儿实时得到这种个性化。

**Andre**：  
正是。  
所以我在想教育形态会变。  
过去是讲义、课程、演讲；现在更像先让 agent 完整理解。  
甚至我会写“skill”来指导 agent 怎么教这个项目。  
比如 microGPT 的 skill：先带用户看哪里，再看哪里，什么顺序更好。  
这像是给模型的课程脚本提示。  
所以未来可能不是“我直接对人解释一切”，而是“我让 agent 完整吃透”。  
当然现在还没到完全替代，我有时仍能讲得比 agent 更好。  
但模型进步太快，我感觉这是一场会输的赛跑。  
教育会被明显重排。  
比如做代码库时，过去你写给人的 HTML 文档；未来更该写给 agent 的结构化 markdown。  
agent 理解后就能把内容以各种形式转译给人。  
这是一种“通过 agent 的重定向”。

**主持人**：  
那也意味着“优秀教师”将来要发展新能力：  
如何更好地向 agent 传达结构与直觉。

**Andre**：  
是。  
举个例子，我让 agent 去写“最简神经网络训练实现”，它理解方向，但写不出我定义的 microGPT 这种极简终局。  
microGPT 是我长期执念的结果：200 行左右、我认为再简就不成立。  
这里是我的价值所在：我给出“关键的少量比特”。  
其余传播和讲解，agent 会越来越擅长。  
它也许暂时不能发明这个终局，但它能理解为什么这么设计。  
所以人的工作会更集中在：注入 agent 还做不到的关键比特——  
核心抽象、课程结构、最有效解释路径。  
agent 已经能做好的部分，未来会更好。  
人要更战略地分配自己的时间。

**主持人**：  
我们非常感谢你贡献这些“关键比特”。谢谢你，Andre。

**Andre**：  
谢谢。
