## 3) “每天 16 小时表达意志给 agents”

**主持人**：  
我记得有次走进办公室，你整个人都“锁死”在工作里。我问你在干嘛，你说：  
“我每天要写 16 小时代码——不，‘写代码’都不是准确动词了。我得把我的意志表达给我的 agents，让它显化。”  
因为能力出现了跳变。到底发生了什么？说说你的体验。

**Andre**：  
我感觉自己一直处在一种“AI 精神错乱感”里，到现在也经常这样。  
因为作为一个个体，你能做到的事情出现了巨大解锁。以前你被打字速度这些因素卡住；但有了 agents 之后，我觉得大概在 12 月的时候，某个开关彻底被拨动了：  
我从“80% 自己写代码、20% 委托 agent”，变成了“20% 自己写、80% 委托”。现在甚至可能连 20/80 都不是了，比例更极端。  
我从 12 月起基本上就没怎么亲手敲过代码。这个变化非常巨大。  
我甚至跟我爸妈聊过这个，我觉得“普通人”还没意识到这件事已经发生，更没意识到它有多剧烈。  
你随便找一个软件工程师，看他现在在工位上的默认工作流——从去年 12 月起几乎已经完全不同了。  
所以我一直处在这种“精神错乱感”中：到底还有什么可能？如何把能力推到极限？如何不只用一个 Claude Code 或 Codex 会话，而是同时拥有更多？怎么正确使用这些“claw”？这些 claw 到底是什么？  
新东西太多了。我想站在最前沿；但我又焦虑自己不够前沿。我看到 Twitter 上很多人都在做各种事，听起来全是好点子，我必须在最前面，不然我会非常紧张。  
所以本质上就是：可能性空间尚未被探索，这种状态让我一直处在“精神错乱感”里。

## 4) 并行多 agent 的新工作方式

**主持人**：  
如果你都紧张，那我们其他人更紧张。  
我们在 Conviction 合作的一支团队，他们的工作方式是：工程师几乎不再手写代码，大家都戴着麦克风，一直对着 agent 低声下指令。是我见过最奇怪的办公环境。  
我一开始以为他们疯了，现在我完全接受了——“哦，原来这才是正确路径”，只是他们更早而已。  
你现在怎么看自己的“探索能力/做项目能力”？它被什么限制？

**Andre**：  
被什么限制？我觉得几乎被一切限制。  
很多东西即使没成功，你也会觉得是“技能问题”，不是能力不存在，而是你还没找到把现有能力串起来的方法。  
比如：是不是我在 `agents.md` 里指令写得不够好？是不是我没放一个足够好的记忆工具？  
所以很多“跑不通”的情况，都会让你感觉：这是我不会用，而不是模型不行。  
你会想办法并行化它们，等等。你会想成为 Peter（他很擅长多窗口并行分配 agent 工作）。  
他有张很有名的图：一个大屏上开了很多个 Codex agent。每个 agent 如果 prompt 对了、开高 effort 模式，大概都要跑 20 分钟。  
他会同时 checkout 多个仓库，在它们之间来回分配工作。  
这时你在软件仓库上的操作粒度变成“宏动作”：不再是“写一行代码、加一个函数”，而是“给 agent1 一个完整新功能”“给 agent2 一个不冲突的另一功能”，然后你再尽量审查它们的产出——审查到什么程度取决于你对那段代码的在乎程度。  
同时还有 agent 在做研究、agent 在写代码、agent 在做新实现计划。  
整个仓库的演化都在“宏动作”层面发生，而你在训练自己对此形成肌肉记忆。  
这非常有回报——第一，它真的能工作；第二，这是全新的技能学习对象。  
所以，精神错乱感由此而来。

## 5) Token 吞吐焦虑：新瓶颈是“你自己”

**主持人**：  
我也有这种直觉：每次在等一个 agent 跑完时，最显然的事就是“我还能继续派活”。  
如果我还有 token 额度，那就该继续并行化、多开任务。  
这很有压力，因为如果你不太受 token 开销约束，那系统里最大的瓶颈就是你自己——你没把能力拉满。

**Andre**：  
对。至少你没把订阅价值榨满。  
理想状态甚至是多 agent、多平台：Codex 用完就切 Claude 之类。我最近就在这么试。  
只要我还有订阅余额，我就会紧张：这说明我的 token 吞吐还没最大化。  
这其实让我想起读博时：你会因为 GPU 没跑满而焦虑。你有算力却没把可用 FLOPs 吃满。  
现在不是 FLOPs 了，是 tokens。  
你的 token 吞吐是多少？你能调度多大的 token 吞吐？  
我反而觉得很有意思：过去很多工程任务里，人们并不觉得自己“受算力限制”。  
现在整个行业都在感受到资源约束。  
当能力跳变之后你会意识到：哦，限制不再是“拿不到算力”，而是“我自己是瓶颈”。

## 6) 从单会话到“claw + 多体协作”

**主持人**：  
技能问题虽然有压力，但也很有掌控感。  
如果大家都每天这样训练，一年后“精通”会是什么样？

**Andre**：  
大家都在往“更高抽象层”走。  
首先不再是单会话，而是多 agent 如何协作、如何组队。  
其次，claw 也是一个很有趣方向。  
我说的 claw，是一种把“持久性”提升到新层级的东西：它会持续 loop，不需要你一直交互盯着；它有自己的 sandbox，会在你不看时也代你执行；可能还有更高级的记忆系统（目前很多 agent 还没实现）。  
OpenClaw 的记忆系统就比默认 agent 强很多。默认通常只是 context 用尽后的“压缩记忆”。

## 7) 工具人格：为什么“像队友”很关键

**主持人**：  
你觉得 OpenClaw 最打动用户的是记忆/持久性，还是更广泛的工具接入？

**Andre**：  
我觉得这里面有很多好想法。开发者做得很出色，而且是多个方向并行创新后整合。  
比如人格设计：如果 agent 像队友，用户体验会明显更好。  
我觉得 Claude 的人格感比较好；Codex 在 coding agent 形态下会更“干”。  
它会说“我实现好了”，但不一定显得真的在乎你在构建什么。  
另外，Claude 的“正反馈阈值”我觉得调得不错：不是每个想法都夸，但真有好点子时会更明确鼓励。  
这会让你产生一种“想赢得它认可”的心理，虽然有点奇怪。  
所以人格确实重要。再加上记忆系统、以及统一入口（比如通过 WhatsApp 操作所有自动化），体验会变得很有吸引力。

## 8) Dobby：把家里 6 个 App 合并成自然语言控制

**主持人**：  
你个人在软件工程之外，用 claw 做过哪些好玩或有趣的事？

**Andre**：  
今年 1 月我有一段“claw 精神错乱期”。我做了一个管我家的 claw，我叫它 Dobby（家养小精灵）。  
我让 agents 去发现我家局域网里的所有智能家居子系统。  
我本来很惊讶它能开箱即用：我只说“我家应该有 Sonos，你去找找”。  
它就扫描局域网设备，找到了 Sonos。  
结果 Sonos 那边几乎没什么额外防护，它直接连上后去查 API endpoint，再回来问我要不要试。  
我说“在书房放点音乐”，它就真的放了。几条 prompt 就完成。  
灯光系统也是同样流程。  
后来它基本把整套系统打通：创建 API、做 dashboard，让我能看到全屋控制中心。  
现在我可以说“Dobby，睡觉时间”，它就把灯光等状态切过去。  
它还管 HVAC、窗帘、泳池和 spa，以及安防。  
我有个朝屋外的摄像头，出现变化时会触发检测，再把画面交给视觉模型分析，最后通过 WhatsApp 给我发消息，例如“FedEx 车到了，可能有包裹”。  
这真的很震撼。  
对我来说，哪怕只是把原来 6 个分散 App 合并成自然语言入口，都已经非常有价值。  
而且这个范式我还远没推到极限。

## 9) “App 可能不该存在”：API-first + Agent-first

**主持人**：  
这是不是说明用户真正想要的，不是更多 UI，而是更低学习成本和更统一的交互？

**Andre**：  
某种程度上是。  
很多智能硬件 App 在未来可能会被弱化：设备更应该暴露 API，让 agents 直接编排。  
因为 LLM 可以跨系统调用工具，做单个 App 做不到的自动化。  
这会推动行业重构：  
客户不再只是“人”，还包括“代表人行动的 agents”。  
今天这些操作还需要一点 vibe coding 和人工盯着。  
但我认为门槛会持续下降，最后变成“基础能力”：  
你只表达意图，系统自动把非技术用户意图翻译成可执行流程。  
软件会越来越“临时可塑（ephemeral）”，大量细节由底层自动化实体处理。

## 10) 为什么要做 Auto Research：把人从环路里拿掉

**主持人**：  
你提到很久了：希望 agents 能直接承担模型训练/优化任务。Auto Research 的动机是什么？

**Andre**：  
核心是“去瓶颈化”：人不该一直在环里点下一步。  
你要把系统组织成尽可能自治，最大化 token 吞吐和杠杆率。  
我投入很少 token，系统替我完成大量研究动作。  
Auto Research 的目标就是：给定目标、指标和边界，让系统自己跑。  
问题不是“我该怎么手动做研究”，而是“我怎么改抽象，让我不在环里仍能持续推进”。

## 11) 结果超预期：在已调优仓库里继续找出增益

**主持人**：  
而且它效果比你预期更好，对吗？

**Andre**：  
是的。  
我本来不觉得会有太大收获，因为我已经用传统方式做了大量调参。  
我做这类研究很多年了，常规方法都很熟。  
但让 Auto Research 跑过夜后，它还是找到了我遗漏的组合项：  
比如某些 weight decay 配置、Adam betas 的联动。  
这说明哪怕仓库已经比较优化，自治搜索仍能继续挖到东西。  
而这还只是单 loop。  
前沿实验室有大规模 GPU 集群，在小模型上做自治探索再外推到大模型，完全可以想象。

## 12) Program.md 与“研究组织即代码”

**主持人**：  
再递归一步：模型什么时候会写出比你更好的 `program.md`？

**Andre**：  
`program.md` 本质就是“研究流程说明书”。  
我现在写的是一个比较粗糙的版本：先做什么、再做什么、优先看架构还是优化器，等等。  
你完全可以把“研究组织”看成一组 markdown 文件：角色定义、流程定义、协作关系定义。  
不同 `program.md` 就会导向不同产出。  
所以“优化 program.md”本身就是元优化问题。

## 13) Contest 想法：同硬件下比谁进步更多

**主持人**：  
我给你发过一个比赛想法：让大家写不同 `program.md`，在同样硬件下看谁改进最多；  
再把这些数据喂给模型，让它写更好的 `program.md`。

**Andre**：  
这个思路很对。  
你可以分析“改进从哪里来”，再反推如何调整流程让这类改进更高频出现。  
也能分析哪些策略无效。  
元优化是可以层层递归的。  
这也解释了为什么会有“无限感”：  
LLM、agent、claw、多体协作、指令工程、指令优化、再到优化优化器……层层往上。

## 14) Caveat：可验证任务最适配，软目标仍有“锯齿感”

**主持人**：  
这是不是意味着很多领域都该做成“指标闭环 + 自动迭代”？  
那性能工程这类工作还在吗？

**Andre**：  
有两个重要前提：  
第一，最适配的是“可验证指标”任务。  
比如 CUDA kernel 优化：行为一致但速度更快，这是完美范式。  
如果无法评估，就很难做 Auto Research。  
第二，系统现在仍有明显“锯齿感（jaggedness）”：  
有时像顶级工程博士，有时像 10 岁小孩。  
你能感到它的强大，但也会遇到离谱错误和错误循环。  
所以方向清楚，不代表今天就能无脑全放手。

## 15) “笑话问题”：能力增长并非全域同步

**主持人**：  
我也会因为 agent 在明显错误上浪费大量算力而烦。  
你觉得这背后是什么？

**Andre**：  
部分原因是当前优化机制偏向“可验证奖励”。  
程序对不对、单测过不过，这些都好训练。  
但对“意图细节”“何时澄清提问”这类软目标，优化就弱。  
所以会出现一种现象：  
你让它做可验证复杂任务，它能跑很久很强；  
你让它讲笑话，却还是很多年前的老梗。  
这说明能力提升并不自动等于“所有维度同步提升”。

## 16) 是否要“拆模型”：从单体到分化（speciation）

**主持人**：  
如果这种锯齿持续，单一大模型形态是否合理？  
要不要拆成更明确的专家系统？

**Andre**：  
现在前沿实验室普遍在追“单一模型覆盖所有域”。  
但我认为长期会出现更多“分化”：  
像生物神经系统一样，不同任务会对应更专门、可能更小但仍有认知核心的模型。  
这样可在特定任务上得到更好的延迟/吞吐效率。  
只是目前这门“可控改造智能体”的工程学还不成熟：  
改 context 比较容易，改 weights 难很多，也更有系统性风险。

## 17) 开放协作研究：不可信算力池 + 可信验证池

**主持人**：  
你提到想把 Auto Research 扩到更大协作面。怎么做？

**Andre**：  
关键在并行化，以及“开放协作但可验证”。  
如果互联网上有大量不可信 worker 提交候选 commit，  
你需要一套可信验证流程去快速验真。  
这在结构上有点像“生成难、验证易”的系统。  
很多问题都符合这个模式：  
Folding@home、SETI@home 都是典型。  
所以理论上，互联网上的 agent swarm 也可能协作改进模型。  
前沿实验室有大量可信算力，但全世界有更大规模的不可信算力。  
如果机制设计好，后者也可能产出非常强的结果。  
那时“贡献”不仅是钱，也可以是算力周期。

## 20) 数字世界会先爆发，物理世界会滞后

**Andre**：  
如果把 AI 看作第三类“数字信息操控者”，  
那么数字空间会先发生大规模重构。  
因为处理 bit 比操控 atom 快太多。  
这并不直接等于“岗位一定增或减”，但意味着工作内容会重写。  
物理世界也会变，但节奏通常更慢。

## 22) 软件工程需求会怎样：杰文斯悖论视角

**主持人**：  
工程岗位需求还在涨，你怎么看？

**Andre**：  
我偏谨慎乐观。  
如果软件供给成本下降，可能触发杰文斯悖论：更便宜反而带来更大需求。  
软件可塑性太强，且数字空间重构潜力很大。  
所以在一段时期内，软件相关需求可能继续上行。  
但远期仍有高不确定性。

---

## 23) “为什么不回前沿实验室？”：影响力与独立性的权衡

**主持人**：  
你也可以去前沿实验室，用更大算力和团队做这些，为什么不？

**Andre**：  
这个问题很复杂。  
在前沿实验室内，资源和前沿信息密度都很高，影响力很直接；  
但也会面临组织激励、叙事压力和独立表达边界。  
在外部生态位，也能做高影响力工作，而且更独立。  
我认为两边都重要，理想状态可能是“在内外之间往返”，既接触前沿又保持独立判断。

---

## 24) 开源 vs 前沿：差距在收敛

**主持人**：  
开源离前沿有多近？这种格局可持续吗？

**Andre**：  
总体上闭源前沿仍领先，但开源追赶速度比很多人预期快。  
很多场景里，开源模型已经“够用甚至很好用”。  
长期可能形成稳定结构：  
前沿闭源探索最前线，开源以时间差扩散能力。  
这对生态韧性是有价值的。

---

## 25) 为什么需要开源：系统性风险与权力平衡

**Andre**：  
如果智能体系只由少数闭源实体掌握，存在系统性风险。  
我更希望有一个全行业可访问的共同能力底座，即便它略落后于最前沿。  
这有助于形成更健康的权力平衡。

---

## 26) “房间里要有更多人”：反对过度中心化

**Andre**：  
我天然希望关键决策“房间里有更多知情者”。  
像模型集成通常优于单模型一样，  
重大问题的认知也应当是“多人集成”，而不是极少数人的封闭决策。

---

## 27) 对当前格局的判断：略滞后但可用，反而是好事

**Andre**：  
某种意义上，我们“偶然”处在一个还不错的位置：  
开源略滞后但持续跟进，前沿仍在推进昂贵突破。  
这两者并存，可能是当前阶段较健康的结构。

---

## 28) 机器人：会发生，但节奏与数字空间不同

**主持人**：  
最近机器人很热，你怎么看？

**Andre**：  
我受自动驾驶经验影响很深。  
机器人是大问题、重资本、长周期、落地极难。  
所以我判断：数字空间先剧变，物理空间滞后。  
但后者市场可能更大，只是门槛更高。

---

## 29) 关键机会在“数字-物理接口”

**Andre**：  
接下来很关键的是接口层：  
传感器如何持续把现实喂给智能系统，执行器如何把智能输出变成对现实的操作。  
这会催生很多公司与新基础设施。  
纯数字任务做深之后，系统最终还是要回到现实世界做实验、拿反馈。

---

## 30) 信息市场与 agentic web 的缺口

**Andre**：  
我惊讶我们还没有足够成熟的信息市场机制。  
比如某地一张实时照片/视频应当能被即时定价和采集。  
这会成为“喂智能”的基础能力之一。  
当前 agentic web 还在早期，很多机制还没建起来。

---

## 31) “人类既是传感器也是执行器”

**Andre**：  
从更宏观角度看，社会可能被重新编排：  
自动化系统提出需求，人类在其中一部分场景里既充当传感器，也充当执行器。  
这是一种系统层面的角色重构。

---

## 32) microGPT：把 LLM 训练“熬到只剩本质”

**主持人**：  
结束前聊聊你的小项目 microGPT。

**Andre**：  
这是我长期执念的延续：把复杂系统还原为最小本质。  
现实里 LLM 训练代码很庞大，但很多复杂度来自“效率优化”。  
如果只看算法骨架，它可以非常短：  
数据、模型、前向、反向、优化器、训练循环，核心可以压到几百行级别。

---

## 33) 教育范式变化：从“给人写文档”到“给 agent 写文档”

**Andre**：  
以前你会写教程给人看；  
现在越来越像是“先让 agent 彻底理解”。  
如果 agent 理解了，它能按不同人的背景无限耐心地解释。  
所以文档形态也会变化：  
不只是面向人的页面，也要有面向 agent 的结构化说明（如 markdown、技能脚本等）。

---

## 34) “人的价值位”会转向：提供 agent 还做不到的关键比特

**Andre**：  
我试过让 agent 自己产出“最简实现”，它常常能理解但不一定能发明。  
人的价值会更集中在“注入关键比特”：  
真正的抽象、关键路径、课程结构、独到洞见。  
agent 能稳定做好的部分，未来会越做越好；  
人要更战略地把时间放在 agent 还做不到的地方。
