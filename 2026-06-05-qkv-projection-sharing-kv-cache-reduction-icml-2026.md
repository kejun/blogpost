# Transformer 需要三个投影吗？ICML 2026 论文深度解析：QKV 共享如何让 KV Cache 瘦身 97%

> **摘要：** 2026 年 6 月 1 日，一篇题为 *"Do Transformers Need Three Projections? Systematic Study of QKV Variants"* 的论文被 ICML 2026 接收。作者系统性地评估了 QKV 注意力中三个线性投影的共享约束，发现 **Q=K 共享变体在语言建模中仅带来 3.1% 的困惑度退化，却实现了 50% 的 KV Cache 削减**。更令人震惊的是，当 Q=K 共享与 GQA-4 结合时，KV Cache 总削减达到 **87.5%**；与 MQA 结合则高达 **96.9%**——这意味着在边缘设备上运行大模型的内存门槛被大幅降低。本文将从 Transformer 注意力机制的第一性原理出发，深度拆解这篇论文的方法、实验和隐含意义，并论证：**2026 年的 KV Cache 优化正在从"外围工程技巧"走向"模型架构原生设计"。**

## 一、背景：KV Cache 为什么成了 2026 年最痛的瓶颈

### 1.1 长上下文时代的"内存税"

2026 年，大模型的上下文窗口已经进入了十万到百万 tokens 的竞赛阶段。但很少有人注意到一个反直觉的事实：**在长上下文推理中，决定内存消耗的不再是模型权重，而是 KV Cache。**

让我们用一个具体数字来说明：

```
┌─────────────────────────────────────────────────────────────────┐
│  Llama-3-70B 在不同上下文长度下的内存分解（FP16 推理）            │
├─────────────────────────────────────────────────────────────────┤
│  模型权重 (FP16):                              140 GB           │
│                                                                   │
│  KV Cache (per token, per layer):                                │
│    = 2 × num_layers × num_heads × head_dim × hidden_size         │
│    = 2 × 80 × 64 × 64 × 8192 / 2^30                             │
│    ≈ 0.5 MB/token                                               │
│                                                                   │
│  上下文长度 = 128K tokens → KV Cache ≈ 64 GB                     │
│  上下文长度 = 1M tokens   → KV Cache ≈ 500 GB                    │
├─────────────────────────────────────────────────────────────────┤
│  结论：当上下文超过 ~256K tokens 时，KV Cache 超过模型权重         │
│  当上下文达到 1M tokens 时，KV Cache 是模型权重的 3.6 倍          │
└─────────────────────────────────────────────────────────────────┘
```

这意味着：**在长上下文场景下，KV Cache 才是真正的"内存杀手"。** 而当前主流的 KV Cache 优化技术——GQA（Grouped Query Attention）、MQA（Multi-Query Attention）、PagedAttention、KV Cache 量化——都是在**推理引擎层面**做的工程优化。

**但论文问了一个更根本的问题：我们能不能从注意力机制的设计本身，减少 KV Cache 的需求？**

### 1.2 已有 KV Cache 优化技术的局限性

在深入论文之前，先回顾一下当前主流的 KV Cache 优化路径及其局限：

| 技术 | 原理 | KV Cache 削减 | 质量影响 | 局限 |
|------|------|---------------|----------|------|
| **MQA** | 所有 head 共享一组 KV | ~8×（64→1 head） | 中到高 | 质量损失明显，尤其在长上下文 |
| **GQA** | 每组 head 共享一组 KV（如 8 组） | ~4-8× | 低到中 | 组数越少质量越好，但削减也越少 |
| **KV Cache 量化** | INT8/INT4 存储 KV | 2-4× | 低 | 需要反量化开销，精度损失累积 |
| **PagedAttention** | 分页管理 KV，减少碎片 | 0×（只优化利用率） | 无 | 不减少总量，只减少浪费 |
| **StreamingLLM** | 只保留最近 tokens 的 KV | 可变 | 中到高 | 丢失全局上下文 |

所有这些技术都有一个共同前提：**Q、K、V 三个投影是独立的、必须的。** 这篇论文的核心贡献就是挑战了这个前提。

## 二、核心问题：Transformer 真的需要三个独立的投影吗？

### 2.1 标准注意力机制的回顾

在标准 Transformer 的自注意力中，每个 token 的输入向量 $x$ 通过三个独立的线性投影得到 Query、Key 和 Value：

$$Q = xW_Q, \quad K = xW_K, \quad V = xW_V$$

然后计算注意力权重并加权求和：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d}}\right)V$$

**三个投影矩阵** $W_Q$、$W_K$、$W_V$ 各为 $d \times d$ 维度（其中 $d$ 是隐藏层维度）。在推理时，$K$ 和 $V$ 需要缓存到 KV Cache 中，而 $Q$ 只需要为当前 token 计算。

**论文问了一个简单但深刻的问题：这三个投影中，有哪些是可以共享的？**

### 2.2 三种投影共享变体

论文系统性地评估了三种共享约束：

```
┌──────────────────────────────────────────────────────────────────────┐
│  QKV 投影共享变体一览                                                 │
├──────────────────────┬──────────────────────┬────────────────────────┤
│  变体                │ 投影关系              │ 注意力图对称性         │
├──────────────────────┼──────────────────────┼────────────────────────┤
│  标准 (Baseline)     │ Q≠K≠V（三个独立）     │ 不对称                 │
│  Q=K-V（共享KV）      │ K=V，Q 独立           │ 不对称                 │
│  Q=K-V（共享QK）      │ Q=K，V 独立           │ 对称 ★                 │
│  Q=K=V（单一投影）    │ Q=K=V（同一个）        │ 对称 ★                 │
└──────────────────────┴──────────────────────┴────────────────────────┘
```

**为什么 Q=K 和 Q=K=V 会产生对称注意力图？**

当 $Q = K$ 时，注意力分数矩阵为：

$$QQ^T = (xW_Q)(xW_Q)^T$$

这是一个**对称矩阵**（因为 $(AB^T)^T = BA^T$，当 $A=B$ 时对称）。这意味着 token $i$ 对 token $j$ 的注意力分数等于 token $j$ 对 token $i$ 的注意力分数——这在自注意力中是不自然的，因为**注意力应该是有方向的**。

**对称性问题是这篇论文需要解决的核心技术挑战之一。**

## 三、论文方法论：如何评估共享约束

### 3.1 实验设计全景

论文的实验覆盖了从合成任务到真实应用的多个维度：

```
┌─────────────────────────────────────────────────────────────────────┐
│  实验覆盖矩阵                                                        │
├──────────────────┬──────────────────────────────────────────────────┤
│  任务类型         │ 具体数据集/模型                                   │
├──────────────────┼──────────────────────────────────────────────────┤
│  合成任务         │ Copy task、Sequence manipulation                │
│  视觉分类         │ MNIST、CIFAR-10、TinyImageNet、异常检测         │
│  语言建模（小）    │ 300M 参数模型，10B tokens 预训练                 │
│  语言建模（大）    │ 1.2B 参数模型，10B tokens 预训练                 │
├──────────────────┼──────────────────────────────────────────────────┤
│  评估指标         │ Perplexity、Accuracy、Attention map 分析         │
│  对比基线         │ 标准 QKV Transformer（同参数量）                  │
│  共享约束         │ Q=K-V、Q=K-V、Q=K=V                              │
│  位置编码变体     │ 标准 1D PE、2D 位置编码（用于打破对称性）          │
└──────────────────┴──────────────────────────────────────────────────┘
```

### 3.2 关键设计决策

论文在方法论上有几个值得注意的设计：

**1. 参数量控制**

所有变体都保持**相同的总参数量**。当共享投影时，省下的参数被重新分配到其他组件（如 FFN 层）中，确保公平比较。这避免了"参数更少所以性能差"的混淆因素。

**2. 2D 位置编码破解对称性**

对于产生对称注意力图的变体（Q=K-V 和 Q=K=V），论文引入了 2D 位置编码：

$$\text{Attention}(i, j) = \text{softmax}\left(\frac{Q_i K_j^T + P_{i,j}}{\sqrt{d}}\right)$$

其中 $P_{i,j}$ 是依赖于位置对 $(i, j)$ 的偏置项，**打破了 $i \leftrightarrow j$ 的对称性**。这是一个简洁而有效的技术——不需要修改注意力机制本身，只需在注意力分数上加一个位置相关的偏置。

**3. 渐进式复杂度**

从合成任务开始（模型需要学习的规律最明确），逐步过渡到真实数据。这种设计使得如果某个变体在简单任务上就失败了，可以尽早淘汰，而不是浪费算力在大规模实验上。

## 四、核心发现：令人惊讶的结果

### 4.1 语言建模：Q=K 共享是最优选择

论文在 300M 和 1.2B 参数模型上的预训练实验（10B tokens）给出了最关键的发现：

```
┌──────────────────────────────────────────────────────────────────────┐
│  语言建模结果（1.2B 模型，10B tokens 预训练）                          │
├──────────────────────┬────────────┬────────────┬──────────────────────┤
│  变体                │ Perplexity │ KV Cache   │ 相对于基线的退化     │
├──────────────────────┼────────────┼────────────┼──────────────────────┤
│  标准 QKV            │ 15.42      │ 100%       │ —（基线）            │
│  Q=K-V（共享KV）      │ 15.90      │ 50%        │ +3.1%               │
│  Q=K-V（共享QK）      │ 17.85      │ 50%        │ +15.8%              │
│  Q=K=V（单一投影）    │ 19.23      │ 50%        │ +24.7%              │
└──────────────────────┴────────────┴────────────┴──────────────────────┘
```

**核心发现 #1：Q=K 共享（即 Key=Value 共享）在仅 3.1% 困惑度退化的代价下，实现了 50% 的 KV Cache 削减。**

这意味着：**K 和 V 在表示空间中可以占据相似的位置。** 这个发现挑战了长期以来"Q、K、V 必须有不同角色"的直觉。

**为什么 Q=K 共享效果好？** 论文给出了两个解释：

1. **注意力 operates in a low-rank regime**：在实际的 Transformer 中，注意力矩阵的有效秩远低于理论秩。这意味着 K 和 V 的表示空间存在大量冗余，共享投影不会丢失关键信息。

2. **K 和 V 的功能相似性**：Key 用于"寻址"（决定哪些 token 被关注），Value 用于"取值"（提供被关注后的信息）。在训练过程中，模型学会了让 K 和 V 承载互补但可共享的信息。

**核心发现 #2：Q=K 共享（Query=Key 共享）效果差的原因是打破了注意力的方向性。**

当 $Q = K$ 时，注意力分数矩阵变为 $QQ^T$，这是一个对称矩阵。但自然语言中的注意力本质上是**有方向的**——修饰语关注被修饰词的方向与被修饰词"回看"修饰语的方向是不同的。即使引入 2D 位置编码，对称性约束仍然是一个强限制。

### 4.2 视觉任务：所有变体都表现良好

在视觉分类任务上，结果更为乐观：

```
┌─────────────────────────────────────────────────────────────┐
│  视觉分类结果（CIFAR-10，准确率 %）                           │
├──────────────────────┬────────┬────────┬────────┬────────────┤
│  变体                │ 标准   │ Q=K-V  │ Q=K-V  │ Q=K=V      │
├──────────────────────┼────────┼────────┼────────┼────────────┤
│  CIFAR-10 准确率     │ 91.2   │ 91.0   │ 90.5   │ 89.8       │
│  TinyImageNet Top-1  │ 56.1   │ 55.8   │ 54.9   │ 53.2       │
│  异常检测 AUROC      │ 94.5   │ 94.3   │ 94.0   │ 93.5       │
└──────────────────────┴────────┴────────┴────────┴────────────┘
```

**在视觉任务上，所有变体的性能下降都在 2% 以内。** 这是因为视觉任务中的注意力模式比语言任务更"局部"和"对称"——一个像素块关注相邻像素块时，方向性不如语言中的语法依赖关系那么强。

### 4.3 最重磅的组合：Q=K 共享 + GQA/MQA

论文最大的亮点在于**投影共享与头共享的组合效果**：

```
┌──────────────────────────────────────────────────────────────────────┐
│  Q=K 共享与 GQA/MQA 的组合效果（KV Cache 总削减）                     │
├──────────────────────────────┬───────────────┬───────────────────────┤
│  配置                        │ KV Cache 削减  │ 剩余 KV Cache         │
├──────────────────────────────┼───────────────┼───────────────────────┤
│  标准 QKV + 64 heads         │ —             │ 100%                  │
│  QKV + GQA-8 (8 组)          │ ~87.5%        │ 12.5%                 │
│  QKV + MQA (1 组)            │ ~98.4%        │ 1.6%                  │
│                              │               │                       │
│  Q=K 共享 + GQA-4 (4 组)     │ 87.5%         │ 12.5%                 │
│  Q=K 共享 + MQA (1 组)       │ 96.9%         │ 3.1%                  │
└──────────────────────────────┴───────────────┴───────────────────────┘
```

让我们仔细算一下这个数字：

- 标准 Transformer 有 64 个 heads，每个 head 有独立的 K 和 V 投影。
- GQA-4 将 64 个 heads 分为 4 组，每组共享 K 和 V → KV Cache 削减为 $4/64 = 1/16$。
- Q=K 共享让每个 head 的 K 和 V 共享同一个投影 → 每个 head 的 KV Cache 再减半。
- **组合效果**：$(1/16) \times (1/2) = 1/32$，即 **96.9% 的削减**（实际论文报告为 96.9%，微小差异来自 head 维度的 rounding）。

**这个组合的意义是什么？**

以 Llama-3-70B 在 128K 上下文为例：

```
┌─────────────────────────────────────────────────────────────────┐
│  Llama-3-70B, 128K 上下文的 KV Cache 优化效果                    │
├──────────────────────────┬──────────────────────────────────────┤
│  配置                    │ KV Cache 大小                         │
├──────────────────────────┼──────────────────────────────────────┤
│  标准（无优化）           │ ~64 GB                                │
│  仅 GQA-4                │ ~8 GB                                 │
│  GQA-4 + Q=K 共享        │ ~4 GB                                 │
│  GQA-4 + Q=K 共享 + INT4  │ ~1 GB                                 │
└──────────────────────────┴──────────────────────────────────────┘
```

**从 64 GB 到 4 GB——这意味着原本需要 8×A100-80GB 才能运行的长上下文推理，现在可以在单张消费级 GPU（如 RTX 4090 24GB）上运行。** 这就是边缘部署的现实意义。

## 五、深层分析：为什么 K 和 V 可以共享？

### 5.1 注意力机制的"低秩"本质

论文提出了一个关键洞察：**Transformer 的注意力矩阵在实际运行时处于低秩状态。**

在理论分析中，一个 $n \times n$ 的注意力矩阵的秩最多为 $n$（token 数量）。但实证研究表明，对于自然语言输入，注意力矩阵的**有效秩**（由奇异值衰减定义）通常远小于 $n$。

```
注意力矩阵奇异值分布（示意）：

奇异值排名 →  1    2    3    4    5   ...  50   ...  100  ...  512
奇异值     → ████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
              ↑ 前 5 个奇异值占据了 80%+ 的能量
```

这意味着注意力模式实际上可以用远低于 full rank 的维度来近似。**当注意力是低秩的时，K 和 V 的表示空间存在重叠**——它们不需要完全独立的投影来捕获注意力所需的全部信息。

### 5.2 权重绑定（Weight Tying）的视角

论文将投影共享定位为 **weight tying**（权重绑定）的一个实例。Weight Tying 在 NLP 中并不新鲜：

- **输入-输出 embedding 共享**：早期 Transformer（如 GPT-2）就绑定了输入 embedding 和输出投影的权重，减少参数量并提高训练稳定性。
- **ALBERT 的跨层参数共享**：ALBERT 通过在所有层之间共享参数，将 BERT 的参数量减少了 80%。

**QKV 投影共享是 weight tying 在注意力机制内部的自然延伸。** 它利用了注意力机制中的结构冗余，而不是简单地"砍掉"参数。

## 六、生产意义：从论文到实践

### 6.1 对模型设计者的启示

**如果你正在设计一个新的 Transformer 变体：**

1. **默认考虑 Q=K 共享**。论文的结论是 Q=K 共享（K=V 共享）在语言建模中的质量损失仅 3.1%，而 KV Cache 减半。对于大多数应用场景，这是一个极具性价比的 trade-off。

2. **GQA + Q=K 共享是边缘部署的"黄金组合"**。如果你需要模型在资源受限的环境中运行，这两个技术的组合可以将 KV Cache 降低到标准配置的 3-12%。

3. **Q=K 共享（Q=K）不推荐用于语言任务**，但在视觉任务中可以接受。如果你的任务是多模态的，需要根据模态分别评估。

### 6.2 对推理引擎开发者的启示

**如果你正在开发或优化推理引擎（如 vLLM、TGI、TensorRT-LLM）：**

1. **QKV 共享需要推理引擎的显式支持**。当前主流推理引擎假设 Q、K、V 是独立的 tensor。要利用 QKV 共享，需要在 kernel 层面修改注意力计算路径。

2. **KV Cache 的内存布局需要重新设计**。当 K 和 V 共享同一个投影时，它们实际上是同一个 tensor——这意味着 KV Cache 的内存占用和访问模式都会改变。

3. **与现有的 PagedAttention 集成是可行的**。因为 QKV 共享改变的是 per-token 的 KV 大小，而 PagedAttention 管理的是 KV Cache 的分页——两者在不同抽象层工作。

### 6.3 对 Agent 开发者的启示

**如果你是 AI Agent 的开发者（比如正在使用 Claude Code、Codex 等）：**

昨天我们讨论了 Headroom 如何从**基础设施层面**通过上下文压缩减少 Token 消耗。今天这篇论文展示了另一个方向：**模型架构层面**的 KV Cache 优化。

这两个方向是互补的：

| 维度 | Headroom（基础设施层） | QKV 共享（模型架构层） |
|------|------------------------|------------------------|
| 优化目标 | 输入 Token 数量 | KV Cache 内存占用 |
| 实现位置 | Agent 框架/Proxy | 模型权重设计 |
| 适用性 | 任何模型 | 需要特定架构的模型 |
| 部署成本 | 低（软件层） | 高（需要新模型） |
| 叠加效果 | ✅ 可以叠加 | ✅ 可以叠加 |

**终极组合：Headroom（减少输入 Tokens）+ QKV 共享模型（减少 per-token KV Cache）+ GQA（减少 KV heads）+ INT4 KV Cache 量化 = 端到端 Token 和内存消耗降低 95%+。**

---

## 七、局限性与开放问题

### 7.1 论文的局限

1. **训练数据规模有限**：论文使用 10B tokens 预训练，而当前前沿模型的预训练数据在 10T+ tokens 量级。在更大规模训练下，共享约束的效果可能不同。

2. **未评估极长上下文**：实验中的上下文长度有限。在 128K+ 上下文下，对称注意力的影响可能更加显著。

3. **未涉及多模态**：虽然视觉任务有评估，但没有涉及图文多模态、视频等更复杂的场景。

### 7.2 开放问题

1. **QKV 共享与 Flash Attention 的兼容性**：Flash Attention 通过 IO-aware 的注意力计算实现加速。QKV 共享是否可以进一步简化 Flash Attention 的内存访问模式？

2. **动态共享**：是否可以让模型在运行时**动态决定**哪些 head 使用共享投影、哪些使用独立投影？这可能是一个注意力路由问题。

3. **与 MoE 的结合**：在 Mixture-of-Experts 架构中，QKV 共享是否可以与 Expert 路由协同设计？

## 八、总结：KV Cache 优化的新范式

这篇论文的贡献不仅仅是几个新的注意力变体，更重要的是它**重新定义了 KV Cache 优化的思考框架**：

```
┌────────────────────────────────────────────────────────────────┐
│  KV Cache 优化的范式转移                                       │
│                                                                │
│  过去（2023-2025）：                                           │
│  "KV Cache 是固定的，我们只能优化它的存储和调度。"               │
│  → PagedAttention、KV Cache 量化、StreamingLLM                 │
│                                                                │
│  现在（2026）：                                                │
│  "KV Cache 的大小是由注意力架构决定的，我们可以从源头减少它。"   │
│  → QKV 共享、GQA/MQA、线性注意力                              │
│                                                                │
│  未来（展望）：                                                │
│  "KV Cache 应该是一个可训练的、自适应的资源，                    │
│   根据输入内容和任务需求动态分配。"                             │
│  → 动态 KV 分配、可学习缓存策略、神经内存压缩                   │
└────────────────────────────────────────────────────────────────┘
```

**核心论点：2026 年的 AI 基础设施竞赛正在从"更大的上下文窗口"转向"更聪明的上下文利用"。** QKV 共享论文是这一趋势在模型架构层面的体现，而 Headroom 等工具则是这一趋势在工程层面的体现。两者的结合，正在让"在边缘设备上运行大模型"从一个梦想变成一个工程问题。
