# 模型路由优化：当"选哪个模型"变成一个多目标优化问题

> 你以为路由只是一个分类器。实际上，它是 Agent 基础设施中最被低估的多目标优化引擎。

2026 年 7 月 15 日，IBM Research 在 Hugging Face Blog 发表了一篇题为 [《Model Routing Is Simple. Until It Isn't.》](https://huggingface.co/blog/ibm-research/model-routing-is-simple-until-it-isnt) 的技术文章，揭示了在生产级 Agent 系统中构建模型路由器（Model Router）时遭遇的三个系统性反直觉现象。这篇文章不是学术论文，而是一份**来自生产一线的作战报告**——他们用真实的 AppWorld 基准测试数据，证明了几乎所有关于模型路由的直觉假设都是错的。

更准确地说，他们证明了一个核心论点：**模型路由不是一个分类问题，而是一个多目标优化问题。**

这篇文章之所以值得认真对待，不是因为它提出了什么新理论，而是因为它**用真实数据拆解了三个被行业广泛接受却从未被验证的假设**，并且给出了一个可操作的解决方案。这正是我近期持续关注的 Agent 基础设施栈（Agent Infrastructure Stack）拼图中的关键一块——当你有了多模型、多 Agent、多工具的生产环境后，"把正确的请求发给正确的模型"这件事的难度，远比大多数人想象的要复杂得多。

---

## 一、天真的路由假设：听起来太完美了

让我们先回顾一下行业里最流行的路由思路，几乎每一篇讲模型路由的文章都在重复同一套逻辑：

1. **按任务难度路由：** 简单任务发给便宜的小模型，复杂任务发给强的大模型
2. **按领域专长路由：** 代码任务发 Claude，多模态任务发 Gemini，数学任务发 o-series
3. **按成本路由：** 哪个模型的 token 单价低就用哪个

这套逻辑听起来无懈可击。分类器判断请求类型，然后查表分派。成本下降，质量不变。皆大欢喜。

IBM Research 的实验结果把这整套假设掀翻了。

他们在 AppWorld Test Challenge 上用同一个 CodeAct Agent 跑了 417 个任务，比较了 Claude Sonnet 4.6 和 GPT-4.1 的真实成本。结果是：

| 模型 | 总成本 | 单任务成本 | 纸面 token 单价 |
|------|--------|-----------|----------------|
| Sonnet 4.6 | **$79** | **$0.19/task** | 较高 |
| GPT-4.1 | **$155** | **$0.37/task** | 较低 |

等等——**GPT-4.1 的 token 定价明明更低，而且 Sonnet 完成同样任务需要大约 3 倍的 reasoning step 数，为什么最终成本反而高了将近一倍？**

答案只有一个词：**缓存。**

---

## 二、维度一：成本不止是定价表

### 2.1 缓存如何颠覆成本模型

这是整篇文章最反直觉的发现，也是对当前行业成本评估方法论最致命的一击。

Agent 工作负载有一个关键特征：**跨步骤的上下文复用率极高**。一个典型的 Agentic 任务会经历多轮思考-行动-观察循环，每一轮都会携带之前所有的对话历史、工具输出和中间状态。这意味着相邻步骤之间的输入上下文有大量重叠。

当缓存命中率高时，有效输入成本会急剧下降——因为缓存读取（cache read）的定价通常远低于新鲜输入（fresh input）。而 Sonnet 4.6 的缓存读取定价显著低于 GPT-4.1，这让 Sonnet 在 Agent 工作负载中获得了不成比例的成本优势，**足以同时抵消它更高的基础定价和更长的推理轨迹**。

这个发现的意义远远超出"哪个模型更便宜"的技术争论。它揭示了一个更深层的结构性问题：

> **几乎所有关于 Agent 成本的公开讨论，都在用定价表（pricing sheet）上的数字做决策，而不是用实际工作负载下的有效成本做决策。**

这就像用汽车的厂商指导价来判断出租车队的运营成本——完全忽略了最重要的变量：实际使用模式。

### 2.2 一个被忽视的成本维度分析框架

让我们把 Agent 的成本结构拆得更细一些。一次 Agent 任务的真实成本至少取决于以下变量的交织：

```
真实成本 = f(
    模型基础定价,          # 这是唯一被公开讨论的
    缓存命中率,             # 几乎从不被讨论
    任务轨迹长度,           # 偶尔被讨论
    上下文复用模式,         # 从不被讨论
    工具调用频率,           # 很少被讨论
    重试率,                # 从不被讨论
    输出 token 分布,        # 很少被讨论
    并发竞争,              # 从不被讨论
    基础设施 overhead       # 从不被讨论
)
```

**9 个变量中，只有 1 个是公开讨论的。** 这意味着基于定价表的路由决策，最多只考虑了真实成本函数的 1/9。

我之前的文章《编码 Agent 的"隐性税"》已经展示了 Harness 层的 Token 开销如何在用户开口之前就烧掉数万个 token。现在 IBM 的研究告诉我们，即使只看模型层，**定价表也只是冰山一角，真正的成本结构藏在水下的 8/9 处。**

### 2.3 缓存命中率的场景差异

不同的 Agent 模式会产生截然不同的缓存行为：

| Agent 模式 | 上下文复用特征 | 缓存友好度 |
|-----------|--------------|-----------|
| 多步推理（ReAct） | 每步携带完整历史 | ★★★★★ |
| 单步问答 | 无上下文复用 | ★☆☆☆☆ |
| 代码修复循环 | 代码上下文高度复用 | ★★★★☆ |
| 多工具编排 | 工具定义重复注入 | ★★★★☆ |
| 批量独立任务 | 无跨任务复用 | ★★☆☆☆ |

一个只按定价表路由的系统，会把所有 ReAct 模式的 Agent 任务错误地分派给"基础定价低但缓存定价高"的模型，而错过真正成本更优的选择。

## 三、维度二：复杂度不止是任务难度

### 3.1 难度的不可见性

第二个被推翻的假设是"按任务难度路由"。

这个策略的核心问题是：**任务的真实复杂度在路由时刻往往是不可见的。**

IBM 举了一个精确的例子：一个"总结这份合同"的请求，看起来是简单的文本摘要任务。但实际执行时，它可能触发：

- 文档检索（从企业知识库拉取相关条款）
- 合规性检查（自动对照监管要求）
- 多轮验证（交叉引用法律数据库）
- 格式化输出（生成结构化报告）

最终这个"简单"请求变成了包含 15 个步骤、调用 6 个工具、耗时 3 分钟的复杂工作流。

反过来，一个看起来极其专业的技术 prompt——比如"用 Rust 实现一个无锁队列"——可能恰好被某个较小的专用模型高效处理，因为它的训练数据中包含了大量系统编程相关内容。

**路由时刻的表面特征和实际执行复杂度之间的相关性，远低于直觉预期。** 这就像 judging a book by its cover——而 Agent 系统的"封面"往往是骗人的。

### 3.2 难度只是信号之一

即使你能完美预估任务难度，这也只是路由器需要考虑的众多信号之一。在生产环境中，路由器实际上在同时平衡至少 5 个维度：

```
┌─────────────────────────────────────────────┐
│                路由器信号空间                  │
├─────────────────────────────────────────────┤
│  成本   │ 质量   │ 延迟   │ 合规   │ 可靠性    │
├─────────────────────────────────────────────┤
│  真实成本 │ 任务准确率│ 端到端响应│ 数据驻留 │ 服务SLA  │
│  缓存效应 │ 领域匹配 │ 路由开销  │ 隐私规则 │ 熔断机制  │
│  定价波动 │ 幻觉率  │ 排队延迟  │ 审批清单 │ 降级策略  │
└─────────────────────────────────────────────┘
```

企业部署还会增加额外的约束层：

- **合规要求：** 某些数据不能离开特定区域，直接排除了某些模型的可用性
- **数据驻留规则：** GDPR、数据本地化法律进一步缩小了可选模型集合
- **隐私约束：** 涉及 PII 的请求必须使用经过认证的处理管道
- **审批模型列表：** 企业 IT 部门可能只批准了有限的模型白名单

一个理论上应该路由到模型 A 的请求，可能因为治理约束不得不去模型 B——而路由器需要优雅地处理这种妥协，而不是简单报错。

**路由器不是在解决一个单一问题。它在持续地同时平衡成本、质量、延迟、合规和可靠性。** 这是一个多目标优化问题，不是一个分类问题。

## 四、维度三：延迟不止是模型速度

### 4.1 用户感知的延迟≠模型推理速度

第三个被推翻的假设是"按模型速度路由"。

直觉上，小模型快、大模型慢。但用户实际体验到的端到端延迟取决于：

1. **路由决策本身的开销：** 每增加一个路由判断点，就增加一次网络往返 + 分类器推理时间
2. **基础设施因素：** 模型运行在什么硬件上？缓存是热还是冷？端点当前负载如何？
3. **路由粒度：** 整个任务路由一次（低开销、低灵活性）还是每步都路由（高开销、高灵活性）？
4. **排队延迟：** 即使模型本身很快，如果端点当前有 50 个请求在排队，用户等到的仍然是慢响应

一个理论上的"快速模型"如果跑在过载的端点上，可能比一个"慢速模型"跑在空闲端点上给出更差的用户体验。

### 4.2 路由粒度的权衡

这是路由架构设计中一个关键但被广泛忽视的决策点：

| 路由策略 | 路由频率 | 开销 | 灵活性 | 适用场景 |
|---------|---------|------|--------|---------|
| 任务级路由 | 每任务 1 次 | 低 | 低 | 简单、可预测的工作流 |
| 步骤级路由 | 每 Agent step 1 次 | 中 | 高 | 动态、不可预测的任务 |
| 混合路由 | 关键节点路由 | 可变 | 可变 | 大多数生产场景 |

IBM 的研究表明，步骤级路由（每步都选择模型）提供了最大的优化空间，但每一步都引入额外的延迟和运维复杂度。任务级路由开销最小，但会错过大量中途切换模型的机会。

**混合路由——在关键决策点进行路由——可能是大多数生产场景的甜蜜点。** 但这本身就是一个优化问题：哪些节点是"关键决策点"？这个判断本身就需要数据驱动。

## 五、IBM 的方案：从分类到优化

### 5.1 核心思想转变

IBM 的关键洞察是：**停止问"哪个模型最适合这个任务"，开始问"在所有约束条件下，哪个配置组合能最优地平衡成本、质量和延迟"。**

这不是一个语义游戏。这是一个算法范式的转变：

- **分类方法：** 输入 → 分类器 → 模型标签。输出是离散的、单一的。
- **优化方法：** 输入 → 多目标优化器 → 帕累托前沿（Pareto Frontier）。输出是一个可选择的配置集合。

### 5.2 帕累托前沿：给你选择权，而不是替你选择

IBM 在 AppWorld 测试中展示了他们路由器的帕累托前沿：

> 每个蓝色方块代表路由器的一个不同配置，勾勒出一条成本-准确率前沿线。重要的不是任何一个单点——而是路由器给了你一系列可选的操作点，让你根据优先级（成本、延迟或准确率）做选择。

关键数据点：

| 配置 | 准确率 | 总成本 | 延迟 | 对比 Opus 基线 |
|------|--------|--------|------|---------------|
| Opus 基线 | 88% | $120 | 91s | 基准 |
| 配置 1（延迟优化） | **84%** | **$93** | **83s** | 成本 -21%, 延迟 -9%, 准确率 -4% |
| 配置 2（成本优化） | ~84% | 更低 | 更高 | 进一步降低成本 |

而作为对比，一个传统的基于难度的路由器（teal diamond）落在相似的准确率区间，但成本更高——**它没有探索完整的权衡空间。**

这就是分类方法和优化方法的本质区别：分类方法给你一个"最佳答案"，但这个答案只在你预设的单一维度上最优；优化方法给你一整条前沿线，让你根据业务需求选择操作点。

### 5.3 轻量级优化器的工程挑战

把路由变成优化问题听起来很棒，但有一个实际的工程挑战：**优化器本身不能成为瓶颈。**

如果路由器需要 500ms 来做决策，而整个任务的推理只需要 2 秒，那路由器就占了 25% 的延迟——这在生产环境中是不可接受的。

IBM 的方案强调优化器需要足够轻量，这意味着：

1. **优化算法必须近似而非精确：** 不需要找到全局最优解，需要在合理时间内找到足够好的解
2. **信号采集必须高效：** 不能为了收集路由信号而引入大量额外的 API 调用
3. **配置空间必须可控：** 如果模型组合和参数空间爆炸，优化器会无法实时运行

## 六、更深层的洞察：为什么这个行业现在才意识到路由的复杂性

### 6.1 行业阶段的必然演进

模型路由问题的复杂性暴露，实际上是 AI Agent 行业成熟度的一个标志。回顾一下发展阶段：

```
阶段 1：单模型时代（2022-2024）
  → 只有一个模型，不需要路由
  → "用 GPT-4 就对了"

阶段 2：多模型试探期（2024-2025）
  → 开始有多个模型可选
  → 简单的 if-else 路由："代码用 Claude，数学用 o1"

阶段 3：规模化部署期（2025-2026）← 我们在这里
  → 企业部署了 5-10 个模型
  → 简单的 if-else 路由开始崩溃
  → 需要系统化的路由优化

阶段 4：动态路由市场（2026+）
  → 模型按需发现、按需定价
  → 实时优化的路由市场
  → 路由器本身成为 Agent 基础设施的核心组件
```

我们正处在阶段 3 的中段。IBM 的研究之所以重要，**因为它标志着行业开始认真对待路由的系统性复杂性，而不是用启发式规则糊弄过去。**

### 6.2 与"隐性税"问题的同构性

这和我之前写的《编码 Agent 的"隐性税"》文章揭示的问题是**同构**的：

- 那篇文章说：**Harness 的隐性开销（系统提示词、工具 Schema）是 Agent 成本的最大变量，但没人测量它。**
- 这篇文章说：**路由的真实成本（缓存效应、基础设施因素）是模型选择的最大变量，但没人考虑它。**

两个问题指向同一个结构性盲区：**我们在用表面指标（token 单价、模型基准分数）做决策，但真正决定 Agent 系统表现的是深层的、系统级的、高度依赖使用模式的变量。**

这种盲区的根源是什么？我认为是：

1. **供应商激励不匹配：** 模型供应商有动机宣传基础定价，但没有动机披露缓存定价的复杂效应
2. **测量工具缺失：** 直到最近才有人开始在 Agent 和 API 之间架日志代理做精确测量
3. **思维惯性：** 工程师习惯用"基准分数÷定价=性价比"的简单公式做决策

### 6.3 路由器可能成为新的"元问题"

这里有一个值得警惕的元问题（meta-problem）：

> **如果路由器本身需要被优化，那谁来做路由器的路由？**

随着路由策略的复杂化，路由器本身变成了一个需要被调优、被监控、被评估的系统。你需要：

- 监控路由器的决策质量（路由对了还是错了？）
- 监控路由器的性能（路由器本身是否成为瓶颈？）
- 评估路由器的 ROI（用了路由器到底省了多少钱、提了多少质量？）
- 调试路由器的错误决策（为什么这个请求被路由到了错误的模型？）

**路由器不再是配置项，而是一个需要被运维的第一公民系统。** 这意味着 Agent 可观测性（Observability）栈中需要增加一个新的维度：路由可观测性。

## 七、实践建议：如何开始构建你的路由优化系统

### 7.1 第一步：测量，不要猜测

在你开始构建任何路由逻辑之前，先回答以下问题：

1. **你的 Agent 工作负载的缓存命中率是多少？** 在每个模型上分别测量
2. **你的任务的真实轨迹长度分布是什么？** 不是预估的，是实际跑出来的
3. **你的每个工具调用的上下文注入量是多少？** 这直接影响缓存行为
4. **你的端到端延迟的分解是什么？** 路由决策、模型推理、网络传输各占多少

没有这些数据，任何路由策略都是盲人摸象。

### 7.2 第二步：从混合路由开始

不要一上来就构建全步骤路由。建议的路径是：

```
1. 任务级路由（if-else）→ 建立基线
2. 在关键节点增加路由判断 → 验证收益
3. 逐步增加路由频率 → 找到甜蜜点
4. 引入多目标优化 → 系统化权衡
```

每一步都需要 A/B 测试来验证收益是否覆盖引入的复杂度。

### 7.3 第三步：构建路由可观测性

你的路由系统需要至少以下监控指标：

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| 路由命中率 | 各模型被选中的比例分布 | 偏离基线 > 20% |
| 路由准确率 | 路由决策后任务的成功率 | 低于基线模型 5% |
| 路由延迟 | 路由决策本身耗时 | 超过任务推理时间的 10% |
| 成本节省率 | 相比全用最强模型节省的比例 | 负值（说明路由反而更贵） |
| 回退率 | 路由失败或降级的比例 | 超过 5% |

### 7.4 第四步：缓存策略是路由策略的一部分

这是大多数团队忽视的关键点：**你的缓存策略和路由策略是耦合的，必须联合优化。**

如果你的路由策略把相似请求分散到不同模型，你可能在破坏每个模型的缓存命中率。反过来，如果你的缓存策略导致某个模型的缓存持续不命中，你可能需要考虑把更多请求路由到有更好缓存定价的模型。

**缓存 × 路由 = 联合优化问题。** 把它们分开优化，得到的局部最优解很可能在联合维度上是次优的。

## 八、未来展望：路由器会成为 Agent 的"操作系统调度器"吗？

让我们做一个大胆的类比。

操作系统的进程调度器负责决定：在有限的 CPU 时间片下，哪个进程应该在什么时候运行。它需要考虑优先级、公平性、响应时间、吞吐量——本质上是一个多目标优化问题。

**Agent 系统中的路由器，正在演变成类似的"模型调度器"角色。**

它需要决定：在有限的预算、延迟约束和质量要求下，哪个模型应该在什么时候处理哪个子任务。它需要考虑成本、质量、延迟、合规、可靠性——本质上也是一个多目标优化问题。

随着 Agent 系统越来越复杂，路由器的角色会从"辅助工具"升级为"核心基础设施"。未来的 Agent 平台竞争，可能很大程度上取决于**谁的路由器更聪明**。

但这带来一个新的风险：**路由器的复杂度本身可能成为 Agent 系统的新瓶颈。** 就像操作系统调度器的过度优化反而会导致上下文切换开销爆炸一样，过度复杂的路由策略可能让 Agent 系统陷入"路由税"（Routing Tax）——花在路由决策上的资源超过了路由带来的收益。

**最优的路由策略，不是最智能的路由策略，而是路由收益恰好覆盖路由成本的策略。** 这是一个需要持续测量的动态平衡点。

---

## 九、结语

IBM Research 的这篇博客之所以重要，不是因为它提出了什么惊天动地的新理论，而是因为它**用真实数据捅破了一个行业泡沫**：模型路由远没有大家想象的那么简单。

当你发现 GPT-4.1 比 Sonnet 4.6 贵了一倍，而它的定价表明明更便宜；当你发现按任务难度路由的策略在真实成本上反而更高；当你发现路由决策本身的延迟可能抵消模型速度带来的收益——你会意识到：

**Agent 时代的工程挑战，正在从"选哪个模型"转移到"如何让多个模型在一个系统里协同工作"。**

这不是一个模型问题。这是一个系统工程问题。

而系统工程师们，是时候把路由当作一个严肃的多目标优化问题来对待了——用数据说话，用实验验证，用帕累托前沿做决策，而不是用 if-else 和直觉糊弄过去。

毕竟，当你的 Agent 系统开始在生产环境里跑的时候，"差不多就行"的路由策略，最终会变成"差很多"的成本账单。
