## 一、引言：Benchmark 分数很高，为什么 Agent 还是不能用？

2026 年 5 月初，IBM Research 在 Hugging Face Blog 发布了一篇深度分析文章——《Inside VAKRA: Reasoning, Tool Use, and Failure Modes of Agents》，同时开源了 VAKRA Benchmark 的完整数据集和执行环境。这篇文章在 AI Agent 社区引发了远超普通论文发布的关注。原因很简单：**它用最残酷的方式揭示了当前 AI Agent 评估体系的系统性失效。**

VAKRA 的全称是 **Verifiable Agent Knowledge Reasoning & Action**（可验证的 Agent 知识推理与行动基准）。它不是又一个选择题测试集，而是一个**可执行的、基于真实 API 交互的评估环境**。它包含 8,000+ 个本地托管的 API、62 个行业领域的真实数据库，以及需要 3-7 步组合推理才能完成的复杂任务。

结果令人震惊：**主流大模型在 VAKRA 上的表现集体崩溃。** 那些在 MMLU、GPQA、AgentBench 上轻松刷到 90%+ 的模型，在面对真实企业级多步骤工作流时，成功率骤降到令人尴尬的低位。

这不是某个特定模型的失败，而是整个评估范式的失败。本文将深入分析 VAKRA 的设计哲学、失败模式，以及它对整个 AI Agent 行业意味着什么。

## 二、VAKRA 是什么：一个"可执行"的评估环境

### 2.1 传统 Benchmark 的根本缺陷

要理解 VAKRA 的价值，先要看清传统 Agent Benchmark 的致命缺陷。

当前主流的 Agent 评估体系——AgentBench、AgentInstruct、WebArena、ToolBench——本质上都是**模拟环境**。它们要么使用预定义的工具列表（通常不超过 20 个），要么依赖静态的 JSON 数据集，要么在有限的模拟网页上进行操作。这种设计存在三个根本问题：

**问题一：工具空间过于狭窄。** 一个包含 20 个工具的 Benchmark，无法测试 Agent 在真实企业环境中面对数百个 API 时的选择能力。OpenAI 的 API 规范明确限制工具列表最多 128 个——这意味着在企业场景下，Agent 必须具备**工具筛选和短列表管理能力**，而这在传统 Benchmark 中从未被测试过。

**问题二：缺乏真实数据分布。** 许多 Benchmark 使用合成数据或简化的数据库结构。但真实企业数据的复杂程度——缺失值、异常值、跨表关联、非标准化字段——远超任何合成数据能模拟的范围。

**问题三：组合推理深度不足。** 大多数 Benchmark 的任务只需 1-2 步工具调用即可完成。而真实的企业工作流通常需要 3-7 步的连续推理和操作链，每一步的输出都是下一步的输入。

### 2.2 VAKRA 的设计哲学

VAKRA 的设计直接针对上述三个缺陷：

```
┌─────────────────────────────────────────────────────────────────┐
│                    VAKRA 评估架构                                │
│                                                                 │
│  ┌─────────────┐    ┌──────────────┐    ┌───────────────────┐  │
│  │  Agent 模型  │───▶│  MCP Server  │───▶│ 本地数据库 (62域) │  │
│  │ (任何 LLM)   │◀───│ (8000+ API)  │◀───│ (真实数据分布)     │  │
│  └─────────────┘    └──────────────┘    └───────────────────┘  │
│         │                                                        │
│         ▼                                                        │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  执行追踪器：记录每一步工具调用、参数、返回值、时间戳       │   │
│  │  验证器：检查最终答案正确性 + 推理路径有效性               │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

VAKRA 包含四个能力维度，每个维度都测试 Agent 的不同面向：

| 能力维度 | 测试实例数 | 领域数 | 工具链深度 | 核心挑战 |
|---------|-----------|--------|-----------|---------|
| **API 链式调用** | 2,077 | 54 | 1-12 步 | 多步骤数据操作与筛选 |
| **工具选择** | 1,597 | 17 | 单次选择 | 从 6-328 个工具中精准定位 |
| **多跳推理** | 869 | 38 | 多步骤 | 跨 API 的信息聚合与推理 |
| **文档+API 混合** | 若干 | 多域 | 混合 | 结构化与非结构化信息的联合推理 |

### 2.3 一个真实案例：从足球数据到企业决策

让我们看一个 VAKRA 中的实际任务实例：

> **查询：** "哪个足球队的 build-up play speed 为 31，build-up play dribbling 为 53，build-up play passing 为 32？"

这个看似简单的问题需要 Agent 执行以下步骤：

```json
{
  "tool_calls": [
    {"name": "get_data", "arguments": {"tool_universe_id": "486ea46224d1-aeb8037c5e78"}, "label": "retrieved_data_1"},
    {"name": "select_data_equal_to", "arguments": {"data_label": "retrieved_data_1", "key_name": "play_speed", "value": 31}, "label": "FILTERED_DF_0"},
    {"name": "select_data_equal_to", "arguments": {"data_label": "FILTERED_DF_0", "key_name": "play_dribble", "value": 53}, "label": "FILTERED_DF_1"},
    {"name": "select_data_equal_to", "arguments": {"data_label": "FILTERED_DF_1", "key_name": "play_passing", "value": 32}, "label": "FILTERED_DF_2"},
    {"name": "get_team_name", "arguments": {"data_label": "FILTERED_DF_2", "n": 1}}
  ],
  "answer": "FC Barcelona"
}
```

这不是简单的"调用一个 API"，而是**一个完整的数据管道**：初始化数据源 → 第一次筛选 → 第二次筛选 → 第三次筛选 → 提取最终答案。任何一步出错，整个链条就断裂了。

## 三、Agent 为什么失败：五大致命故障模式

VAKRA 团队对失败模式进行了系统性分析，揭示了 Agent 在真实企业环境中的五个致命弱点。

### 3.1 工具选择灾难：当 328 个 API 摆在面前

VAKRA 的 Capability 2 测试了 Agent 在大规模工具集中的选择能力。每个领域平均有 116 个工具，最多的达到 328 个。而 OpenAI 的 API 规范明确限制工具列表最多 128 个——这意味着：

**Agent 必须具备工具短列表（shortlisting）能力。**

但当前大多数 Agent 框架根本没有这个功能。当工具数量超过 LLM 的上下文窗口或 API 限制时，Agent 要么随机截断工具列表，要么根本无法启动。VAKRA 团队在 baseline 实现中加入了一个简单的工具短列表组件，但这恰恰说明：**工具发现和筛选本身就是 Agent 能力的核心组成部分**，而不是一个可以忽略的前置步骤。

> **关键洞察：** 在真实企业环境中，Agent 面临的不是"20 个工具选 1 个"的选择题，而是"在 500 个 API 中找到你需要的那 3 个"的搜索问题。这与传统 Benchmark 的设定有着本质的区别。

### 3.2 链式推理崩溃：多步骤操作的误差累积

在 Capability 1 和 3 中，任务需要 3-12 步的连续工具调用。这里存在一个**误差累积效应**：

```
步骤 1 正确率: 85%
步骤 2 正确率: 85%
步骤 3 正确率: 85%
...
步骤 7 整体正确率: 85%^7 ≈ 32%
```

即使每一步都有 85% 的成功率（这在单步 Benchmark 中已经是非常优秀的成绩），7 步任务的整体成功率也只有 32%。而现实中，Agent 在复杂操作中的单步正确率往往远低于 85%。

更糟糕的是，**误差不仅累积，还会放大**。第 1 步的错误输出可能让第 2 步在一个完全不同的数据集上操作，导致后续所有步骤都基于错误的前提。

### 3.3 数据协议适配失败：LLM 不懂 DataFrame

VAKRA 的设计中有一个精妙的细节：`get_data` 工具不会返回完整数据集，而是返回一个**数据预览**：

```json
{
  "handle": "retrieved_data_1",
  "num_records": 2,
  "key_details": [
    {"name": "team_name", "dtype": "str", "first_3_values": ["FC Barcelona", "Manchester City"]},
    {"name": "play_speed", "dtype": "int32", "first_3_values": [31, 40]},
    {"name": "play_dribble", "dtype": "int32", "first_3_values": [53, 30]},
    {"name": "play_passing", "dtype": "int32", "first_3_values": [32, 16]}
  ]
}
```

这种设计避免了通过 MCP 协议传输大量数据的低效问题，但也给 Agent 提出了新的挑战：**它必须仅凭数据预览（schema + 样本值）就理解数据结构，并生成正确的后续操作。**

许多 Agent 模型在处理这种"不完整数据"时表现拙劣。它们习惯了 Benchmark 中那种"数据完整给你"的模式，面对这种需要"看 schema 猜结构"的企业级场景时，往往无法做出正确的推理。

### 3.4 领域知识缺失：从足球到金融，Agent 的知识断层

VAKRA 覆盖了 62 个行业领域，从体育数据到金融指标，从供应链到人力资源。这意味着 Agent 不仅要具备工具使用能力，还要具备**跨领域的领域知识**。

一个 Agent 可能在"查询客户信息"的任务上表现良好，但面对"筛选 build-up play speed 为 31 的足球队"时，它可能根本不知道 `play_speed` 和 `play_dribble` 是什么含义，应该用什么工具来操作。

这暴露了当前 Agent 训练的一个根本矛盾：**工具使用能力与领域知识之间的耦合问题。** 我们训练 Agent "会用工具"，但工具的语义含义完全取决于领域上下文。没有领域知识的工具调用，就像一个不懂 SQL 的人面对数据库——他知道要"查询"，但不知道怎么写 `WHERE` 条件。

### 3.5 非结构化与结构化的鸿沟：混合推理的困境

VAKRA 的 Capability 4 测试了 Agent 同时处理结构化 API 和非结构化文档的能力。这是最贴近真实企业场景的任务：

> "查找最近一个季度销售额下降的产品，并阅读相关市场分析报告，给出可能的原因。"

这需要 Agent 在两种完全不同的数据模式之间切换：
- **结构化数据**：API 调用、数据库查询、数值计算
- **非结构化数据**：文档检索、语义理解、文本摘要

当前大多数 Agent 框架将这两种能力割裂处理——RAG 系统负责文档，Tool Use 负责 API。但在真实场景中，**信息需要在两种模式之间流动和融合**。这正是 VAKRA 要测试的核心能力。

## 四、为什么现有 Benchmark 测不出这些问题？

### 4.1 Benchmark 的"温室效应"

用一个比喻来说：当前的 Agent Benchmark 就像温室里培育的植物——温度恒定、湿度可控、没有害虫。而 VAKRA 模拟的是真实的企业环境——风吹日晒、土壤酸碱度变化、病虫害随时可能出现。

| 维度 | 传统 Benchmark | VAKRA |
|------|--------------|-------|
| 工具数量 | 5-20 个 | 6-328 个/域 |
| 数据源 | 合成/静态 | 8000+ API，真实数据库 |
| 推理深度 | 1-2 步 | 3-12 步 |
| 领域覆盖 | 1-3 个 | 62 个 |
| 评估方式 | 最终答案匹配 | 执行路径追踪 + 答案验证 |
| 数据模式 | 单一 | 结构化 + 非结构化混合 |

### 4.2 "过拟合 Benchmark"的系统性风险

更令人担忧的是，随着 Agent Benchmark 的普及，社区已经开始出现**过拟合现象**。模型和 Agent 框架的开发者会针对特定 Benchmark 进行优化——调整 prompt 模板、微调工具描述、甚至直接针对测试集进行 fine-tuning。

VAKRA 的设计从源头阻止了这种过拟合：
- **可执行环境**：答案不是预设的文本，而是通过真实 API 执行得到的结果
- **大规模工具集**：无法为每个工具写专门的 prompt
- **跨领域覆盖**：无法针对某个特定领域进行优化
- **数据预览机制**：Agent 必须动态理解数据结构

这就像是从"开卷考试"变成了"实操考核"——你没法靠背答案通过。

## 五、与其他 Agent 评估趋势的关联

### 5.1 从"选择题"到"实操题"的范式转移

VAKRA 不是孤立的。2026 年以来，Agent 评估领域正在经历一场范式转移：

- **VAKRA（IBM）**：8000+ API 的可执行评估环境
- **Local Deep Research**：~95% SimpleQA 分数，在本地 LLM 上实现，强调可验证的研究能力
- **VectifyAI PageIndex**：无向量、基于推理的 RAG 索引，挑战传统 RAG 评估范式
- **ServiceNow PipelineRL**：揭示了 RL 训练中推理引擎正确性对评估的致命影响

这些项目的共同点是：**不再满足于"模型回答了正确的问题"，而是要求"模型在真实环境中正确执行了正确的操作"。**

### 5.2 评估成本危机

但这里有一个尖锐的矛盾：VAKRA 这样的可执行评估环境，其计算成本远高于传统 Benchmark。

运行一次 VAKRA 评估需要：
1. 部署 8000+ 个 API 服务
2. 维护 62 个领域的数据库
3. 对每个 Agent 执行完整的工具调用链
4. 追踪和验证每一步的执行结果

相比之下，跑一次 MMLU 只需要一个模型 + 一个评分脚本。

这直接引出了另一个话题：**评估成本正在成为 Agent 发展的瓶颈。** 正如 5 月 4 日我们在《AI 评测的计算成本危机》中讨论的，当评估的成本超过训练时，谁来为真相买单？

## 六、对 Agent 开发的启示：从 Benchmark 到生产

### 6.1 工具编排能力的重新定义

VAKRA 告诉我们，Agent 的工具编排能力不仅仅是"选择合适的工具"，而是包含以下层次：

```
层级 5: 混合推理 ── 在结构化 API 和非结构化文档之间自由切换
         │
层级 4: 跨领域迁移 ── 将工具使用能力应用到陌生领域
         │
层级 3: 链式操作 ── 多步骤操作的误差控制和回滚机制
         │
层级 2: 工具发现 ── 从数百个 API 中筛选出需要的子集
         │
层级 1: 单步调用 ── 给定工具列表，选择正确的工具
```

大多数 Agent 框架还停留在层级 1-2，而企业级应用需要层级 4-5。

### 6.2 三个具体的改进方向

**方向一：工具短列表（Tool Shortlisting）作为基础设施。**

每个生产级 Agent 系统都应该内置工具发现和筛选能力。这不是可选的优化，而是面对大规模工具集时的必备能力。可以参考 VAKRA baseline 中的 `tool_shortlister.py` 实现，结合语义搜索和领域分类来动态构建工具子集。

**方向二：执行路径的容错与回滚机制。**

在多步骤操作中，Agent 需要能够检测中间步骤的异常并回滚。这类似于数据库事务的 ACID 特性——要么全部成功，要么安全回滚到检查点。当前大多数 Agent 框架缺乏这种机制，一旦中间步骤出错，整个操作链就彻底失败。

**方向三：领域知识的注入策略。**

与其让 Agent "从零学习"每个领域，不如建立结构化的领域知识注入管道。当 Agent 进入一个新领域时，先加载该领域的工具描述、数据 schema 和常见操作模式，再进行工具选择。这类似于人类专家进入新领域时的"速成学习"过程。

## 七、结论：Benchmark 的终结，还是 Benchmark 的重生？

VAKRA 的意义不在于"证明当前 Agent 不行"——这一点大家都知道。它的真正价值在于**提供了一个可执行的、难以作弊的、贴近真实场景的评估标准**。

在 VAKRA 之前，我们问的是："你的 Agent 在 MMLU 上多少分？"

在 VAKRA 之后，我们应该问的是："你的 Agent 能在我公司的 500 个 API 上完成日常工作流吗？"

这两个问题的答案，可能截然不同。

对于 Agent 开发者来说，VAKRA 是一个警钟：**Benchmark 分数再高，也不等于生产可用。** 真正衡量 Agent 能力的，不是它在标准测试集上的表现，而是它在真实、复杂、充满噪声的企业环境中完成实际工作的能力。

对于整个行业来说，VAKRA 代表了一种新的评估哲学：**不要测试 Agent 知道什么，测试它能做什么。**
