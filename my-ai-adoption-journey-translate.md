我的经验是，采用任何真正有分量的工具时，我都会经历三个阶段：(1) 低效期，(2) 勉强够用期，最后是 (3) 改变 workflow 乃至生活的发现期。

在大多数情况下，我不得不强迫自己度过第 1 和第 2 阶段，因为我通常已经有一套自己满意且熟悉的 workflow。采用新工具感觉像是额外的工作，我**不想**付出努力，但为了成为一个更全面的从业者，我通常还是会这么做。

这是我如何发现 AI 工具价值以及接下来尝试什么的旅程。在一个充满过度戏剧化和炒作观点的海洋中，我希望这代表了我对 AI 的看法及其随时间变化的更细致、更谨慎的态度。

**这篇博客文章完全是手工写的，用我自己的话。我讨厌不得不说这个，但考虑到主题，我想明确说明这一点。**

---

## 第一步：放弃 Chatbot

立即停止尝试通过 Chatbot（例如 ChatGPT、网页版 Gemini 等）完成真正有意义的工作。Chatbot 当然有价值，也是我日常 AI workflow 的一部分，但它们在编码上的实用性非常有限，因为你大多只能寄希望于它们根据既有训练直接给出正确答案；一旦出错，就需要人类（也就是你）反复告诉它们错在哪里。这很低效。

我认为每个人的第一次 AI 体验都是通过聊天界面。我认为每个人尝试用 AI 编码的第一次体验都是要求聊天界面写代码。

虽然我曾经是一个重度 AI 怀疑论者，但我的第一个"哇哦"时刻是将 Zed 命令面板的截图粘贴到 Gemini 中，要求它用 SwiftUI 复现它，结果它做得**非常好**。Ghostty 今天随 macOS 发布的命令面板只比 Gemini 几秒钟内生成的内容做了轻微修改。

但当我尝试在其他任务上复现这种体验时，我失望了。在 brownfield 项目的语境里，我发现聊天界面经常给出糟糕的结果，而我自己则不停地在不同界面之间复制粘贴代码和命令输出，沮丧不已。这显然比我亲自上手干活低效得多。

要找到价值，你**必须**使用 **Agent**。Agent 是业界通用术语，指能够在循环中对话并调用外部能力的 LLM[^1]。最低限度，Agent 必须能够：读取文件、执行程序、发出 HTTP 请求。

## 第二步：复现你自己的工作

我旅程的下一阶段，是尝试 [Claude Code](https://github.com/anthropics/claude-code)。直说吧：我一开始并不觉得它有多惊艳。我就是没法从 session 里拿到足够好的结果。我感觉它产出的所有东西我都得手动修，而这个过程花的时间比我自己直接做还多。我读了博客文章，也看了视频，但依然没有被真正打动。

我没有放弃，而是**强迫自己把所有手工 commit 都再用 Agent 重做一遍**。我真的把同一份工作做了两遍：先手动完成，再跟 Agent 周旋，让它产出在质量和功能上都完全一致的结果（当然，不能让它看到我的手工解法）。

这**极其痛苦**，因为它妨碍了我把事情直接做完。但我用过足够多非 AI 的工具，知道这种摩擦是自然的；如果我不把该花的时间和精力都花足，就无法得出坚定、站得住脚的结论。

但是，专业能力也正是在这个过程中形成的。我很快从第一性原理出发，自己发现了那些别人早就在说的东西；而亲自发现这件事，又带来了更扎实的底层理解。

- 将 session 拆成彼此独立、清晰且可执行的任务。不要试图在一个 mega session 里一步到位完成所有事。
- 对于模糊的请求，把工作拆成独立的 planning session 与 execution session。
- 如果你给 Agent 一种验证自己工作的办法，那么多数情况下它会自己修正错误，并防止 regression。

更一般地说，我也逐渐摸清了 Agent——至少在当时——擅长什么、不擅长什么，以及对于那些它们擅长的任务，怎样才能拿到我想要的结果。

所有这些都带来了显著的效率提升，以至于我开始很自然地使用 Agent，而且感觉它已经不比我亲自做更慢了（虽然我依然不觉得它更快，因为我大部分时间都在 babysit Agent）。

这里的“反面空间”值得再强调一次：效率提升的一部分，恰恰来自知道何时**不**该用 Agent。把 Agent 用在它大概率会失败的事情上，显然是在浪费时间；而知道如何避开这些场景，本身就能节省大量时间[^2]。

在这个阶段，我已经从 Agent 身上获得了足够的价值，也乐于把它们纳入自己的 workflow；只是我依然不觉得自己获得了净效率提升。不过我并不在意，到这个时候，我已经接受了 AI 作为一种工具的价值。

## 第三步：每日收尾 Agent

为了进一步挖掘一点效率，我接下来尝试了一个新模式：**每天最后 30 分钟启动一个或多个 Agent**。我的假设是：**也许**如果 Agent 能在我本来就无法工作的时间里取得一些**正向进展**，那我就能获得额外效率。也就是说：不是试图在我拥有的时间里做更多，而是试图在我没有的时间里做更多。

与之前的任务类似，我起初发现这既不成功又令人烦恼。但是，我很快发现了不同类别的工作，非常有帮助：

- **Deep research session**：我会让 Agent 调研某个领域，比如找出某门特定语言里、满足特定 license 类型的所有库，并为每个库生成多页摘要，覆盖其优缺点、开发活跃度、社区舆情等。
- **并行 Agent 尝试不同的模糊想法**：我脑子里有一些想法，但没时间真正启动。我并不期待它们直接产出可发布的东西，但它们也许能在我第二天处理这些任务时，照亮一些原本未知的未知。
- **Issue 和 PR 的 triage/review**。Agent 很擅长使用 `gh`（GitHub CLI），所以我手动写了一个快捷方法，并行启动多个 Agent 来做问题分类。我**不**允许 Agent 直接回复，我只想在第二天拿到报告，帮我把注意力引向高价值或低投入的任务。

需要说明的是，我并没有像有些人那样让 Agent 整夜跑 loop。大多数情况下，Agent 会在不到半小时内完成任务。但工作日的后半段，我通常已经很累，正在退出心流，个人效率也明显下降；所以把这段时间拿来启动这些 Agent，反而会让我在第二天早上获得一种“warm start”，比原本更快进入工作状态。

我很开心，因为我开始感觉自己比引入 AI 之前做得更多了，哪怕只是多了一点点。

## 第四步：外包必胜任务

到这个阶段，我已经非常清楚 AI 擅长什么、不擅长什么。对于某些任务，我几乎可以确定 AI 会给出一个基本正确的解法。所以我旅程的下一步就是：**让 Agent 去做所有这类工作，而我去做别的事**。

更具体地说，我会在每天开始时查看前一晚那些 triage Agent 的结果，手动筛选出几乎可以确定 Agent 能很好解决的问题，然后让它们继续在后台运行（一次一个，不并行）。

与此同时，**我去做别的事情**。我不会因此比平时更沉迷社交媒体，也不会去看视频之类的。我只是回到自己那种正常的、AI 出现之前就有的深度思考模式里，去做我想做或必须做的工作。

**这个阶段非常重要：关闭 Agent 的桌面通知**。context switching 非常昂贵。为了保持效率，我发现应该由我这个人来决定何时打断 Agent，而不是反过来。不要让 Agent 来通知你；在你工作中的自然停顿点，再切过去看一眼，然后继续往下做。

重要的是，我认为“去做别的事”这一点，有助于对冲那篇广为传播的 [Anthropic skill formation paper](https://www.anthropic.com/research/AI-assistance-coding-skills)。你实际上是在做一种 trade-off：你不会在那些委托给 Agent 的任务上形成技能，但你仍会在自己继续手动完成的任务上自然形成技能。

到这个时候，我已经明确进入“再也回不去了”的阶段。我感觉自己更高效了；但即便事实并非如此，我最喜欢的一点仍然是：我现在可以把编码和思考集中在那些我真正喜欢的任务上，同时依然把那些我不喜欢的任务做得足够好。

## 第五步：工程化 Harness

冒着说废话的风险：当 Agent 第一次就产出正确结果，或者至少产出只需要极少修改的结果时，效率会高得多。要做到这一点，最可靠的方法就是给 Agent 提供快速、高质量的工具，让它能自动知道自己什么时候做错了。

我不知道这是不是一个已经被行业广泛接受的说法，但我已经习惯把它叫作 "Harness Engineering"。它的核心思想是：每当你发现 Agent 犯了一个错，你就花时间工程化出一套方案，让 Agent 以后都别再犯同样的错。我不需要在这里发明新术语；如果将来出现了更通行的说法，我也会顺势采用。

这有两种形式：

- **更好的 implicit prompting（AGENTS.md）**。对于一些简单问题，比如 Agent 反复运行错误命令，或总是找到错误的 API，就去更新 `AGENTS.md`（或等效文件）。这是 [Ghostty 的示例](https://github.com/ghostty-org/ghostty/blob/ca07f8c3f775fe437d46722db80a755c2b6e6399/src/inspector/AGENTS.md)。那个文件里的每一行，几乎都对应一种糟糕的 Agent 行为，而它几乎把这些问题全都解决掉了。
- **真正写出来的工具**。例如截图脚本、按条件筛选运行测试的脚本等等。通常这也会和 AGENTS.md 的修改配套出现，好让 Agent 知道这些工具已经存在。

**这就是我今天所在的位置**。每当我看到 Agent 做出某件坏事，我都会认真地想办法，防止它以后再做同样的坏事。反过来说，我也会认真努力让 Agent 能够验证：自己现在做的是一件好事。

---

## 第六步：始终保持 Agent 运行

与第 5 步同时推进的，还有一个目标：**始终保持 Agent 在运行**。如果没有 Agent 在跑，我就会问自己：“现在有没有什么事情是 Agent 可以替我做的？”

我尤其喜欢把这件事和更慢、也更深思熟虑的模型结合起来，比如 Amp 的 [deep mode](https://ampcode.com/news/deep-mode)（基本上就是 GPT-5.2-Codex）。它可能要花 30 多分钟才能完成一些小改动，但另一面是，它确实往往能给出非常好的结果。

**我还没有同时运行多个 Agent，而且目前也不太想这么做**。对我来说，现在只运行一个 Agent 是一种很好的平衡：我既能做自己喜欢的深度、手工工作，也能顺手照看这个有点笨、却又神秘地高产的机器人朋友。

“始终保持 Agent 在运行”目前仍然只是一个目标。以现在来看，在一个正常工作日里，我大概只有 10% 到 20% 的时间真正有效地让后台 Agent 跑起来。不过，我正在积极改进这一点。

**我不想为了运行 Agent 而运行 Agent**。我只想在有真正能帮到我的任务时才让它们运行。这个目标的一部分挑战，在于持续改进我自己的 workflow 和工具，让我手头始终有高质量、可委托的 workflow。即便没有 AI，这件事也同样重要。

## 今天

这就是我今天的位置。

通过这段旅程，我个人已经走到了一个阶段：我能比较成功地使用现代 AI 工具，而且我相信自己是以一种扎根现实、足够审慎的态度在接近它。我其实并不在乎 AI 最终会不会留下来[^3]；我只是一个软件工匠，单纯因为热爱这门手艺而想不断做东西。

整个领域变化得太快了，我很确定自己很快就会回头看这篇文章，并嘲笑此刻的天真。但正如那句老话所说：如果你不会为过去的自己感到尴尬，那你大概就没有在成长。我只希望自己是朝着正确的方向成长。

我在这里没有任何利益相关[^4]；当然，除了实际效用之外，人们也完全可能基于其他理由选择不使用 AI。我完全尊重每个人自己的决定。我不是来这里说服你的！如果你恰好对此感兴趣，我只是想分享我自己是如何摸索这些新工具的，也顺带展示一下：**一般来说**，无论是不是 AI，我都是如何面对新工具的。

# 深入解读与启示（译者补充）

这一部分不是原文直译，而是我在完整对照英文原文后，对文章脉络做的精简提炼。

## 这篇文章最值得读的地方

这篇文章最有价值的地方，不在于作者“支持 AI”还是“反对 AI”，而在于他把自己的采用过程拆成了一条非常诚实的学习曲线：先低效、再勉强够用、最后才进入真正改造 workflow 的阶段。它提醒我们，AI 的价值并不是一上来就能兑现的，很多时候反而要先经历一段明显更慢、更痛苦的适应期。

另一个重要点是，作者强调的从来不是“把工作扔给模型就好”，而是把 **Agent、验证方法、AGENTS.md、工具脚本、通知管理** 一起视为一个完整系统。换句话说，真正带来收益的不是某个模型本身，而是围绕模型搭起来的工作方式。

## 六个阶段背后的真正主线

回头看全文，这六个阶段其实可以浓缩成三条主线：

1. **先理解边界，再谈效率。**  
   作者前半段花了大量时间去确认 Chatbot 不适合什么、Agent 适合什么。这种“先摸清边界”的过程，本身就是效率的一部分。

2. **把“试用工具”升级为“重写 workflow”。**  
   从 end-of-day Agent，到 outsource the slam dunks，再到 always have an Agent running，作者真正改变的并不是某个局部动作，而是每天如何分配自己的注意力与时间。

3. **把错误沉淀成系统能力。**  
   “Harness Engineering” 是全文最关键的一个概念。它不是一次性的技巧，而是一种长期习惯：发现 Agent 会犯什么错，就反过来补一层 prompt、文档、脚本或验证工具，让同样的错误不要再次发生。

## 对今天做 AI 开发的人，有什么直接启发

如果只提炼成最实用的几点，我认为有三条：

- **不要把 Chatbot 体验误当成 Agent 体验。** 很多人在第一步就因为 Chatbot 写代码效果一般，而过早得出“AI 没用”的结论。
- **给 Agent 可验证的任务，收益会高很多。** 一旦任务能被测试、脚本、截图、命令输出或其他工具校验，Agent 的表现通常会稳定很多。
- **人依然要掌控节奏。** 关闭通知、避免 context switching、不为了运行 Agent 而运行 Agent，这些看似不起眼，但其实决定了你是在利用工具，还是被工具反过来牵着走。

## 我自己的一个补充判断

这篇文章最成熟的地方，是作者始终保留了边界感。他既没有把 AI 神化成万能生产力，也没有因为早期低效就直接否定它；他只是持续地测试、校准、保留有效部分。这种态度比任何单一技巧都更重要，因为模型、产品和工程实践都还在快速变化，但“谨慎试用、逐步工程化、把经验沉淀为系统”的方法，反而更可能长期成立。
