## 为什么需要记忆？

因为这对于代理的身份和性能都是至关重要的部分。我在 Anthropic 的大量工作都是关于记忆的，所以我自然很想知道 Claude 的记忆方法是否有根本性的不同。

Claude 没有一个记忆系统。它有**五个**，每个都有不同的目的、成本和最终用户。

### 1. 对话历史（Conversation History）

这是最普通的。它是用户提示和 Claude 响应的 JSONL 记录，你可以通过 `/export` 命令导出这个记忆。每个项目限制为 100 条条目。大型粘贴内容会被哈希并单独存储，以保持日志精简。  
**成本：零**

### 2. 会话记忆（Session Memory）

这是一个结构化的 Markdown 记忆，由后台代理在你的对话增长时构建和更新。它的最终消费者不是用户，而是一种压缩策略。目前，当对话增长到上下文窗口的约 85% 时，你的对话会被总结。使用会话记忆，这将实时进行。然而，这被隐藏在 `tengu_session_memory` 后面，这是许多神秘的 Claude tengu 标志之一。我不确定这是已发布的功能还是未来的功能。但它确实存在于代码中。

### 3. CLAUDE.md

你一定很熟悉这个。这是人类控制的编码风格、偏好、需要记住的事情等。当你与 Claude 开始会话时，你的 CLAUDE.md 文件的内容会立即被添加。当 Claude 遍历分层文件夹结构以发现它们时，可以加载多个 CLAUDE.md 文件。

### 4. 自动记忆（Auto-memory）

这就是我们谈论 Claude Code 记忆时所指的。这是 Claude Code 从多个对话中了解用户的地方，比如他们的角色、偏好、项目上下文和指向外部系统的指针。这些存储在 `~/.claude/projects/<slug>/memory/` 中。

### 5. 团队记忆（Team Memory）

很无聊。就不深入讨论了。

本文的重点是**自动记忆**。这些几乎是 Claude 为你保留的所有记忆模块，都是本地的且对用户可见。

为了更好地理解，我将与写入记忆相关的内容和与检索 + 使用记忆相关的内容分开。

## 记忆管理

Claude 的自动记忆不是草率的或仅追加的，它管理得相当好。它经历三个管理阶段。

### 写入阶段

一个后台代理会检查最后 N 条消息，决定什么值得记住，然后决定是添加新的记忆文件还是更新现有的文件。

它在开始工作之前会接收所有记忆，这是为了确保它更新现有的记忆而不是创建新的重复文件，并探索本地记忆。

写入过程也很严格，当为记忆创建新主题文件时，它们的引用/索引会记录在 MEMORY.md 中。

```markdown
MEMORY.md（索引）：
  - [用户简介](user_profile.md) — 后端工程师，5 年 Python 经验，刚接触这个仓库的 React 前端
  - [测试策略](feedback_testing.md) — 集成测试中永远不要模拟数据库
  - [认证重写](project_auth_rewrite.md) — 由合规性驱动，而非技术债务，截止日期 2026-04-15
  - [Bug 跟踪器](reference_linear.md) — 管道 bug 在 Linear 项目 INGEST 中跟踪

feedback_testing.md（一个记忆文件）：
  ---
  name: Testing policy
  description: Integration tests must use real database connections, never mocks
  type: feedback
  ---
```

如果你有多个本地上下文文件，这是一个很好的方法，保持目录 + 一行摘要，这将帮助代理稍后找到正确的记忆文件。

### 四种记忆类型

`user` | `feedback` | `project` | `reference`

这些将帮助代理稍后检索正确的记忆，本质上是选择代理的标签。

- **user**：你是谁（角色、专业知识、偏好）
- **feedback**：你希望 Claude 如何工作（更正和确认的方法）
- **project**：代码/Git 无法告诉你的正在发生的事情（截止日期、决策、动机）
- **reference**：在仓库外哪里查找（Linear 看板、Slack 频道、仪表板）

### "所有这些后台工作不会拖慢速度吗？"

不会。提取代理作为"分支子代理"运行，共享父对话的提示缓存。系统提示前缀在父代理和分支之间，所以大多数输入令牌都是缓存命中的。用户永远不会等待记忆操作。

分支也是沙盒化的。它可以读取文件系统上的任何内容，但只能写入记忆路径（`isAutoMemPath()`）。

MCP 工具、Agent 工具和具有写入能力的 Bash 都被拒绝。

它被限制为 5 轮，以防止陷入死胡同，提示明确告诉它："不要浪费轮次尝试验证内容。不要 grep 源文件，不要阅读代码。"

还有一个去重机制。如果你告诉 Claude"记住我们使用 bun，而不是 npm"，而 Claude 在对话期间直接写入记忆，后台提取代理会检测到（`hasMemoryWritesSince()`）并完全跳过。

主代理和后台代理在每个轮次窗口中是互斥的。它们永远不会重复写入。

### AutoDream

这是由一个名为 `autoDream` 的功能处理的。它是一个后台进程，当积累足够的时间和会话时触发（默认值：自上次合并以来 24 小时和 5 个会话）

触发时，它会对整个记忆目录运行四遍：

1. **第一遍**：读取 `MEMORY.md` 并浏览现有主题文件
2. **第二遍**：从日常日志和会话记录中收集最近的信号（通过窄 grep，从不完整读取）
3. **第三遍**：继续将新信息合并到现有文件中，将相对日期转换为绝对日期，并删除与当前代码库矛盾的事实
4. **第四遍**：通过删除过时的指针、缩短冗长的条目以及解决文件之间的矛盾来修剪记忆。合并提示很明确："如果两个文件有分歧，修复错误的那个。"

AutoDream 本身被隐藏在一个标志后面（`tengu_onyx_plover`）。

它作为分支子代理运行，具有通用的读/写工具和防止跨会话并发合并的锁文件。

没有过期或计划清理。记忆被删除的唯一方式是通过代理决定它是否不再与代码库相关、与其他记忆矛盾等。记忆不断被完善，但除非由 Claude Code 代理完成，否则永远不会静默删除。

## 记忆检索

Claude 如何决定加载哪些记忆，以及它如何对待它找到的内容？

显然，Claude 不会将所有记忆加载到上下文中；这将违背复杂记忆设计的整个目的。

`MEMORY.md`（索引，限制为 200 行/25KB）总是被加载到系统提示中，但单个记忆文件不会。

系统使用 Sonnet（即使你的主模型是 Opus）作为更快的相关性过滤器。当你发出请求时，在主模型开始思考之前，一个非阻塞进程：

1. 扫描所有记忆文件的 frontmatter（最多 200 个文件，按最新优先排序）
2. 格式化清单：`[type] filename (timestamp): description`
3. 将此清单 + 你的查询发送给 Sonnet
4. Sonnet 返回最相关的 5 个文件名
5. 只有这 5 个文件被加载到上下文中

这就是为什么 frontmatter 中的 `description` 字段如此重要，因为它是 Sonnet 在决定相关性时看到的唯一内容。

选择器的提示是："如果你不确定记忆是否有用，不要包含它。"这将消除误报，只允许最相关的记忆被获取。

它还过滤掉 Claude 已经使用的工具的文档，但选择关于这些工具的警告和注意事项。

先前轮次中已经显示的文件也会被排除，以保持 5 个槽位的预算用于更重要的新记忆。

超过一天的记忆会在其内容旁边注入一个明确的警告："此记忆已有 47 天历史。记忆是时间点观察，而非实时状态。关于代码行为或 file:line 引用的声明可能已过时。在作为事实推荐之前，请针对当前代码进行验证。"

系统提示还通过题为"在根据记忆推荐之前"的部分来加强这一点：如果记忆命名了文件路径，检查文件是否存在；如果它命名了函数，grep 它；如果用户即将根据推荐采取行动，先验证。

## 安全细节

这是一个有趣的安全细节。`autoMemoryDirectory` 设置允许你自定义记忆存储的位置。但它只能在全局 claude 设置 `~/.claude/settings.json` 中设置，而不能在项目级设置 `.claude/settings.json` 中设置。

原因是，如果恶意仓库将此路径设置为包含敏感文件的目录（如 `~/.ssh`），它将授予对这些文件的静默写入访问权限。路径还会针对遍历攻击（`..` 段）、根路径和空字节进行验证。

提取代理的沙盒增加了另一层。即使发生奇怪的事情，它也只能写入通过 `isAutoMemPath()` 的路径。其他一切都被拒绝。

## 总结

有趣的是，即使是管理像 Opus 4.6 这样强大模型的记忆，也有严格的限制和边界设置。

模型不会被留下来决定如何格式化记忆或检索它。

格式被强制执行（YAML frontmatter、单行索引条目、四种固定类型）。检索由单独的、更便宜的模型（Sonnet）处理，主模型甚至无法控制它。

删除没有自动触发器。过时警告由 harness 注入以强调其日期。沙盒限制提取代理可以写入的位置。

甚至不保存什么的决定都被硬编码到提示中。

模型很强大，但 harness 不信任它在自己不受监督的情况下管理自己的记忆。每一步都有约束。
