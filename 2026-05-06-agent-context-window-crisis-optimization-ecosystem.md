## 二、上下文危机：Agent 为什么被上下文"噎死"

### 2.1 上下文消耗的三重绞肉机

一个典型的 AI Agent 在执行长程任务时，上下文窗口被三个方向同时消耗：

```
┌─────────────────────────────────────────┐
│          上下文窗口消耗模型                │
├─────────────────────────────────────────┤
│                                         │
│  [输入侧] 工具返回结果                   │
│    · Playwright 截图: ~56KB/次          │
│    · 20个 GitHub Issue: ~59KB           │
│    · 一条访问日志: ~45KB                │
│    · 大型 JSON 响应: 10-200KB           │
│                                         │
│  [历史侧] 对话历史记录                    │
│    · 每轮对话: 系统提示 + 用户输入 +     │
│      模型输出 ≈ 2-5K tokens             │
│    · 30轮后: ~100K tokens               │
│                                         │
│  [输出侧] 模型废话                        │
│    · 礼貌用语、重复解释、冗余描述          │
│    · 占总输出 tokens 的 30-50%          │
│                                         │
└─────────────────────────────────────────┘
```

context-mode 项目首页的数据令人触目惊心：**一个 Agent 运行 30 分钟后，40% 的上下文已被消耗殆尽**。当模型触发上下文压缩（compaction）时，它会忘记正在编辑的文件、进行中的任务、以及用户最后的要求——这正是 Agent 在长程任务中"突然变蠢"的根本原因。

### 2.2 上下文 vs 推理质量的非线性关系

很多人有一个误解：更大的上下文窗口 = 更强的推理能力。但真实情况更复杂。

DeepSeek-V4 论文中的 MRCR（Multi-Range Context Retrieval）基准测试揭示了一个关键发现：

| 上下文长度 | 8-needle 检索准确率 |
|-----------|-------------------|
| 32K       | 0.96              |
| 64K       | 0.93              |
| 128K      | 0.89              |
| 256K      | 0.82              |
| 512K      | 0.71              |
| 1M        | 0.59              |

从 32K 到 1M token，检索准确率从 96% 暴跌到 59%。**上下文越长，"大海捞针"的针就越难找到**。这意味着单纯增加上下文窗口尺寸并不能解决 Agent 的长程推理问题——你还需要让模型在长上下文中保持"注意力"的精度。

### 2.3 KV Cache：隐藏的性能杀手

对于 Agent 开发者来说，KV Cache 是一个比上下文窗口更隐蔽但同样致命的瓶颈。

在传统的 Grouped Query Attention (GQA) 架构中，每个 token 的 KV Cache 以 BF16 格式存储。当上下文达到 100K token 时：
- 一个 70B 参数模型的 KV Cache 需要约 **160GB 显存**
- 这意味着即使用 8 张 H100（每张 80GB），也**无法容纳 100K 上下文的 70B 模型**

这就是为什么许多 Agent 框架在长程任务中突然变慢甚至崩溃——KV Cache 爆了，GPU 开始疯狂 swap，推理延迟从 50ms/token 飙升到 500ms/token。

## 三、模型层革命：DeepSeek-V4 的混合注意力架构

### 3.1 从"更大的窗口"到"更聪明的注意力"

DeepSeek-V4 的架构创新不在于把上下文窗口做大（1M token 窗口本身并不稀奇），而在于它**让 1M 上下文的推理成本降低到了 2025 年主流模型的 1/10 以下**。

具体数据：

| 指标 | DeepSeek-V3.2 | DeepSeek-V4-Pro | DeepSeek-V4-Flash |
|------|--------------|-----------------|-------------------|
| 单 token 推理 FLOPs（1M 上下文） | 基准 | 27% | 10% |
| KV Cache 内存占用 | 基准 | 10% | 7% |
| KV Cache vs GQA-BF16 | 基准 | ~2% | ~2% |

**V4-Flash 在 1M 上下文下的推理成本仅为 V3.2 的 10%，KV Cache 仅为 7%。** 这意味着同样的硬件可以处理 10 倍的上下文深度。

### 3.2 CSA + HCA：双层注意力压缩机制

DeepSeek-V4 的核心创新是**混合注意力机制**：将 KV 条目通过两种不同的压缩策略处理，并在网络层交替部署。

```
输入序列 → [压缩器] → 压缩 KV → [注意力计算]
                 │
    ┌────────────┴────────────┐
    │                         │
   CSA (4x压缩)            HCA (128x压缩)
    │                         │
   FP4 Lightning            密集注意力
   Indexer 选Top-k           在压缩序列上
    │                         │
   稀疏注意力 +              全连接注意力
   滑动窗口分支              + 滑动窗口分支
```

#### Compressed Sparse Attention (CSA)

CSA 沿序列维度将 KV 条目压缩 4 倍，然后通过一个 **Lightning Indexer**（FP4 精度，ReLU 评分的多头点积）为每个查询选择 Top-K 个压缩块。

关键技术细节：
- 压缩器使用 softmax-gated pooling + 学习到的位置偏置
- Lightning Indexer 的搜索空间比原始序列小 4 倍
- 保留一个滑动窗口分支来处理最近的未压缩 token

这相当于**在注意力计算前先做一次"粗筛"，只让最相关的 KV 条目参与精细计算**。

#### Heavily Compressed Attention (HCA)

HCA 走更激进的路线：将 KV 条目压缩 **128 倍**，然后直接做密集注意力。

因为压缩后的序列非常短（1M token → ~7.8K 个压缩块），密集注意力的计算成本完全可以接受。

#### 交替部署

在 V4-Pro 的 61 层网络中：
- 第 0-1 层：HCA（浅层做全局粗关注）
- 第 2-60 层：CSA 和 HCA 交替（深层做精细选择）
- 最后的 MTP 块：仅滑动窗口（输出聚焦局部）

**不同层携带不同的注意力模式，强制所有层使用同一种机制会浪费容量。** 这是一个反直觉但极其有效的架构决策。

### 3.3 Agent 专属后训练决策

DeepSeek-V4 在架构之外还做了三个针对 Agent 场景的后训练优化：

#### 跨工具调用 interleaved thinking

V3.2 的设计有一个致命缺陷：当用户在一个多轮 Agent 会话中发送 follow-up 消息时，模型会**丢弃已积累的全部推理内容**，被迫重新构建状态。

V4 改变了这一行为：
- **有工具调用**：推理内容跨用户消息边界保留，形成完整的累积思维链
- **无工具调用**（纯对话）：每轮刷新推理内容，保持上下文精简

这意味着对于一个 SWE-bench 任务或长时间终端操作，Agent 的"记忆"不会在用户插话时断裂。

#### XML 工具调用 schema

V4 引入了 `|DSML|` 专用 token 和 XML 格式的工具调用。相比 JSON-in-string 格式，XML 格式**显著降低了嵌套转义失败率**——这是 Agent 框架中最常见的工具调用错误来源之一。

schema 还区分了字符串参数（`string="true"` 原样传递）和结构化参数（`string="false"` 解析为 JSON），消除了 JSON 工具调用中对数字和布尔值的解析错误。

#### DSec：为 RL 训练构建的沙箱基础设施

Agent 行为是通过 RL 在真实工具环境中训练出来的。DeepSeek 构建了 **DSec（DeepSeek Elastic Compute）**——一个 Rust 平台，在统一 Python SDK 下暴露四种执行基质：
1. 函数调用
2. 容器
3. microVM（Firecracker）
4. 完整 VM（QEMU）

单个集群可运行**数十万个并发沙箱**。三个关键特性：
- **分层 3FS 存储快速镜像加载**：RL rollout 不需要等待容器启动
- **抢占安全的轨迹回放**：中断的训练步骤可恢复，无需重跑工具调用
- **跨基质统一 API**：训练 harness 可以无缝在函数调用和完整 VM 间切换

这套基础设施直接支撑了 V4 在 Agent 基准测试中的优异表现。

## 四、工程层革命：上下文优化工具链的崛起

### 4.1 context-mode：上下文优化的"瑞士军刀"

context-mode 之所以在 GitHub 上迅速走红，不是因为它做了什么惊天动地的创新，而是因为它**系统性地解决了上下文消耗的全部四个维度**。

#### 四维上下文优化模型

```
                  ┌──────────────┐
                  │  上下文优化   │
                  └──────┬───────┘
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ 输入侧   │  │ 历史侧   │  │ 输出侧   │  │ 连续性   │
    │ 沙箱化   │  │ FTS5索引 │  │ 极简输出 │  │ 会话记忆 │
    │ 315KB    │  │ BM25检索 │  │ 65-75%   │  │ SQLite   │
    │ → 5.4KB │  │ 按需召回 │  │ 压缩率   │  │ 精准恢复 │
    └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

#### 核心创新 1：沙箱化工具

context-mode 的核心思想是**不要让原始数据进入上下文窗口**。

它提供了一组 MCP 沙箱工具（`ctx_execute`、`ctx_batch_execute`、`ctx_execute_file`），让 Agent **用代码来分析数据，而不是把数据读给模型看**。

一个典型案例：
```javascript
// 之前：47次 Read() = 700KB 上下文
// 之后：1次 ctx_execute() = 3.6KB 上下文

ctx_execute("javascript", `
  const files = fs.readdirSync('src').filter(f => f.endsWith('.ts'));
  files.forEach(f => console.log(f + ': ' + 
    fs.readFileSync('src/'+f,'utf8').split('\\n').length + ' lines'));
`);
```

**一个脚本替换 10 次工具调用，节省 100 倍上下文。** 这就是 context-mode 的核心理念：把 LLM 当作**代码生成器**，而不是**数据处理器**。

#### 核心创新 2：SQLite + FTS5 会话记忆

当上下文压缩触发时，大多数 Agent 框架的做法是简单截断对话——这就像把一个人的短期记忆强行删除一半。

context-mode 的做法：
1. 每次文件编辑、git 操作、任务状态、用户决策都记录到 SQLite
2. 使用 FTS5 全文搜索引擎索引所有事件
3. 上下文压缩后，通过 BM25 搜索**按需召回**相关上下文

模型不会"忘记"，只是把不需要的信息移到"硬盘"上，需要时再读回"内存"。

#### 核心创新 3：极简输出模式

context-mode 强制模型输出采用极简模式：
- 去掉冠词、填充词（just/really/basically）、礼貌用语
- 使用短句和片段式表达
- 模式：[对象] [动作] [原因]。[下一步]。
- 对安全警告和不可逆操作自动恢复完整描述

实测**输出 token 减少 65-75%，技术准确性完全保留**。

#### 平台覆盖

context-mode 已支持 14 个平台：
- Claude Code（插件市场，全自动）
- Gemini CLI
- VS Code Copilot
- JetBrains Copilot
- Cursor
- Windsurf
- Claude Desktop
- Cline
- 等等

这种跨平台兼容性意味着上下文优化正在成为一个**通用 Agent 基础设施层**。

### 4.2 上下文优化的三层架构

将 context-mode 与 DeepSeek-V4 放在一起看，可以发现 Agent 上下文优化正在形成三层架构：

```
┌─────────────────────────────────────────────────┐
│  L1: 模型层（注意力压缩）                         │
│  · CSA/HCA 混合注意力                            │
│  · FP4/FP8 KV Cache 量化                        │
│  · 128x 重度压缩                                 │
│  · 代表：DeepSeek-V4                             │
├─────────────────────────────────────────────────┤
│  L2: 框架层（会话管理）                           │
│  · FTS5 索引 + BM25 按需召回                     │
│  · 上下文压缩安全机制                            │
│  · 会话状态持久化                                │
│  · 代表：context-mode                            │
├─────────────────────────────────────────────────┤
│  L3: 应用层（工具沙箱化）                         │
│  · 代码执行替代数据读入                          │
│  · 输出极简模式                                  │
│  · 工具输出过滤                                  │
│  · 代表：context-mode sandbox tools             │
└─────────────────────────────────────────────────┘
```

**这三层叠加的效果是乘法的，不是加法的。** 如果每层能减少 70% 的上下文消耗，三层叠加后总压缩率可达 97.3%（0.3 × 0.3 × 0.3 = 0.027）。

## 五、架构层洞察：VAKRA 基准测试揭示的 Agent 失败模式

### 5.1 VAKRA 是什么

IBM Research 于 2026 年 4 月发布的 VAKRA 基准测试，是目前**最接近真实企业场景的 Agent 能力评估基准**。

与 SWE-bench、MMLU 等传统基准不同，VAKRA 的核心特点是：
- **可执行环境**：Agent 在真实工具环境中交互
- **8000+ API**：覆盖 62 个领域的本地托管 API，由真实数据库支撑
- **组合推理**：任务需要 3-7 步推理链，结合结构化 API 交互和非结构化检索
- **轨迹评估**：不仅评估最终答案，还评估完整的工具执行轨迹

### 5.2 VAKRA 四大能力维度

```
┌─────────────────────────────────────────────────────┐
│  VAKRA Benchmark 能力维度                             │
├────────────────┬────────────────────────────────────┤
│ Capability 1   │ API 链式调用                        │
│ 2,077 实例     │ 54 个领域，1-12 步工具调用链          │
│                │ SLOT-BIRD / SEL-BIRD 工具集          │
├────────────────┼────────────────────────────────────┤
│ Capability 2   │ 工具选择                            │
│ 1,597 实例     │ 17 个领域，每域 6-328 个工具          │
│                │ 平均 116 个工具/域                    │
├────────────────┼────────────────────────────────────┤
│ Capability 3   │ 多跳推理                            │
│ 869 实例       │ 38 个领域，1-5 个逻辑跳              │
│                │ 纯 API 多跳                          │
├────────────────┼────────────────────────────────────┤
│ Capability 4   │ 多跳 + 多源 + 策略遵从               │
│ 644 实例       │ 41 个领域                           │
│                │ API + RAG 混合跳                     │
│                │ 多轮对话 + 工具使用策略              │
└────────────────┴────────────────────────────────────┘
```

### 5.3 令人震惊的失败率

VAKRA leaderboard 揭示了一个行业不愿面对的事实：**即使是最强的 Agent 模型，在 VAKRA 上的得分也远低于人们的期望**。

IBM 官方博客中的原始措辞是：*"models perform poorly on VAKRA"*（模型在 VAKRA 上表现很差）。

这不是某个模型的个别问题，而是 Agent 架构的系统性缺陷。让我们从 VAKRA 的错误分析中提取核心失败模式：

#### 失败模式 1：工具调用链断裂（Capability 1）

在需要 5 步以上 API 链式调用的任务中，大部分模型在**中间步骤**出错：
- 错误传递中间结果的数据标签
- 忘记调用初始化 API（`get_data`）
- 在多步过滤中丢失上下文引用

**根因**：每增加一步，前序步骤的中间结果在上下文中的"位置"就变得更模糊。这不是模型推理能力的问题，而是**上下文中的中间状态管理**问题。

#### 失败模式 2：工具选择迷失（Capability 2）

每个领域平均有 116 个工具，但 OpenAI API 规范限制工具列表最大长度为 128 个工具。当工具数量接近上限时：
- 模型倾向于选择"看起来相关"但不精确的工具
- 工具描述的 token 消耗挤占了推理空间
- 短列表（shortlisting）机制本身的引入又增加了额外的不确定性

**根因**：工具列表本身就是上下文消耗大户。116 个工具的描述可能需要 20-50K tokens——这还没开始推理。

#### 失败模式 3：多跳推理中的注意力分散（Capability 3）

VAKRA 发现，当推理跳数从 1 增加到 5 时，模型准确率不是线性下降，而是**指数级衰减**：

```
跳数:  1      2      3      4      5
      ▲      ▲      ▲      ▲      ▲
      │      │      │      │      │
   ██████  ████   ██     █      ▏
   ~80%    ~55%   ~35%   ~18%   ~8%
```

**根因**：每多一跳，模型需要在上下文中同时维护"当前步骤"和"所有前序步骤的结果"。随着跳数增加，注意力的有效浓度被稀释。

#### 失败模式 4：多源融合中的来源混淆（Capability 4）

当查询需要从 API 和文档检索两个来源交替获取信息时（例如 API → RAG → API 的三跳混合推理）：
- 模型经常混淆信息来源（把文档中的信息当作 API 返回的结果）
- 在工具使用策略约束下（"某类查询只能用文档检索"），模型违反策略的比例超过 40%

**根因**：多源推理要求模型在上下文中维护"信息溯源"能力——知道每个信息片段来自哪里。当前的 Agent 架构缺乏这种原生的信息溯源机制。

### 5.4 VAKRA 与上下文危机的关联

VAKRA 揭示的失败模式与上下文危机之间存在深刻的关联：

| VAKRA 失败模式 | 上下文维度的根因 |
|---------------|----------------|
| 工具链断裂 | 中间结果在长上下文中被稀释 |
| 工具选择迷失 | 工具描述挤占推理空间 |
| 多跳注意力分散 | 前序步骤结果占用上下文，有效注意力降低 |
| 多源来源混淆 | 不同来源信息在上下文中缺乏结构化标记 |

**VAKRA 本质上是在测量"上下文质量"，而不仅仅是"模型能力"。** 当一个 Agent 需要维护 5 步推理链时，它真正需要的是一个能精确管理 5 个中间状态的上下文系统，而不是一个"更聪明"的模型。

## 七、实战建议：如何构建上下文友好的 Agent 架构

基于以上分析，以下是构建上下文友好 Agent 系统的实践建议：

### 7.1 架构原则

**原则 1：上下文是稀缺资源，按优先级分配**
- 系统提示词：固定保留，不压缩
- 当前任务上下文：高优先级，尽量保留
- 历史工具调用结果：中优先级，可摘要化
- 用户对话历史：低优先级，可索引化后按需召回

**原则 2：让模型写代码，而不是读数据**
- 用 `ctx_execute` 替代多次 `Read()`
- 用脚本分析日志，而不是把日志读进上下文
- 用聚合查询替代批量数据拉取

**原则 3：结构化中间状态**
- 每次工具调用的结果不仅存入上下文，也写入结构化存储
- 用明确的标签标记每个中间结果的来源和用途
- 上下文压缩时，保留结构化摘要而非原始数据

**原则 4：输出极简但安全**
- 正常输出使用极简模式（省 65-75% token）
- 安全警告、不可逆操作自动切换为完整模式
- 用户明确表示困惑时自动扩展解释

### 7.2 工具链选型参考

| 场景 | 推荐方案 | 预期效果 |
|------|---------|---------|
| 编码 Agent | context-mode + Claude Code | 98% 上下文节省 |
| 长程研究 | DeepSeek-V4-Pro + interleaved thinking | 1M 上下文，推理不中断 |
| 企业 API 集成 | VAKRA 式沙箱 + get_data 模式 | 大数据不进入上下文 |
| 多 Agent 协作 | ruflo Agent Swarm | 上下文压力分散到多节点 |
| 长时间运行任务 | cocoindex 增量引擎 | 状态持久化 + 断点续跑 |

### 7.3 性能基线建议

对于一个生产级 Agent 系统，建议监控以下指标：

- **上下文使用率**：单任务消耗 / 总窗口大小，目标 < 60%
- **上下文压缩触发率**：压缩次数 / 总对话轮数，目标 < 10%
- **中间状态丢失率**：压缩后需重新获取的信息量 / 总中间状态，目标 < 5%
- **工具调用效率**：有效工具调用 / 总工具调用，目标 > 80%
- **输出冗余率**：废话 token / 总输出 token，目标 < 20%

## 八、结语：上下文不是容量问题，是架构问题

2026 年的 Agent 开发者终于开始面对一个被忽略了太久的真相：

**上下文窗口从来不是一个容量问题。它是一个架构问题。**

更大的窗口只是给了 Agent 更多的"纸"，但如果没有好的"笔记方法"，更多的纸只会让 Agent 在更多的纸中找到更少的有用信息。

这三者的交汇点指向了一个清晰的方向：**下一代 Agent 基础设施的核心竞争力不是模型参数规模，而是上下文管理能力。**
