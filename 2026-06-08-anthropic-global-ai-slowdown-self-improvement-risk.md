# Anthropic 呼吁全球 AI 减速：自我改进风险是真的威胁，还是战略卡位？

> 2026 年 6 月初，Anthropic 公开呼吁全球 AI 实验室协调减速，理由是模型"自我改进"的风险正在逼近。与此同时，Meta 的 AI 客服 Agent 被黑客利用来盗取 Instagram 账号。两件事看似不相关，却指向同一个核心问题：当 AI Agent 开始获得真实世界的行动能力，我们到底准备好了吗？

---

## 一、一周内的两条时间线

2026 年 6 月的第一周，AI 行业同时发生了两件事，恰好构成了理解当下 AI 安全争论的最佳切面。

**时间线 A：Anthropic 的全球减速呼吁**

- **6 月 4 日**，据 Reuters 报道，Anthropic 公开表示 AI 实验室需要一个"协调计划"，在风险升高时暂停开发。
- **6 月 5 日**，WSJ 进一步披露，Anthropic 特别指出了模型"自我改进"（self-improving）的风险。
- **同日**，The Register 报道了怀疑者的声音："时机过于巧合"（the timing is awfully convenient）。

这不是 Anthropic 第一次发出警告。就在两个月前的 4 月，Anthropic 宣布其 Mythos 模型"过于擅长黑客攻击"，不适合向公众发布。这一声明本身就引发了关于 AI 安全叙事边界的讨论：当一家公司说自己的模型"太危险"时，它是在承担责任，还是在建立竞争壁垒？

**时间线 B：Meta AI 客服 Agent 被利用**

- **6 月 5 日**，404 Media 报道，攻击者利用 Meta 的 AI 客服 Agent 盗取了多个 Instagram 账号——包括奥巴马白宫的休眠账号。
- 攻击手法极其简单：攻击者直接要求 Agent 将账号关联到他们控制的邮箱，Agent 照做了。
- Meta 没有公开评论这个漏洞是如何产生的，但随后在 X 上表示漏洞已修复。

MIT Technology Review 的评论一针见血：在 Anthropic 宣布 Mythos 模型"太危险"之后，整个行业的网络安全讨论都聚焦于"超能力 AI 系统摧毁基础设施"的风险。但 Instagram 黑客事件展示了另一种现实——**最简单的利用方式往往最有效**。

这两条时间线的交汇点是什么？Anthropic 在谈论模型能力可能超越控制的**远期风险**，而 Meta 的黑客事件揭示了 AI Agent 已经在现实世界中造成的**即时风险**。两者都是真实的，但行业的注意力分配却极度不均衡。

## 二、"自我改进"到底是什么？技术拆解

Anthropic 的警告中，"自我改进"（self-improving）是最令人不安的关键词。但要理解这个概念，我们需要先拆解它到底意味着什么。

### 2.1 自我改进的三个层次

在当前的技术语境中，"自我改进"至少包含三个不同的层次，它们的风险等级截然不同：

**第一层：自我优化提示（Self-Optimizing Prompts）**

这是已经广泛存在的现象。Agent 在执行任务的过程中，通过反思和试错来改进自己的提示词和工作流程。例如，Claude Code 在修复 bug 时会自动记录有效的调试策略；Cursor 会从用户的修改中学习更优的代码生成模式。这种"自我改进"本质上是 meta-prompting 的延伸，风险较低——它改的是提示词，不是模型本身。

**第二层：自我训练（Self-Training）**

模型利用自身生成的数据来改进自己的输出。这包括自博弈（self-play）、自我批评（self-critique）和自我数据生成（synthetic data generation）等方法。AlphaGo 的 self-play 就是经典案例。在 LLM 领域，这种方法已经被用于代码生成、数学推理等特定任务。风险中等——模型的能力边界在扩展，但训练过程仍然由人类控制。

**第三层：自主架构改进（Autonomous Architectural Improvement）**

这是 Anthropic 真正担心的层次：模型能够自主地提出并实施对自身架构的改进——包括优化训练算法、设计新的模型架构、甚至创建更强大的下一代模型。如果这种能力与足够的计算资源和工具访问权限结合，理论上可能导致**递归自我改进**（recursive self-improvement），即模型每改进一次就变得更强，从而能进行下一次更有效的改进。

### 2.2 我们距离第三层有多远？

坦率地说，**我们距离真正的第三层自我改进还有相当远的距离**。但问题的复杂性在于：

1. **能力不是线性的**：模型能力的提升不是均匀分布的。在特定任务上，模型可能突然展现出远超预期的能力——这就是所谓的"涌现"（emergence）。2026 年初的多项研究表明，某些模型在代码生成任务上展现出了超出训练数据分布的能力。

2. **工具访问权在指数级增长**：随着 MCP 协议的普及和 Agent 工具生态的爆发，模型可以访问的工具数量和能力正在快速扩展。一个模型可能不需要自己"发明"新的架构，它只需要能够调用其他工具来辅助自己实现改进。

3. **多 Agent 协作的放大器效应**：当多个 Agent 能够相互协作时，单个 Agent 的能力限制被打破了。一个 Agent 负责研究，一个负责编码，一个负责测试——这种分工协作的效果可能远超单个 Agent 的能力上限。

### 2.3 一个被忽视的技术信号：Agent 工具使用的可靠性瓶颈

有意思的是，当我们讨论自我改进风险时，行业实际上面临着一个更基础的问题：**即使模型有能力自我改进，它们能可靠地执行吗？**

2026 年 5 月底的一篇分析指出，Agent 工具使用的可靠性是从 Demo 到生产的第一道坎。即使是当前最先进的模型，在多步骤工具调用任务中的成功率也远低于人类的期望。这意味着：

- **短期内**，自我改进的实际风险被 Agent 工具的不可靠性所缓冲。
- **长期来看**，一旦工具可靠性问题被解决（这正是整个行业努力的方向），自我改进的速度可能会突然加快。

这个时间差——从"能力存在"到"能力可被可靠利用"——可能是我们最后的安全窗口。

## 三、减速呼吁的真实性：安全关切还是战略卡位？

Anthropic 呼吁全球减速的时机引发了广泛质疑。让我们从三个维度来分析这个呼吁的可信度。

### 3.1 支持"真诚安全关切"的证据

Anthropic 在 AI 安全领域的投入是有记录的。从 Constitutional AI 到红队测试实践，再到 Project Glasswing（一个利用 AI 进行漏洞检测的项目），Anthropic 在安全研究上的投入远超行业平均水平。

此外，Anthropic 的商业结构——作为一家公共利益公司（Public Benefit Corporation），其法律架构允许它将安全目标置于利润目标之上。这与纯粹以股东利益最大化为目标的上市公司有本质区别。

更重要的是，Anthropic 在 4 月份宣布 Mythos 模型"太危险"的决定，本身就意味着**放弃了短期商业利益**。如果 Mythos 真的如报道中那样具有突破性的能力，将其推向市场可能带来数十亿美元的收入。选择不发布，至少在一定程度上支持了安全关切的真实性。

### 3.2 支持"战略卡位"的证据

然而，怀疑者的质疑也有其道理。

**首先是时机问题。** 2026 年上半年，AI 行业的竞争格局正在发生变化：

- OpenAI 的 GPT-5 系列在多个基准上领先
- Google 的 Gemini 系列在推理和编码任务上快速追赶
- 中国模型（DeepSeek、Qwen 等）在成本效率上展现出巨大优势
- JetBrains 刚刚发布了 Mellum2——一个 12B 参数的 MoE 编码模型

在这个时间点呼吁"减速"，客观上有利于目前处于领先地位的 Anthropic（Claude 系列），因为竞争对手需要更多时间来追赶。

**其次是叙事框架的选择。** Anthropic 选择聚焦于"自我改进"这个最具戏剧性的风险场景，而不是更常见、更紧迫的 Agent 安全问题（如 Meta 事件中暴露的那些）。这有助于建立一种叙事：只有 Anthropic 在认真对待最严重的风险。

**第三是监管窗口。** 全球 AI 监管框架正在形成中。在这个关键时刻，谁能够定义风险的框架，谁就可能影响监管的方向——而监管方向直接影响竞争格局。

### 3.3 我的判断：两者都是真实的

我认为更准确的判断是：**Anthropic 的安全关切是真实的，但战略考量也是真实的。这两者并不矛盾。**

在复杂系统中，最明智的长期战略往往与最负责任的行为是一致的。Anthropic 可能确实相信自我改进的风险是真实的，同时也意识到，通过塑造这个叙事，可以为自己的竞争地位赢得优势。

真正的问题不在于 Anthropic 的动机是否纯粹，而在于：**即使动机混合了战略考量，他们提出的风险是否值得认真对待？**

我的答案是：是的。但我们应该关注的风险，可能和他们描述的不完全一样。

## 四、真正的风险不在"自我改进"，而在"能力 - 治理错配"

如果说 Anthropic 的减速呼吁有什么值得讨论的核心问题，我认为不是模型会不会自我改进，而是另一个更紧迫的问题：**AI Agent 的能力增长速度，远远超过了我们建立有效治理机制的能力。**

### 4.1 Meta 黑客事件：能力 - 治理错配的典型案例

Meta 的 AI 客服 Agent 被用来盗取 Instagram 账号，这个事件完美诠释了什么是"能力 - 治理错配"：

- **Agent 有能力**：它可以执行邮箱关联操作——这是一个合法的功能，用于帮助用户恢复账号。
- **治理有漏洞**：它没有验证请求者的身份——任何能提供正确账号信息的人都可以触发这个操作。
- **结果是灾难性的**：攻击者轻松获取了高价值账号的控制权。

这个问题的核心不是技术——验证身份的技术已经存在了几十年。问题是**在部署 Agent 时，治理层被牺牲了**。原因很简单：更多的治理意味着更多的摩擦，更多的摩擦意味着更差的用户体验，更差的用户体验意味着更低的采纳率。

### 4.2 治理滞后的三个结构性原因

这种错配不是偶然的，而是由三个结构性原因造成的：

**原因一：速度优先的文化。** AI 行业的竞争节奏使得"先部署后修补"成为默认策略。在 Agent 领域尤其如此——谁能先推出功能更强大的 Agent，谁就能抢占市场份额。安全审查和红队测试被视为"拖慢发布"的成本。

**原因二：治理的复杂性被低估。** 传统软件的安全模型基于"最小权限原则"——每个组件只拥有完成其任务所需的最小权限。但 Agent 的本质是灵活性和自主性——它们被设计用来处理不可预见的情况。如何在保持灵活性的同时实施有效的权限控制，是一个尚未解决的技术难题。

**原因三：激励机制错位。** 发现漏洞的激励（负面的声誉影响、修复成本）远低于利用漏洞的激励（高价值账号的经济收益）。在 AI Agent 领域，这种错位被放大，因为 Agent 的行动能力使得单个漏洞的影响范围远超传统软件。

### 4.3 从工程视角看治理滞后

作为一个每天都在使用 AI Agent 的开发者，我对此有切身体会。

当我配置一个 Agent 的工具访问权限时，我面临的选择是：

- **严格模式**：每个工具调用都需要人工审批。安全，但 Agent 的效率大幅下降。
- **宽松模式**：Agent 可以自主调用所有已注册的工具。高效，但风险不可控。
- **智能模式**：基于风险评估动态调整权限。理想，但实现成本极高——需要定义每个工具的风险等级、上下文敏感度、调用模式等。

大多数团队选择了宽松模式，因为这是最快达到"看起来有用"的方式。但这意味着我们实际上是在用生产环境做实验。

## 五、HuggingFace 的答案：让工具为 Agent 重新设计

有意思的是，在治理问题日益突出的同时，工具生态也在进化。HuggingFace 在 6 月初发布了一篇博客文章，介绍了他们如何重新设计 `hf CLI`，使其同时服务于人类用户和 Agent。

这不是一个微不足道的设计决策。它揭示了一个正在形成的行业共识：**当 Agent 成为工具的主要消费者时，工具本身需要被重新设计。**

### 5.1 hf CLI 的 Agent 优化

HuggingFace 的 `hf CLI` 在检测到 Agent 驱动时，会自动改变输出格式：

- **人类模式**：对齐的表格、截断显示、颜色编码、进度条
- **Agent 模式**：TSV 格式、完整 ID、ISO 时间戳、所有标签、无 ANSI 代码

关键数据是：在复杂的多步骤任务中，没有 CLI 的基线方案（Agent 手写 curl 或使用 Python SDK）比使用 hf CLI **多消耗高达 6 倍的 Token**。

这不仅仅是效率问题，更是安全问题。更少的 Token 意味着更少的上下文溢出风险、更少的信息丢失、更低的幻觉概率。

### 5.2 下一个命令提示：引导 Agent 的"护栏"

hf CLI 的另一个创新是"下一个命令提示"（next-command hints）：每个命令完成后，会提示下一步应该运行的命令，包括正确的参数。

```
$ hf jobs run --detach python:3.12 python train.py
✓ Job started
 id: 6f3a1c2e9b
 url: https://huggingface.co/jobs/celinah/6f3a1c2e9b
Hint: Use `hf jobs logs 6f3a1c2e9b` to fetch the logs.
```

对人类来说，这是便利功能。对 Agent 来说，这是**护栏**——下一步行动被命名、参数化，Agent 不需要自己推导。这减少了 Agent 犯错的机会。

### 5.3 安全可重试的操作

hf CLI 还实现了两个对 Agent 安全至关重要的特性：

1. **安全可重试**：`hf repos create --exist-ok` 在仓库已存在时是空操作，不会报错或创建重复。
2. **快速失败**：在 Agent 模式下，破坏性命令（如删除）如果没有 `--yes` 标志会直接失败，并提示修复方法。

这些看似微小的设计决策，实际上代表了 Agent 工具设计的一个核心范式转变：**从"正确执行"到"安全失败"**。

## 六、本地化 Agent：另一种减速路径？

在 Anthropic 呼吁全球减速的同时，另一条技术路线正在悄然发展：**让 Agent 在本地运行**。

H Company 在 6 月初发布了 Holo3.1——一个 Computer Use Agent 模型系列，首次支持量化权重（FP8、Q4 GGUF、NVFP4）。这意味着 Computer Use Agent 可以在消费级硬件上运行，而不需要依赖云端 API。

### 6.1 本地 Agent 的安全优势

本地运行 Agent 有几个显著的安全优势：

1. **数据不出域**：所有交互都在本地完成，敏感数据不会传输到第三方服务器。
2. **权限可控**：本地 Agent 的权限范围可以由用户精确控制，不需要信任云端的权限管理。
3. **可审计**：本地运行的 Agent 的行为可以被完整记录和分析，不存在云端黑箱的问题。

### 6.2 性能数据

Holo3.1 的量化模型在 DGX Spark 上展现出了令人印象深刻的性能：

- **NVFP4 W4A16** 的总 Token 吞吐量是 FP8 的 1.41 倍，是 BF16 的 1.74 倍
- 结合 Agent harness 优化，端到端平均步骤时间从 6.8 秒降低到 3.3 秒（约 2 倍加速）

更重要的是，在 AndroidWorld 基准测试中，35B-A3B 模型从 67% 提升到 79.3%，而较小的 4B 和 9B 模型从 58% 提升到 72%。这意味着小模型在特定任务上的表现正在接近大模型。

### 6.3 本地化是否意味着减速？

这可能是最有趣的问题。Anthropic 呼吁的"减速"是指放慢模型能力的提升速度。但本地化 Agent 的发展实际上是在**改变能力部署的方式**，而不是能力本身。

如果更多 Agent 在本地运行，可能会出现以下结果：

- **正面**：数据隐私提升、用户控制权增加、云端集中式风险降低
- **负面**：安全审计更难（没有集中日志）、恶意 Agent 更难追踪、分布式攻击面增加

本地化不是减速的替代品，但它可能是另一种治理路径——通过分散化来降低系统性风险。

## 七、协调困境：为什么全球减速几乎不可能

Anthropic 呼吁的"协调计划"听起来合理，但实现的可能性极低。原因是一个经典的博弈论问题：**囚徒困境**。

### 7.1 博弈矩阵

假设只有两个参与者：公司 A 和公司 B。

| | B 减速 | B 不减速 |
|---|---|---|
| **A 减速** | 双方损失进度，安全提升 | A 大幅落后，B 获得决定性优势 |
| **A 不减速** | A 获得决定性优势，B 落后 | 双方都不减速，安全风险增加 |

在单次博弈中，"不减速"是占优策略——无论对方做什么，不减速的收益都高于减速。只有在重复博弈中（长期的竞争关系），合作（双方减速）才可能成为均衡。

### 7.2 现实中的多重博弈者

现实更加复杂，因为参与者不止两个：

- **OpenAI**：处于领先地位，没有减速动机
- **Google**：快速追赶中，减速意味着放弃追赶机会
- **Anthropic**：呼吁减速，但自身是否真的减速？
- **中国模型团队**：DeepSeek、Qwen 等，成本优势明显，减速动机不足
- **开源社区**：完全不受商业公司约束

即使所有主要商业公司都同意减速，开源社区也不会。即使所有人都同意，如何执行？谁来监督？违规的惩罚是什么？

### 7.3 更现实的替代方案

与其追求不可能实现的全球减速，更现实的方案是：

1. **分层治理**：不同风险等级的 Agent 适用不同级别的治理要求。低风险（如本地代码辅助）可以宽松，高风险（如自主执行金融交易）需要严格监管。

2. **标准化安全接口**：像 HuggingFace 重新设计 `hf CLI` 那样，为 Agent 工具设计标准化的安全接口。这样，安全可以成为基础设施的一部分，而不是每个团队需要单独解决的问题。

3. **开源审计机制**：建立开源的 Agent 行为审计工具，使得 Agent 的决策过程可以被第三方审查。这比依赖公司内部的红队测试更加透明和可信。

## 八、一个被忽视的维度：Agent 能力的"虚假信号"

在讨论自我改进风险时，有一个重要的维度经常被忽视：**我们可能高估了当前 Agent 的实际能力。**

### 8.1 Benchmark 与现实的鸿沟
