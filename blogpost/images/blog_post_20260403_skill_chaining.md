## 引言

大多数 Agent 系统将技能视为手动调用的静态提示。我们相信 Agent 技能应该更接近其人类原型的本质：**情境化的行为，动态地用于解决任务**。技能应该是 Agent **做**的事情，而不是 Agent **读**的东西。

在这篇产品博客中，我们讨论了当前 Agent 技能实现的局限性、我们的重新框架化，以及我们如何在 Slate 中实现技能。我们还将介绍导致我们当前设计的约束，以及如何在 Slate 中设置技能自动化。

首先，我们简要介绍一下 Slate 的工作原理。

---

## Slate 执行模型简要回顾

了解 Slate 的工作原理对后续内容非常有用。

Slate 通过**线程（threads）**采取**增量行动（incremental actions）**。

- **行动（Action）**：低范围、单一的步骤，朝向目标迈进。例如："启动开发服务器"、"审查文件 X 中的更改以用于 Y 原因"、"在 Agent 上点击目标路径"

- **线程（Thread）**：一个隔离的工作器，拥有自己的上下文窗口、范围化的能力和范围化的任务执行。它执行一个经过许可的、范围化的任务，并返回其行动的压缩表示。

> *小声说：对于编程语言理论（PLT）研究者来说，这些相对接近 Lisp 中的 continuation——线程可以部分执行，返回代表该部分执行的状态，并且可以在以后恢复。*

*要进一步了解 Slate 的工作原理以及我们如何处理情景记忆，请查看我们的[技术博客](https://randomlabs.ai/blog/slate)*

好了，进入正题。

## 背景

在人类中，技能是**习得的**。我们通常看到别人做我们想执行的任务，然后我们尝试复制这个行为足够多次，将其内化为情境条件的行为。然后我们继续精炼并重用它。

人类中的技能学习**可以**通过以下两种方式映射到 LLM：

1. **预训练期间的行为克隆**（对某些演示的 token 序列）
2. **后训练期间的 Agent RL**（教授模型有用的策略和战术）

这些行为然后在测试时被激发。

然而，如果没有 LLM 的持续学习，就没有太多途径来激发模型知道但不主动选择采用的行为。

就本篇博客而言，我们可以说 LLM 不自然采用的所有策略和战术都在**知识悬置（knowledge overhang）**之内。

> **知识悬置**：模型选择做什么与模型知道如何做之间的差距。技能提供了克服这一点的领域特定解决方案（类似于规则），通过直接注入文本来条件化模型采取它原本不会采取的行动，尽管它知道如何做。

这正是 Agent 技能（连同良好的指令调优）能够有意义地提高性能的地方，因为它们允许你引入指令，将那些分布外（out-of-distribution）的知识拉入分布内（in-distribution）的行为。这种现象依赖于**情境学习（in-context learning）**。[[1]](http://randomlabs.ai/blog/skill-chaining#ref-1)

## 规则 vs. 技能

有一个很明显的问题：**为什么不直接用规则来做这个？**

为了让大家理解一致：

- **规则**：你的 Agent Harness 强制加载到上下文中的文件
- **技能**：遵循**渐进披露**原则（Agent 被逐步给予关于技能的更多信息）

Anthropic 引入了[技能](https://claude.com/blog/skills)作为一种在需要时将行为修补到模型中的方法。

这样做的核心好处是，由于技能应该是特定于上下文的，主 Agent 的上下文不会立即被数万 token 的**并非总是有用**的条件指令淹没。当有效上下文长度仍远低于 200k token 时，前缀上下文是宝贵资源。

期望是模型根据需要激活和停用技能，动态地选择在不同情况下需要的分布外上下文。

**实际上发生的是，用户倾向于自己激活技能。**

你会手动为模型激活技能，除了激活之外没有任何明确的生命周期管理。更重要的是，人们将技能（可以说应该是持续学习的一种形式）视为终端 Agent 中的斜杠命令。

### 线程中固定的技能上下文

技能上下文在线程的上下文窗口顶部固定，持续整个会话（episode）。当会话结束时，技能上下文被清理并从线程中移除。

**这似乎不是我们真正想要的从技能中获得的情境化行为修改。**

## 将技能建模为情境化行动

技能的定义更自然地与**情境有用的行为**对齐，这些行为是情境范围的，并且在使用后不会污染主上下文。这非常类似于你如何减少认知工作来执行你已经掌握的技能。

技能似乎也相对好地映射到**情景记忆**的概念。例如，当你在汽车上更换轮胎时，你可能清楚地知道你在序列的哪一步。但关键在于：你已经学习了一系列行动，这些行动建立在其他已学习的子程序之上，这些子程序组合成这个名为"更换轮胎"的新技能，现在这是一个你可以在其中工作和记忆的新的隔离上下文，作为一个"会话"。

之前所有这些子程序都是你被单独教授的（或者可能在更换轮胎的上下文中），然后你决定将它们全部应用于更高阶的序列，这使你在任务上成功。

完全靠自己学习它们需要很多工作，对吧？但有人可能教过你这些片段或完整序列。

以这种方式，被教授一项技能类似于给模型一个技能。被教授如何将它们排序类似于给模型一个如何排序它们的技能。

但要重申，这里仍然有一个问题。

制作你学习视频的那个 YouTuber 并不会仁慈地注视着你并运行一个斜杠命令让你记住你应该遵循序列或回看那个视频。**你实际上自己有动力去回忆你需要什么来执行技能并将其应用于这个目标行动序列。**

因此，出现了关于技能应该是什么的更好定义：

> **技能应该是情境化的行为，组合成更大的序列，其中目标和环境状态指导该技能的使用。**

如果这是一个恰当的定义，并且鉴于我们对 Slate 的了解，Slate 将是更自然技能使用的完美架构。Slate 的线程提供了一种动态配置权限、提示和隔离上下文以采取范围化行动的能力。本质上，这就是你在动态运行技能的权限和执行模型中想要的一切，将技能添加为参数化线程的方式应该相对简单。

## 在 Slate 中构建动态技能

我们用 Slate 测试的第一件事是为主线程（"编排器"）提供一个可用技能列表、查看和搜索它们的方法，以及直接使用技能实例化线程的参数。

Slate 的架构理论上使其能够在采取**行动**时激活技能**实例**。在这种情况下，行动仅仅是应用技能的步骤序列！

太好了，技能解决了，一切都很好，世界永远改变了等等等等。

哈。不。

我们的初始解决方案是这样的：

我们给主线程一个 `skills` 工具，上面有 `view` 和 `list` 子命令，以及它的编排能力。

然后它可以直接将类似 `frontend-engineering` 的技能应用于它启动的线程，并且该线程在该会话期间将激活该技能。技能在会话终止时停用，线程的上下文被清理！

很好，所以我们有情境范围的技能使用，主 Agent 可以根据它对技能应该被使用的情况的了解来激活技能。

它实际上对像 [默认的 Anthropic 前端技能](https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design/SKILL.md) 这样的事情效果很好。

**然而，有一个核心问题：你如何处理用户交互？**

有些技能需要用户交互来进行规划、决策等。

这是我们想出的选项：

1. 为线程提供一个工具，获取阻塞弹出对话框，用户可以直接响应 Agent
2. 强制线程升级到主 Agent 如果需要与用户交谈，并使用线程驱动主 Agent 的用户交互
3. 秘密的第三件事

### 选项 1：直接单次使用对话框

我们添加这个的方式是给线程提供一个类似于 `ask_question` 的工具。模型可以向用户发送消息并获得响应。这有点像权限请求对话框。

主 Agent 在提供需要用户交互的技能时，可以设置为配置子 Agent 与用户聊天。

**有两个问题：**

1. UX 很糟糕，作为用户会很糟糕
2. 子 Agent 实际上无法合理地与用户进行对话

这两点使得交互式/对话式技能几乎不可能

### 选项 2：强制线程升级

这遵循更确定性的基于失败的方法。基本上我们可以告诉子 Agent："嘿，如果你接到一个需要与用户交谈的任务，你实际上不能这样做，所以你应该向编排器报告问题并终止会话"。

**这显然很糟糕**，因为编排 Agent 会生成一个子 Agent，仅仅是为了它撞到墙上然后驱动主 Agent 的行为。

这不仅浪费周期并混淆模型，而且还为线程驱动编排器的行为提供了激励。

我们之前已经讨论过多 Agent 的问题，这种解决方案将线程驱动主 Agent 的行为使它们处于相同的优先级，这意味着你需要一个共识机制，最好避免。

### 选项 3：秘密的第三件事

我们非常努力地思考。我们最聪明的朋友都聚集在一个房间里，讨论我们到底需要什么。

然后我们意识到：**如果我们改变执行模型会怎样？**

**如果不是所有线程都异步和并行驱动会怎样？**

这条质疑线恰好引出了一个涵盖所有可能交互情况、产生最优 UX、并保持线程所有范围化执行好处的解决方案。

我们更新了 Slate 的执行模型以支持一个新原语：**分叉（forking）！**

![三种交互式技能选项](https://pbs.twimg.com/media/HEuP0yjbUAAC5zc.jpg)

*三种交互式技能方法。选项 1（弹出对话框）提供一次性交互，UX 差。选项 2（线程升级）反转控制并混淆模型。选项 3（分叉）实现完整的双向对话，同时保持情景隔离——首选方法。*

## 上下文分叉和交互式技能使用

> **[注：截至 2026 年 4 月 1 日，此功能处于 Alpha 阶段，已被推迟以确保可靠性]**

Slate 之前强制所有线程在后台运行，这意味着线程无法合理地与用户交互以进行技能。

我们添加了**同步分叉**，以便用户可以继续与基本上相同的编排器 Agent 交谈，只是增加了一个好处：一旦技能使用完成，隔离和情景记忆仍然会生效。

工作原理是：Slate 可以选择生成一个分叉，然后阻塞整个系统。这类似于运行一个同步函数。

**分叉不能像线程那样继续，它们立即阻塞编排器采取的所有其他行动。**

在 UI 中，它们在底层强制接管 UI 交互，这意味着 UX 只是与 Agent 的连续聊天。

作为用户，你不会改变太多，但它解锁了交互式技能使用。

主 Agent 为每个技能做出路由决策：**需要用户交互的技能被路由到分叉，行动技能在线程中自主运行。**

![技能路由：线程 vs 分叉](https://pbs.twimg.com/media/HEuP0yjbUAAC5zc.jpg)

*主 Agent 根据技能是否需要用户交互来路由每个技能激活。行动技能在后台工作线程中运行；交互式技能在同步分叉中运行。两者都向编排器返回压缩的会话输出。*

## 这一切给你带来了什么？

**这给你带来了技能自动化。** 通过改变 Slate 的实现模型以包括分叉，这意味着 Slate 的委托能力涵盖了在线程中将技能作为行动使用的所有可能功能需求。由于我们现在通用地涵盖了支持技能作为子线程的需求空间，我们可以开始编排它们。

因此，我们还引入了另一个概念：**编排技能（orchestration skills）**

> **编排技能**是你（用户）根据需要激活在主编排器 Agent 上的技能。与普通技能引用脚本和资源不同，**编排技能**应该引用……其他技能！

重点是现在你可以选择性地使模型理解如何使用方法、线程、分叉和技能的**条件激活序列**。

你可以定义这样的东西：

```markdown
------
这个技能允许你在代码库中正确实现功能。
当用户想要实现新功能时，建议这个技能。
------

当这个技能激活时，你应该首先分叉并运行 `plan` 技能。
...
一旦分叉完成规划过程，Slate 开始实施计划。
...
一旦你审查了代码，通过运行 `/qa` 技能来验证输出。
...
一旦 `/qa` 技能成功完成，你应该向用户返回技能使用的结果。
```

![编排技能链](https://pbs.twimg.com/media/HEuP0yjbUAAC5zc.jpg)

*编排技能将子技能链接成序列：plan（分叉）输入到 implement（线程），然后是 review（分叉），然后是 QA（分叉）。主 Agent 按顺序激活每个，每个会话的输出成为下一个技能的输入。*

这个技能将教导它应该使用你所有其他技能执行的行动序列。

此外，因为它又只是一个技能，你可以有多个不同的操作模式，在编排技能中定义不同的子程序包。

然后主模型可以相当程序化地条件执行这些子程序，与现有的 Harness 相比。

**不再需要在会话结束时手动运行 /review！**

我们很高兴地宣布技能链终于发布了。

让我们知道你是如何使用这些新功能的，或者如果你卡住了，请发邮件至 team@randomlabs.ai

## 附录

### 分叉作为 Slate 执行模型中原语的好处

分叉作为原语还提供了另一个好处：作为用户，你现在可以直接与执行实际工作的 Agent 一起工作。这意味着你可以与任何其他 Agent 做所有相同的高接触工作，但只在你需要的时候。

虽然这在某种程度上改变了 Slate 的执行模型，但我们认为这里的可用性好处非常大，特别是当模型在棕地项目上仍然有完全自主性问题时。

### 与操作系统的比较

我们发现自己越来越多地转向使用 Unix 和 OS 术语来描述我们正在构建的东西。Slate 的线程、分叉、线程权限模型、将上下文视为进程内存等，似乎都遵循类似的模式。

目前尚不清楚这将如何继续，因为我们自然地得出这些结论。

本节在这里的唯一原因是，早期读者和我们遇到的人提出了这个类比值得指出。有人建议我们明确注意这一点作为一个有趣的旁注。

## 经验总结与方法论

### 核心洞察

1. **技能应该是行动，不是提示**
   - 人类技能是情境化的行为，通过重复内化
   - LLM 技能应该激发模型知道但不自然采用的行为
   - 填补"知识悬置"：模型知道如何做 vs 选择做什么的差距

2. **渐进披露优于强制加载**
   - 规则：强制加载到上下文，浪费宝贵 token
   - 技能：按需激活，情境范围，使用后清理
   - 避免上下文污染，保持有效上下文长度

3. **情境隔离是关键**
   - 线程提供隔离的上下文窗口
   - 技能在会话期间激活，结束后清理
   - 类似 Lisp continuation：可部分执行、保存状态、恢复

### 架构演进

**问题：如何处理需要用户交互的技能？**

| 方案 | 方法 | 问题 |
|------|------|------|
| 选项 1 | 弹出对话框 | UX 差，无法真正对话 |
| 选项 2 | 线程升级到主 Agent | 浪费周期，混淆模型，需要共识机制 |
| 选项 3 | 同步分叉 | ✅ 双向对话 + 情景隔离 |

**解决方案：分叉（Forking）**
- 同步阻塞，接管 UI 交互
- 保持线程的隔离和情景记忆好处
- 主 Agent 路由决策：交互式→分叉，行动式→线程

### 技能链（Skill Chaining）

**编排技能**：引用其他技能的技能，定义条件激活序列

```
plan (分叉) → implement (线程) → review (分叉) → QA (分叉)
```

**好处：**
- 自动化技能序列
- 条件执行子程序
- 无需手动调用 /review 等

### OS 类比

Slate 架构越来越像操作系统：
- 线程 = 进程
- 分叉 = fork()
- 权限模型 = 进程权限
- 上下文 = 进程内存

这种自然趋同表明架构设计的正确性。
